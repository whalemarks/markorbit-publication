# B07-CH-25 — Typed Return and Resumption of Workplace Truth

## A managed network is incomplete until the result returns

Routing work outward is only half of the system. The result must return to the Originating Workplace with its source, meaning, uncertainty and authority intact.

```text
Execution Provider Workplace
→ Evidence and outcome
→ Return Envelope
→ MGSN validation and coordination
→ Originating Workplace
→ Owning Service validation
→ formal-state update where authorized
```

Without governed Return, cross-Workplace execution becomes another operational black hole.

## Return is more than a message

An email saying “filed” or “completed” may be useful Communication, but it is not a complete Return contract.

A Return Envelope should identify:

- the originating engagement;
- provider and responsible Workplace;
- service and jurisdiction;
- milestone or outcome type;
- source-qualified status;
- effective and source dates;
- included Evidence;
- unresolved issues;
- deadlines or next actions;
- confidence and uncertainty;
- corrections or deviations;
- financial-reconciliation projection;
- confidentiality and access scope.

## Typed outcomes

Different outcomes must not be flattened into one generic `done` state. Return types may include:

- provider accepted;
- filing prepared or submitted;
- official receipt obtained;
- authority action received;
- response filed;
- publication or registration confirmed;
- certificate received;
- correction required;
- deadline risk;
- provider unable to proceed;
- replacement initiated;
- engagement cancelled;
- outcome unknown.

Typed Return enables the receiving Workplace and Owning Service to apply the correct next rule.

## Source qualification

A Return must state what supports the outcome:

```text
Official authority source
Provider professional assertion
Provider-generated Evidence
Third-party operational Evidence
Customer-supplied Evidence
Operator observation
Unknown or unverified source
```

The same words may carry different authority. “Registered” supported by an official register is not equivalent to a provider expectation that registration will occur.

## Return received is not formal-state mutation

The Originating Workplace may receive a Return, but the responsible Owning Service must validate structure, source, authority and consistency before applying a formal transition.

```text
Return received
→ structural validation
→ source and authority validation
→ consistency check
→ human review where required
→ Owning Service transition
```

This protects Core and operational truth from malformed, stale or unsupported network outcomes.

## Resumption of Workplace truth

The Originating Workplace remains the business-sovereignty boundary for the customer relationship and operating context. A valid Return allows it to resume responsibility for:

- customer communication;
- Matter coordination;
- next-step approval;
- internal Task and Deadline management;
- Document retention;
- invoicing and reconciliation;
- further routing where needed.

The Execution Provider Workplace does not acquire the customer relationship merely because it performed the work.

## Unknown, incomplete and conflicting outcomes

The network must preserve uncertainty explicitly.

```text
Unknown outcome
≠ failed outcome
≠ completed outcome
≠ permission to invent status
```

When Evidence is incomplete, the Return should state what is known, what is missing, who must verify it and what Deadline or risk remains.

Conflicting provider statements, official records, customer Documents or earlier observations should trigger source comparison, freshness review, identity checks, Correction Request, Incident or Dispute where material, and a reasoned resolution. Conflicting Evidence must not be overwritten silently.

## External-route Evidence bridge

R1 external self-managed work may return through a controlled bridge. The Originating Workplace may upload an external provider report, official receipt, authority status, certificate, Deadline notice or other Evidence.

That Evidence must retain its external source classification. MGSN must not imply that it managed, verified or guaranteed the external work.

## Return and network learning

A completed Return may later support Delivery Performance Evidence, Trust assessment, procurement review, Qualification refresh, Incident analysis and a Network Learning Candidate.

Learning must rely on governed, source-aware Evidence rather than private customer data or unreviewed allegations.

## Part V conclusion

CH21–CH25 have completed the managed-service operating loop:

```text
Managed Network Engagement
→ Funds Requirement and release conditions
→ Fulfillment Milestones and Evidence
→ Incident, Correction or Replacement where required
→ typed Return
→ Workplace resumption
→ Owning Service validation
```

Part VI now asks how the network remembers relationships and performance without converting history into permanent ownership, universal ratings or unreviewable sanctions.
