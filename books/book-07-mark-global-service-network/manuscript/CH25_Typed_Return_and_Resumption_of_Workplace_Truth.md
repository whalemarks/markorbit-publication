# B07-CH-25 — Typed Return and Resumption of Workplace Truth

## 1. A managed network is incomplete until the result returns

Routing work outward is only half of the system. The result must return to the Originating Workplace in a form that preserves source, meaning, uncertainty and authority.

```text
Execution Provider Workplace
→ evidence and outcome
→ Return Envelope
→ MGSN validation and coordination
→ Originating Workplace
→ Owning Service validation
→ formal business-state update where authorized
```

Without governed Return, cross-Workplace execution becomes another operational black hole.

## 2. Return is not a message

An email saying “filed” or “completed” may be useful communication, but it is not a complete Return contract.

A Return Envelope should identify:

- originating engagement;
- provider and responsible Workplace;
- service and jurisdiction;
- milestone or outcome type;
- source-qualified status;
- effective date and source date;
- evidence included;
- unresolved issues;
- deadlines or next actions;
- confidence and uncertainty;
- corrections or deviations;
- financial reconciliation projection;
- confidentiality and access scope.

## 3. Typed outcomes

Different outcomes must not be flattened into one generic “done” status.

Possible Return types include:

- provider accepted;
- filing prepared;
- filing submitted;
- official receipt obtained;
- authority action received;
- response filed;
- publication confirmed;
- registration confirmed;
- certificate received;
- correction required;
- deadline risk;
- provider unable to proceed;
- replacement initiated;
- engagement cancelled;
- outcome unknown.

Typed Return allows the receiving Workplace and Owning Service to apply the correct next rule.

## 4. Source qualification

A Return must state what supports the outcome.

```text
Official authority source
Provider professional assertion
Provider-generated evidence
Third-party operational evidence
Customer-supplied evidence
Operator observation
Unknown or unverified source
```

The same wording can have different authority depending on source. “Registered” based on an official register is not equivalent to “the provider expects registration”.

## 5. Return Received is not formal-state mutation

The Originating Workplace may receive a Return, but formal records must not change automatically unless the responsible Owning Service validates the input and is authorized to apply the transition.

```text
Return received
→ structural validation
→ authority and source validation
→ consistency check
→ human review where required
→ Owning Service transition
```

This protects Core and operational truth from malformed, stale or unsupported network outcomes.

## 6. Resumption of Workplace truth

The Originating Workplace remains the business-sovereignty boundary for the customer relationship and operating context.

A valid Return allows it to resume responsibility for:

- customer communication;
- Matter coordination;
- next-step approval;
- internal task and deadline management;
- document retention;
- invoicing and reconciliation;
- further routing if needed.

The provider does not acquire the customer relationship merely because it performed work.

## 7. Unknown and incomplete outcomes

The network must support uncertainty explicitly.

```text
Unknown outcome
≠ failed outcome
≠ completed outcome
≠ permission to invent status
```

When evidence is incomplete, the Return should state what is known, what is missing, who must verify it and what deadline or risk remains.

## 8. Conflicting Returns

Conflicts may arise between provider statements, official records, customer documents and previous observations.

A conflict should trigger:

- source comparison;
- freshness review;
- identity and reference check;
- correction request;
- incident or dispute where material;
- temporary status hold;
- reasoned resolution.

The system should preserve the conflicting evidence rather than overwrite it silently.

## 9. External-route evidence bridge

R1 external self-managed work may also return through a controlled bridge.

The Originating Workplace may upload or enter:

- external provider report;
- official receipt;
- authority status;
- certificate;
- deadline notice;
- other evidence.

Such evidence must retain its external source classification. MGSN must not imply that the platform managed, verified or guaranteed the external work.

## 10. Return and network learning

A completed Return may later support:

- delivery performance evidence;
- Trust assessment;
- procurement review;
- provider qualification refresh;
- incident analysis;
- network learning candidates.

Learning must use governed, source-aware evidence rather than private customer data or unreviewed allegations.

## 11. Product rule

Every managed engagement must end with a typed, source-qualified and auditable Return. The Return restores continuity to the Originating Workplace, but only the authorized Owning Service may convert it into formal business truth.
