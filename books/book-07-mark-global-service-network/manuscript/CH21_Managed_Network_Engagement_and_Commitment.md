# B07-CH-21 — Managed Network Engagement and Commitment

## 1. Engagement begins after route selection, not before

A recommendation is not an engagement. A user confirmation is not an engagement. Even a Provider Allocation is not yet a completed engagement.

A Managed Network Engagement begins only when the relevant commercial, authority and provider-acceptance conditions are satisfied.

```text
Recommended Route
→ User Disposition
→ Provider Allocation
→ Provider Acceptance
→ Managed Network Engagement
```

Each transition answers a different question:

- **Recommendation**: which route appears most suitable?
- **User Disposition**: which route does the Originating Workplace authorize?
- **Provider Allocation**: which provider is the platform preparing to instruct?
- **Provider Acceptance**: is the provider willing and able to accept the defined work?
- **Managed Network Engagement**: have the parties and platform reached a governed operating commitment?

Collapsing these decisions creates silent appointment, stale consent and disputed responsibility.

## 2. The engagement boundary

A Managed Network Engagement is the MGSN Product-local record that binds together:

- the authorized Capability Need;
- the selected route;
- the accepted Service Offer;
- the allocated and accepting provider;
- the service scope and exclusions;
- funds requirements;
- expected milestones and evidence;
- incident and replacement rules;
- Return obligations;
- the Originating Workplace and Execution Provider Workplace.

It does not replace the Core Order, Matter, Task, Workflow or Finance Ledger.

```text
Managed Network Engagement
≠ Customer ownership
≠ Matter formal state
≠ Finance ledger
≠ Provider professional file
```

It is the network operating envelope around those authoritative records.

## 3. Provider commitment

Provider Acceptance must be explicit, current and scoped.

A valid Provider Commitment should confirm:

- the provider identity and responsible Workplace;
- service and jurisdiction;
- accepted scope;
- exclusions and assumptions;
- expected timing;
- required inputs;
- commercial terms;
- conflicts and restrictions checked;
- current capacity;
- required evidence and Return obligations.

A general statement such as “we can handle trademarks in this country” is not sufficient commitment for a specific engagement.

## 4. Instruction preparation

The platform may prepare an instruction package, but instruction must remain controlled.

The package should contain only the information necessary for the provider to perform the accepted work. Sensitive customer or Matter information should be disclosed progressively and according to authority.

```text
Need-level projection
→ provider acceptance
→ authorized instruction package
→ execution disclosure as required
```

No provider should receive unrestricted Workplace data merely because it was considered as a candidate.

## 5. Responsibility allocation

The Managed Network Engagement should make responsibility visible.

### Originating Workplace

The Originating Workplace remains responsible for:

- customer relationship;
- authorization to proceed;
- customer-facing explanation;
- source information supplied to the network;
- internal approvals;
- receiving and validating Return.

### Platform-owned MGSN

MGSN remains responsible for:

- controlled provider routing;
- procurement and Service Offer governance;
- engagement coordination;
- funds and fulfillment checkpoints;
- incident, correction and replacement governance;
- Return orchestration;
- network evidence and Trust inputs.

### Execution Provider Workplace

The provider remains responsible for:

- professional judgment;
- local legal and procedural work;
- timely performance;
- accurate evidence;
- disclosure of conflicts, inability or delay;
- Return in the required form.

### Owning Service

The Owning Service remains responsible for formal business-state mutation.

## 6. Material change after engagement

Provider acceptance does not freeze reality. Official fees, facts, deadlines, scope or legal conditions may change.

A material change should trigger a governed branch:

```text
Continue within accepted tolerance
or
Issue Correction / Additional Approval
or
Re-price and reconfirm
or
Replace provider
or
Cancel / refund / dispute
```

The platform must not silently expand scope, substitute a provider or treat a changed engagement as if the original authorization still applied.

## 7. Commitment is not completion

A provider commitment means the provider has accepted responsibility for the defined work. It does not mean the work has been filed, completed, accepted by an authority or returned to the Originating Workplace.

```text
Provider Acceptance
≠ Filing
≠ Completion
≠ Official acceptance
≠ Formal-state update
```

That distinction is essential for reliable downstream funds, milestones and Return controls.

## 8. Product rule

A Managed Network Engagement is valid only when route authorization, provider acceptance, scope, commercial terms, responsibility and Return obligations are all explicit and mutually consistent.
