# B07-CH-30 — The Operator-Assisted MVP 0

## 1. MVP 0 is a controlled operating proof

MGSN MVP 0 is not a miniature public marketplace and not a promise to automate the entire global service network.

Its purpose is narrower:

> prove that an Originating Workplace can express a bounded Capability Need, receive an explainable managed route, confirm that route, obtain Provider Acceptance, pass through funds and fulfillment checkpoints, and receive a typed Return without losing Workplace sovereignty or platform control.

The first MVP must demonstrate one complete governed loop before it attempts scale.

## 2. The minimum closed loop

```text
Originating Workplace
→ Capability Need Projection
→ admitted supply evaluation
→ Recommended Route
→ user confirmation
→ Provider Allocation and Acceptance
→ funds checkpoint
→ fulfillment milestone and Evidence
→ typed Return
→ downstream validation
```

Every step must remain inspectable. A partially automated black box is not a stronger MVP than an operator-assisted flow whose decisions can be explained and corrected.

## 3. Initial participant scope

MVP 0 should begin with a deliberately small participant group.

Demand participants may include:

- MarkReg self-operated Workplace;
- selected trademark agencies using MarkReg or Workplace;
- selected Lite users with a genuine cross-border service need;
- internal pilot teams acting under explicit test authority.

Supply participants should be limited to providers that have completed:

- Organization and identity review;
- Provider Network Profile;
- Capability Claim and Evidence review;
- qualification verification where applicable;
- procurement and Service Package admission;
- communication and Evidence expectations;
- participation terms and audit consent.

A provider name in an old spreadsheet is not admitted supply.

## 4. Initial jurisdiction and service scope

MVP scope should favor services that are frequent, sufficiently standardized and operationally observable.

Suitable initial candidates may include:

- routine trademark applications;
- renewals and maintenance filings;
- recordals with clear document requirements;
- selected declarations or formal post-registration actions;
- straightforward office-action work with defined review boundaries.

The MVP should not begin with every country and every legal service.

High-uncertainty litigation, emergency disputes, unusual regulatory work and services with unresolved funds or licensing requirements should remain operator-only, deferred or outside MVP scope.

## 5. Operator responsibilities

The operator is not a hidden substitute for Product logic. The operator performs explicit responsibilities while the system captures the structured decision path.

MVP operators may:

- review Capability Need completeness;
- verify disclosure scope;
- confirm provider eligibility and current Availability;
- request conflict confirmation;
- assemble or approve Candidate Route Sets;
- review recommendation explanations;
- coordinate Provider Acceptance;
- verify funds checkpoints with Finance;
- monitor milestones and Evidence;
- open Incident or Correction records;
- coordinate replacement;
- review Return completeness;
- escalate uncertain or protected actions.

Operator actions must be attributable, reasoned and auditable.

## 6. What must remain human

MVP 0 must retain human authority for:

- final user route confirmation;
- material provider allocation decisions;
- conflict interpretation;
- exception pricing;
- funds release authorization;
- legal or professional judgment;
- provider replacement where customer interest may be affected;
- adverse Trust consequences;
- suspension or retirement;
- formal-state mutation in an Owning Service;
- any External Protected Action.

```text
Automation support
≠ delegated professional authority
```

## 7. Product surfaces

The MVP may use a small number of clear surfaces rather than a complete platform redesign.

### Originating Workplace surface

Shows:

- Capability Need summary;
- disclosure scope;
- selected route type;
- Recommended Route and bounded alternatives;
- user-facing price and validity;
- unresolved conditions;
- confirmation or rematch action;
- milestone and Return status.

### Operator surface

Shows:

- admitted provider supply;
- qualification and Availability freshness;
- procurement terms;
- Candidate Route Set;
- funds and fulfillment checkpoints;
- Incident, Correction and Replacement controls;
- audit and explanation context.

### Provider surface

Shows only the authorized package necessary to:

- accept or refuse;
- identify responsible professional or team;
- confirm timing and exceptions;
- deliver milestones and Evidence;
- return completion or incident information.

The provider surface must not expose the Originating Workplace's full customer relationship or margin.

## 8. Manual controls are not Product failure

Some MVP steps may be operator-assisted, including:

- provider invitation;
- evidence review;
- procurement term entry;
- conflict confirmation;
- candidate comparison;
- funds verification;
- milestone review;
- Return qualification.

The Product requirement is that these steps use controlled records and explicit decisions.

A manual action inside a governed workflow is preferable to false automation outside the authority model.

## 9. No-candidate and unknown outcomes

MVP 0 must support safe non-success outcomes:

```text
No eligible candidate
Provider refused
Funds condition not met
Milestone overdue
Evidence incomplete
External outcome unknown
Return disputed
```

These states may trigger operator sourcing, rematch, R1, deferment, correction or replacement.

They must not be converted into a fictional successful route merely to improve conversion metrics.

## 10. MVP acceptance baseline

MVP 0 must demonstrate all sixteen acceptance criteria in `B07-SPEC-0004`, including:

- Workplace-scoped participation;
- platform-owned supply control;
- Claim and Evidence separation;
- R1, R2 and R3;
- explainable recommendation;
- bounded alternatives;
- user confirmation;
- Provider Acceptance;
- funds checkpoints;
- fulfillment milestones;
- typed Return;
- Incident and Replacement;
- Trust and Relationship Provenance;
- dual-role separation;
- no-candidate safety;
- protected-action boundaries.

Passing the happy path alone is insufficient.

## 11. Evaluation metrics

The first evaluation should focus on network quality and continuity:

- Capability Need coverage;
- time to an eligible route;
- recommendation explanation completeness;
- user confirmation and rematch behavior;
- Provider Acceptance rate;
- milestone timeliness;
- Evidence completeness;
- incident recovery;
- replacement recovery;
- repeat use by service scope;
- procurement improvement;
- privacy, authority and funds-control findings.

Raw provider count, page views, bidding activity and gross transaction volume do not prove that MGSN works.

## 12. Exit criteria for MVP 0

MVP 0 should progress only when it demonstrates:

1. repeatable route formation for the selected services;
2. no hidden bypass of eligibility or user confirmation;
3. reliable Provider Acceptance and fulfillment Evidence;
4. safe handling of no-candidate, refusal and unknown outcomes;
5. auditable funds and Return coordination;
6. no material Workplace privacy or authority breach;
7. operator effort that is understood well enough to prioritize later automation.

## 13. Product rule

MGSN MVP 0 is an operator-assisted proof of a governed network loop. It succeeds when the platform can safely coordinate real global capability with clear authority, Evidence and recovery—not when it merely automates the largest number of screens.
