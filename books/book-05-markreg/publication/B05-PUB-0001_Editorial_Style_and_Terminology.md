# B05-PUB-0001 — Editorial Style and Terminology

## Status

- **Status:** Controlled Scaffold
- **Source:** B05-REV-0012 and B05-PUBLICATION-FINISHING-PACK-001
- **Primary workstream:** PF-06
- **Applies to:** CH00–CH47, Appendix A–G and publication apparatus

## 1. Purpose

This record defines the editorial rules used to finish Book 05 without changing Product semantics silently.

The final edit should make the book shorter, clearer and more native in English while preserving source, version, responsibility, authority and lifecycle distinctions.

## 2. Voice

Book 05 should be:

- direct;
- precise;
- professionally neutral;
- readable by Product, legal, operations and engineering audiences;
- explicit about uncertainty and authority;
- concrete through artifacts, examples and scenarios.

It should avoid promotional claims, vague futurism and unsupported legal certainty.

## 3. Controlled Capitalization

Use capitalization where the term refers to the defined MarkOrbit or MarkReg concept.

| Controlled term | Usage |
| --- | --- |
| Product | a focused governed Product under Book 04 architecture |
| Workplace | the organization-level operating environment and Orbit context |
| Core | Book 02 shared semantics and contracts |
| Execution | Book 03 governed execution system or defined execution context |
| Review | a defined accountable review action or record |
| Decision | a defined accountable decision, not any informal choice |
| Communication | a governed communication artifact or action |
| Document | a formal or controlled document object, not every uploaded file |
| Evidence | controlled evidence with provenance and purpose |
| Knowledge | governed reusable Knowledge, not any text or memory |
| Capability | a typed ability with evidence and scope |
| Matter / Order / Task | defined formal shared objects where applicable |
| Human Review / Human Judgment / Human Selection | accountable Human roles and actions |

Lowercase may be used for ordinary language where the defined concept is not intended.

## 4. Artifact Naming

Named artifacts should use a stable noun phrase and consistent capitalization.

Examples:

- Need Brief;
- Recommendation Set;
- Option Set;
- Proposal;
- Quote;
- Commercial Instruction;
- Formal Intake;
- Requirement Set;
- Readiness Assessment;
- Handoff Envelope;
- Filing Package Candidate;
- Filing Approval;
- Official Event Snapshot;
- Right Baseline;
- Maintenance Obligation Set;
- Portfolio Action Plan;
- Product Session;
- Return Envelope.

Do not alternate casually between `form`, `record`, `package`, `context`, `snapshot`, `view`, `set`, `envelope` and `baseline` where the distinction is material.

## 5. Authority Language

Prefer explicit verbs:

- proposes;
- recommends;
- records;
- confirms;
- reviews;
- approves;
- instructs;
- accepts;
- executes;
- reports;
- acknowledges;
- verifies;
- supersedes.

Avoid vague verbs such as `handles`, `owns everything`, `automates`, `completes` or `decides` unless the actor and authority are clear.

## 6. State Language

Do not write `filed`, `registered`, `renewed`, `transferred`, `accepted` or `completed` without identifying the source or state domain where ambiguity is possible.

Prefer:

- provider reports submitted;
- official acknowledgement received;
- registration recorded by the office;
- renewal request sent;
- renewed status verified from the official record;
- assignment signed;
- ownership recordal acknowledged.

## 7. AI Language

Do not describe AI as an accountable actor.

Preferred language:

- AI assists;
- AI proposes;
- AI extracts;
- AI compares;
- AI drafts;
- AI flags uncertainty;
- a Human reviews, accepts, modifies or rejects the output.

Avoid:

- AI approves;
- AI files autonomously;
- AI determines the legal owner;
- AI guarantees availability;
- AI creates official truth.

## 8. Source and Confidence Language

Every material rule, fee, deadline, official event or outcome should be described with appropriate source and freshness language.

Use distinctions such as:

- official source;
- verified provider evidence;
- professional interpretation;
- organization experience;
- secondary source;
- Product projection;
- current, checked, stale, disputed, superseded or unknown.

Avoid `the law says` or `the deadline is` where the underlying source, jurisdiction and effective period are not stated or controlled.

## 9. Chapter Structure

Publication editing should preserve a clear chapter progression:

1. purpose or user question;
2. controlled concepts and artifacts;
3. authority and source boundaries;
4. reference journey or example;
5. conformance scenarios;
6. AI assistance and prohibitions where relevant;
7. failure modes;
8. minimum Product lock or conclusion;
9. handoff to the next chapter.

Not every chapter must use identical headings after compression. The logical sequence should remain visible.

## 10. Lists and Tables

Use a table when the reader needs to compare dimensions, roles, states or alternatives.

Use bullets for a small set of parallel items.

Avoid long repeated lists that restate the same constitutional rules. Reference the named rule or appendix where possible.

## 11. Examples

Prefer `EMBERLOOP` and `RIVERKITE` for recurring examples.

A new example should be introduced only when it tests a materially different behavior and should state why the existing journeys are insufficient.

Examples must not be presented as legal advice, precedent or guaranteed outcome.

## 12. Cross-References

Use chapter, appendix, specification and review identifiers consistently.

Examples:

- CH24;
- Appendix B;
- B05-SPEC-0001;
- B05-REV-0012;
- MR-CR-04.

Final rendered editions may add page references, but the controlled source should retain stable identifiers.

## 13. Metadata Rule

Active manuscript metadata should use the owner-accepted chapter map and the current publication status.

Historical review records may retain their original status language.

PF-01 must remove active `Candidate` chapter-map wording and normalize chapter statuses before RC1.

## 14. Prohibited Editorial Drift

Editing may not silently:

- redefine a Core object;
- move authority from Human to Product or AI;
- merge readiness, approval, Execution and official effect;
- convert provider evidence into official truth;
- convert one jurisdiction or case into a global rule;
- turn MGSN into an open marketplace;
- make Product metrics imply legal causation;
- expand the supported Product or implementation scope.
