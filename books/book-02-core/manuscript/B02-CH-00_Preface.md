# Preface

Book 01 established the operating philosophy of MarkOrbit.

It described why global brand services require a new kind of professional operating system: one that can organize reality, facts, knowledge, judgment, action, coordination and experience into a reusable service foundation.

Book 02 begins from that foundation.

Its purpose is different.

Book 02 does not repeat the philosophy of Book 01.

Book 02 specifies the Core that makes the philosophy executable.

---

# Why This Book Exists

MarkOrbit is not intended to become a collection of disconnected tools.

If each product defines its own data, knowledge, workflows, AI agents, customers, trademarks, brands, opportunities, documents, service providers and communication records, the ecosystem will eventually fragment.

MarkReg would become one system.

MarkOrbit Lite would become another system.

MGSN would become a separate network.

AI agents would depend on inconsistent knowledge.

Orders and matters would be disconnected.

Brand assets and trademark assets would be duplicated.

Opportunities would be generated from partial information.

Professional judgment would remain trapped inside individuals, emails, spreadsheets and isolated workflows.

The purpose of the MarkOrbit Core is to prevent this fragmentation.

The Core provides the shared professional foundation that later products consume.

It defines the stable models, domains, objects, services, execution primitives and contracts that allow all future products to remain connected to the same operating logic.

Products may evolve.

User interfaces may change.

AI models may improve.

Service networks may expand.

The Core must remain stable enough to support all of them.

---

# What Book 02 Is

Book 02 is the Core Specification of MarkOrbit.

It defines:

- what the Core is;
- where the Core sits;
- what the Core owns;
- what the Core does not own;
- how the Core is structured;
- how Core concepts are specified;
- how products consume the Core;
- how the Core becomes executable through MVP and Codex implementation.

Book 02 is not merely descriptive.

It is intended to guide future implementation.

A concept defined in this book should eventually be able to influence one or more executable assets, including:

- a domain specification;
- an object specification;
- a service specification;
- an event specification;
- an API contract;
- an AI capability specification;
- a workflow contract;
- a Codex development task.

Book 02 is therefore both a publication and an architectural source of truth.

It is written to be read by humans, but it must also be precise enough to guide software development.

---

# What Book 02 Is Not

Book 02 is not a product requirements document.

It does not define the full feature set of MarkReg, MarkOrbit Lite, MGSN, Brand Asset Vault or Opportunity Engine.

Book 02 is not a user interface manual.

It does not define screens, buttons, layouts, visual styles or user journeys.

Book 02 is not an API tutorial.

It may define API contract principles and execution boundaries, but detailed API specifications belong in `core-specs/api/`.

Book 02 is not an AI prompt manual.

It may define AI capability and agent governance, but production prompts belong to implementation-level documents.

Book 02 is not a deployment guide.

It does not define infrastructure vendors, runtime environments, DevOps procedures or operational runbooks.

Book 02 defines the Core.

Products consume the Core.

Implementation realizes the Core.

# The Core Boundary

The boundary of the Core must remain clear.

The Core owns:

- canonical models;
- Core Domains;
- Core Objects;
- Core Services;
- execution primitives;
- Core Contracts;
- consumption rules;
- AI capability governance.

The Core does not own:

- product user interfaces;
- product-specific features;
- product pricing;
- product onboarding flows;
- marketing content;
- detailed deployment procedures;
- product-specific workflow experience;
- production prompt libraries.

This boundary protects the long-term architecture.

If Book 02 absorbs too much product detail, it will become unstable.

If Book 02 defines too little, products will fragment.

The correct balance is to define the Core strongly, while allowing products to consume and specialize it.

---

# Core, Workplace and Product

A recurring distinction in this book is the distinction between Core, Workplace and Product.

```text
Core
    provides models, objects, services, primitives and contracts

Workplace
    operates work through composition, workflow and experience

Product
    delivers user-facing functionality
```

This distinction is especially important for workflow.

The Core may define:

- Event Primitive;
- Task Primitive;
- State Primitive;
- Context Primitive;
- Notification Contract;
- Workflow Contract.

The Core does not own product-specific workflow execution.

Workplace operates work.

Products deliver user experience.

This prevents Book 02 from absorbing the responsibility of Book 03, Book 04, Book 05 or Book 06.

---

# AI in the Core

AI is important to MarkOrbit.

But AI is not the Core.

AI is one governed execution form of Core Capability.

Book 02 treats AI as part of the Core only when it is connected to:

- identity;
- permission;
- authorized knowledge;
- context;
- events;
- audit;
- human review;
- professional responsibility.

AI may help retrieve knowledge, analyze trademarks, draft documents, recommend classifications, detect opportunities, summarize communication and assist routing decisions.

AI shall not define professional truth.

AI shall not bypass permission.

AI shall not silently modify critical data.

AI shall not produce high-risk professional outputs without review.

The goal is not autonomous performance for its own sake.

The goal is governed professional capability.

# Manuscript and core-specs

Book 02 is not only a manuscript.

It is supported by executable specification files.

The intended structure is:

```text
book-02-core/

├── manuscript/
│   └── Book 02 chapters

├── planning/
│   └── Core positioning, blueprint, domain map and execution matrix

├── editorial/
│   └── Editorial protocol and chapter writing rules

├── core-specs/
│   └── Detailed domain, object, service, event, agent, API and workflow specifications

└── indexes/
    └── Domain, object, service, event, API, agent and Codex task indexes
```

The manuscript explains and defines the Core.

`core-specs/` contains executable details.

The manuscript shall not duplicate every domain, object, service, API or event.

`core-specs/` shall not redefine Core meaning independently from the manuscript.

Both are necessary.

The manuscript provides architectural meaning.

`core-specs/` provides implementation-ready structure.

# Editorial Principle

This book follows one editorial principle:

```text
One chapter
    =
one Core responsibility
```

A chapter may reference many concepts.

But it should own only one primary responsibility.

This rule is necessary because MarkOrbit is a complex ecosystem.

Without strict editorial discipline, the Core Specification would quickly become a mixture of philosophy, product planning, database design, API design, AI strategy, workflow manuals and marketing language.

Book 02 must remain focused.

It must define the Core.

# Closing Note

The MarkOrbit Core is not an abstract architecture exercise.

It is the foundation for a real system.

The work of global brand services is complex, fragmented and highly dependent on professional judgment.

A single tool cannot solve this problem.

A disconnected set of tools will only reproduce the same fragmentation in software form.

MarkOrbit needs a Core because professional service work needs shared memory, shared knowledge, shared objects, shared capabilities, shared events and shared contracts.

Book 02 defines that Core.

Everything that follows depends on whether this foundation is clear.
