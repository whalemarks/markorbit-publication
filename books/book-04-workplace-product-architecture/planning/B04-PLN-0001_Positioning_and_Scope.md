# B04-PLN-0001 — Positioning and Scope

## Core Question

> How does an independent professional organization establish its own operating orbit and consume MarkOrbit Core, Execution, knowledge, intelligence, capabilities, products and network services through its Workplace?

## Target Readers

- MarkOrbit architects.
- Product owners.
- Professional-service organization leaders.
- Product and engineering teams.
- AI governance and knowledge-system designers.

## Book 04 Purpose

Book 04 positions Workplace as the independent organizational Orbit and defines how products consume MarkOrbit foundations without redefining Core or Execution.

## Six-Part Structure

1. The Workplace as an Independent Orbit.
2. Organization Context and Operating Environment.
3. Knowledge, Intelligence and Capability Consumption.
4. Product Architecture and Product Embedding.
5. Outcomes, Artifacts and Delivery.
6. Network Participation and Orbital Ecosystem Evolution.

## In-Scope Subjects

Workplace identity, people, roles, permissions, accountability, clients, relationships, private knowledge, authorized AI context, operating surfaces, data boundaries, Information, Knowledge, Intelligence, Capability, Skill, Assistant, Guide, AI Agent placement, recommendations, governed Execution handoff, Product Architecture, product embedding, Lite, MarkReg and MGSN architectural profiles, Artifact and Delivery boundaries, routing, capability evidence, trust and orbital ecosystem learning.

## Explicit Non-Goals

Book 04 is not a replacement for Book 01, a semantic amendment to Book 02, a replacement for Book 03, a Product feature catalog, a Lite PRD, a MarkReg PRD, an MGSN marketplace specification, an Agent framework tutorial, a Prompt engineering manual, an Information & Value Stack manual, a microservice decomposition plan, a database schema specification, an API implementation specification, a deployment architecture, a code repository design or a production-readiness authorization.

## Relationship to Book 01–03

Book 01 supplies Operating System principles. Book 02 supplies frozen Core semantics and contracts. Book 03 supplies governed Execution boundaries. Book 04 consumes those authorities and must not replace them.

## Relationship to Book 05–07

Book 04 provides architectural foundations for Book 05 MarkReg, Book 06 MarkOrbit Lite and Book 07 MGSN. It does not fully specify those books and does not become their PRD.

## Relationship to Code Repositories

Code repositories may provide implementation and validation inputs. Code may demonstrate conformance. Code does not define professional architecture.

## Five Value Tests

1. Reduce Missed Work
2. Improve Professional Judgment
3. Accelerate Service Conversion
4. Improve Delivery Quality
5. Accumulate Organizational Capability

Every Book 04 architectural principle, product boundary or Workplace capability should support at least one of these values.

## Book 04 Scope Review Questions

1. Does the chapter preserve organizational autonomy?
2. Does it consume Core and Execution without redefinition?
3. Does it clarify a real Workplace or Product Architecture boundary?
4. Does it preserve human accountability and protected-action limits?
5. Does it help downstream products without becoming a PRD?

## Product Loop First, Shared Platform Extraction Second

Product learning should begin with controlled product loops. Shared platform extraction should follow evidence from validated product loops, not precede them as speculative platform design.
