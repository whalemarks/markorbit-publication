## Chapter Purpose

CH13 defined information access, provenance, and authorized context assembly.

CH14 defined governed Knowledge consumption and the Brain boundary.

CH15 defined Capability discovery and Skill selection.

This chapter defines the user-facing and runtime roles through which those foundations become usable inside Workplace:

- Assistant;
- Guide;
- AI Agent.

These three concepts are related.

They are not interchangeable.

An Assistant is an open and continuous user-facing assistance entry.

A Guide is a structured, goal-specific interaction that helps a user move through a bounded professional journey.

An AI Agent is a governed runtime role with explicit identity, purpose, Capability, Skill access, context, Permission, Policy, Human Review, and audit boundaries.

An Assistant may use one or more AI Agents.

A Guide may use one or more AI Agents.

An AI Agent may operate without being presented as a conversational Assistant.

A deterministic Guide may operate without AI.

If these concepts are collapsed, the system may:

- hide which Agent actually acted;
- treat a user-facing persona as professional authority;
- give a conversational surface unrestricted access;
- confuse guidance with Workflow;
- convert recommendations into silent actions;
- carry private context across organizations;
- treat personalization as delegated decision authority;
- make it impossible to test, suspend, or replace underlying Agents and Skills.

The central proposition is:

```text
Assistant
=
Continuous User Assistance Surface
+ User Context
+ Authorized Memory
+ Capability Access
+ Transparent Handoffs
```

```text
Guide
=
Defined Goal
+ Bounded Journey
+ Required Inputs
+ Structured Choices
+ Progress State
+ Exit Conditions
+ Governed Handoffs
```

```text
AI Agent
=
Explicit Agent Identity
+ Purpose
+ Agent Contract
+ Capability Profile
+ Selected Skills
+ Authorized Context
+ Permission
+ Policy
+ Output Mode
+ Human Review
+ Audit
```

The constitutional distinctions are:

```text
Assistant ≠ AI Agent

Guide ≠ Workflow

Persona ≠ Identity authority

Conversation ≠ permission

Personalization ≠ professional delegation

Agent registration ≠ authorization

Agent Capability ≠ Permission

Agent context ≠ all Workplace data

Agent recommendation ≠ decision

Agent preparation ≠ execution

User confirmation ≠ Human Review

Assistant memory ≠ organizational memory

Guide completion ≠ Task completion

Agent output ≠ formal business fact

Autonomy ≠ unbounded authority
```

Book 02 remains authoritative for Agent identity, Agent Registry, AI Context, Permission, Policy, Knowledge access, data scope, and Agent governance.

Book 03 remains authoritative for Workflow, Task, Review, Communication, protected action, and Owning Service handoff.

This chapter defines Workplace roles and consumption boundaries.

It does not redefine Core or Execution.

## 1. Why the Three Roles Must Be Separate

Users experience MarkOrbit through helpful surfaces.

They may ask questions, receive recommendations, follow structured journeys, prepare documents, compare providers, or review next actions.

The visible experience may feel unified.

The underlying responsibilities must remain distinct.

The Assistant answers:

> Where can the user continuously ask, explore, prepare, and receive help?

The Guide answers:

> How can the user move toward one defined goal through a bounded sequence of questions, choices, explanations, and prepared outputs?

The AI Agent answers:

> Which governed runtime identity is performing a specific AI-assisted operation under an explicit contract?

The separation allows Workplace to:

- preserve a stable user experience;
- replace underlying Agents or Skills;
- expose different Guide designs;
- apply different data boundaries;
- require Human Review;
- suspend one Agent without disabling the entire Assistant;
- support non-AI Guides;
- explain which operation occurred.

The relationship is:

```text
User
  │
  ├── Assistant
  │      ├── Guide
  │      ├── Product surface
  │      └── Agent operation
  │
  ├── Guide
  │      ├── deterministic logic
  │      ├── Knowledge
  │      ├── Skills
  │      └── Agent operations
  │
  └── Product
         ├── Assistant
         ├── Guide
         └── Agent operation
```

The experience may be integrated.

The identities and authorities remain separate.

## 2. Assistant Is the Continuous Assistance Entry

Assistant means:

> an open, persistent, user-facing assistance role through which a person can ask questions, explore context, receive explanations, prepare work, and move into Products, Guides, reviews, or governed actions.

An Assistant may support:

- question answering;
- orientation;
- Knowledge retrieval;
- summarization;
- drafting;
- Today Feed explanation;
- recommendation explanation;
- work preparation;
- Product handoff;
- Guide launch;
- source trace;
- missing-context detection.

The Assistant is continuous because it is not limited to one goal or one fixed sequence.

It may accompany the user across:

- clients;
- Matters;
- Products;
- operating surfaces;
- professional goals.

Continuity does not mean unrestricted memory or access.

The Assistant must re-evaluate context as the user changes organization, Product, client, Matter, or purpose.

---

## 3. Assistant Is Not Necessarily One AI Agent

A user may perceive one Assistant persona.

The implementation may use:

- no AI for some operations;
- one Agent for Knowledge retrieval;
- another Agent for drafting;
- a deterministic Skill for validation;
- a Product-specific Agent;
- a local Agent for restricted data;
- a human handoff.

The Assistant is therefore an orchestration and presentation role.

It is not necessarily one Agent object.

The surface should preserve enough trace to explain material AI participation.

The architecture is:

```text
Assistant experience
        ↓
operation classification
        ↓
Capability and Skill selection
        ↓
eligible Agent or deterministic service
        ↓
result returned through Assistant
```

The user-facing continuity must not erase runtime accountability.

---

## 4. Assistant Persona Is Not Agent Identity

A persona may define:

- name;
- tone;
- visual identity;
- communication style;
- user-facing role;
- personalization.

Agent identity defines:

- Agent reference;
- Agent type;
- purpose;
- registry key;
- contract;
- Capability scope;
- status;
- data access;
- audit trace.

One persona may use several Agents.

One Agent may support several persona surfaces.

The distinction is:

```text
Persona
→ how assistance is presented

Agent identity
→ which governed runtime actor performed the operation
```

A friendly persona must not appear to be:

- a licensed attorney;
- the user;
- a human employee;
- a provider;
- a final decision-maker.

## 5. Assistant Continuity Must Be Contextual

The Assistant may maintain continuity across interactions.

But continuity must be scoped.

Possible context includes:

- current organization;
- current user;
- role;
- Product;
- selected client;
- Matter;
- current goal;
- approved preferences;
- recent accepted work;
- authorized private Knowledge;
- pending review.

The Assistant must not assume that previous context remains valid.

A context change may require:

- clearing client scope;
- reassembling AI Context;
- changing available Skills;
- removing private Knowledge;
- changing output mode;
- requiring new Permission;
- invalidating a prepared action.

Continuity means coherent transition.

It does not mean perpetual context accumulation.

The personal AI clone may imitate style.

It does not inherit human identity or professional accountability.

```text
Style continuity
≠
identity substitution

Preference learning

```

## 7. Assistant Memory Is Not Unlimited Memory

Assistant memory may include authorized:

- preferences;
- recent context references;
- accepted corrections;
- user-selected defaults;
- interaction summaries;
- reusable private instructions.

It must remain distinct from:

- raw conversation archive;
- full organizational memory;
- all client history;
- every Document;
- unrestricted Product history;
- shared Knowledge.

Memory must have:

- scope;
- source;
- purpose;
- owner;
- validity;
- correction;
- deletion;
- access boundary.

The Assistant should use dynamic retrieval where possible rather than copying all available information into permanent memory.

## 8. Assistant May Explain but Must Not Conceal Uncertainty

The Assistant may simplify complex professional context.

It should still disclose material:

- source;
- uncertainty;
- missing information;
- conflicting Knowledge;
- AI participation;
- review requirement;
- downstream service requirement.

A confident conversational tone must not conceal uncertainty.

The Assistant should be able to say:

```text
The current official requirement is not verified.

Two sources conflict.

The provider fee remains unconfirmed.

This is a draft recommendation.

Human review is required before use.
```

Natural interaction must not become false certainty.

## 9. Assistant May Prepare but Must Not Silently Execute

Assistant operations may include:

- prepare a draft;
- prepare a Task suggestion;
- prepare a Communication;
- prepare a quotation candidate;
- prepare a filing package;
- prepare a provider comparison;
- prepare a Workflow preview.

Preparation must remain visible as preparation.

The Assistant must not silently:

- create an active Task;
- send a Communication;
- appoint a provider;
- accept client instructions;
- file an application;
- approve a payment;
- change Matter state;
- publish content.

The handoff from preparation to action is developed in CH18.

## 11. Guide Is a Goal-Specific Structured Journey

Guide means:

> a bounded user-facing interaction that helps a person reach a defined goal through structured questions, explanations, choices, validations, progress, and governed handoffs.

Examples include:

- choose filing jurisdictions;
- select trademark classes;
- prepare an application;
- respond to an office action;
- compare service providers;
- prepare an evidence package;
- create a client proposal;
- review a renewal portfolio.

A Guide differs from the open Assistant because it has:

- a defined goal;
- entry conditions;
- required inputs;
- structured stages;
- completion criteria;
- exit paths.

---

## 12. Guide Reduces Questions Through Context and Intelligence

A professional Guide should not simply reproduce a long form as a chat.

It should use authorized context to reduce unnecessary questions.

The Guide may:

- reuse confirmed client data;
- retrieve organization preferences;
- infer candidates;
- explain options;
- present structured choices;
- request only missing material facts;
- show cost and consequence;
- prepare a recommendation.

The intended pattern is:

```text
Understand existing context
→ identify missing decisions
→ ask the minimum useful questions
→ explain tradeoffs
→ present choices
→ prepare the next governed step
```

A Guide may be conversational.

Its architecture remains structured.

---

## 13. Guide Is Not a Questionnaire

A questionnaire primarily collects answers.

A Guide helps the user make progress toward a goal.

It may include:

- context reuse;
- professional explanation;
- dynamic branching;
- option comparison;
- recommendation candidates;
- validation;
- progress;
- prepared outputs.

A static form can be part of a Guide.

It is not the full Guide.

The distinction matters because MarkReg should not reduce AI-assisted filing intake to a conversational version of a traditional form.

---

## 14. Guide Is Not Workflow

A Guide is a user-facing navigation and decision-support structure.

A Workflow is governed multi-step execution with controlled state, actors, Tasks, reviews, and service handoffs.

A Guide may:

- collect intent;
- prepare inputs;
- explain Workflow;
- preview next steps;
- launch a governed Workflow request.

It must not silently become the Workflow runtime.

Guide progress such as:

```text
Step 3 of 6
```

does not prove:

- Task status;
- Matter progress;
- review completion;
- filing state;
- Communication delivery.

The Guide can project formal state.

It does not own it.

---

## 15. Guide Progress Is Presentation State

Guide progress may include:

- not started;
- current stage;
- completed question;
- skipped optional step;
- saved draft;
- ready for review.

These are Guide-level states.

They must not redefine Core or Execution lifecycle.

A Guide may say:

```text
Your intake is complete.
```

This means the Guide has collected the required intake information.

It does not necessarily mean:

- an Order exists;
- a Matter exists;
- documents are verified;
- the application is ready to file;
- professional review is complete.

The interface should identify the exact meaning of completion.

A Guide should end with an identifiable result.

Possible results include:

- context package;
- recommendation candidate;
- completed intake draft;
- document checklist;
- quotation candidate;
- prepared action;
- Product handoff;
- review request;
- Workflow application request;
- safe refusal.

The result should preserve:

- Guide identity;
- Guide version;
- goal;
- user;
- organization;
- selected options;
- source references;
- missing context;
- Agent participation;
- review requirement;
- downstream service.

Guide completion must not be represented as formal execution unless the relevant service confirms it.

## 17. Guide Branching Must Remain Explainable

A Guide may branch based on:

- user answer;
- Product state;
- client context;
- Knowledge;
- risk;
- Policy;
- recommendation;
- missing information.

Material branching should be explainable.

For example:

```text
This additional POA step appears because:
- the selected jurisdiction requires it;
- the applicant is a legal entity;
- the current service type is assignment;
- the organization’s rule requires review.
```

The system need not expose internal code.

It should explain professional consequence.

## 18. Guide Recommendations Remain Candidates

A Guide may recommend:

- jurisdictions;
- classes;
- goods and services;
- service package;
- provider candidates;
- evidence;
- next step.

These remain recommendations or candidates.

A user’s selection may establish intent.

It does not automatically create:

- an Order;
- a Matter;
- a filing instruction;
- a provider appointment;
- a formal Opportunity;
- a professional approval.

CH17 defines candidates and Next Best Action in detail.

## 19. Guide May Use Deterministic and AI Components

A Guide does not require AI for every step.

It may combine:

- deterministic rules;
- validated Knowledge;
- Product logic;
- human-authored explanation;
- AI extraction;
- AI recommendation;
- local Skills;
- external data;
- Human Review.

The architecture should choose AI only where it improves value.

Deterministic rules may be preferable for:

- required fields;
- status mappings;
- known fee formulas;
- fixed validation;
- controlled eligibility.

AI may be useful for:

- explaining;
- summarizing;
- classifying;
- comparing;
- drafting;
- identifying missing context.

## 20. AI Agent Is a Governed Runtime Role

AI Agent means:

> a non-human governed runtime actor that performs declared AI-assisted operations under an explicit Agent identity, purpose, Agent Contract, Capability profile, Skill access, data scope, Permission, Policy, Human Review, and audit boundary.

An AI Agent is not merely:

- a model;
- a prompt;
- a chatbot;
- a Tool;
- a Product feature;
- a background script.

The Agent identity allows MarkOrbit to answer:

```text
Which Agent acted?

For which organization?

For which purpose?

Using which Capability and Skill?

With access to which data?

Under which Permission and Policy?

What output mode applied?

What Human Review was required?

Which downstream service was needed?
```

No governed AI or automation behavior should exist outside the Agent Registry.

Registration should identify:

- registry key;
- Agent type;
- Agent role;
- purpose;
- allowed Capabilities;
- restricted Capabilities;
- default data access;
- allowed data access;
- default execution mode;
- review triggers;
- required services;
- Permission;
- Policy;
- status.

Registration is a governance prerequisite.

It is not runtime authorization.

The rule is:

```text
Registry declares eligibility.

Permission and Policy decide contextual use.
```

## 22. Agent Identity Must Not Be Hidden Behind Human Identity

An AI Agent must not operate as though it were the human user.

The operation may involve:

- human actor;
- Agent;
- organization;
- target object.

These references must remain distinguishable.

A user may request an action.

The Agent may prepare or perform an allowed operation.

The result should preserve both identities.

The Agent must not:

- use the human’s identity as its own;
- imply that the human personally authored unreviewed output;
- impersonate a provider;
- appear as a hidden employee;
- fabricate a reviewer identity.

## 25. Agent Contract Defines the Operational Boundary

Agent Contract should define:

- Agent identity;
- purpose;
- eligible Capabilities;
- allowed Skills;
- input types;
- output modes;
- data access;
- prohibited data;
- required Knowledge;

- Policy;
- Human Review;
- downstream services;

## 26. Agent Context Must Be Assembled Per Operation

An Agent should not retain unrestricted global context.

Each operation should assemble:

- actor;
- organization;
- Product;
- purpose;
- subject;
- Capability;
- Skill;
- permitted sources;
- data access;
- output mode;
- review requirement;
- expiry.

The same Agent may receive different context for different operations.

The Agent should not carry client or Matter data into a later unrelated request.

The principle is:

```text
Agent identity may persist.

Operational context must be purpose-specific.
```

## 27. Agent Data Access Follows Least Privilege

The default access should be narrow.

Conceptual access modes include:

- NoAccess;
- MetadataOnly;
- SafeSummaryOnly;
- PolicyFilteredContent;
- RestrictedContentWithHumanApproval.

The Agent should not access all Knowledge, Documents, Evidence, pricing, contacts, or Communications by default.

A human user’s access does not imply Agent access.

Policy may further narrow a permitted scope.

The Agent must not receive:

- raw credentials;
- API keys;
- payment credentials;
- privileged strategy;
- unrestricted Evidence;
- unrelated client content.

## 33. Agent May Request Action but Must Not Self-Authorize

An Agent may identify that a downstream action is needed.

It may prepare a request.

- grant itself Permission;
- change Policy;
- expand its data access;
- select a different Agent with broader authority;
- approve its own output;
- mark Human Review complete;

Escalation must route to:

- user;
- reviewer;
- administrator;
- Policy authority;
- owning service;
- governed Workflow.

## 34. User Confirmation Is Not Always Human Review

A user may click:

- Continue;
- Accept;
- Use this;
- Looks good;
- Submit for review.

These interactions may establish user intent.

They do not automatically establish professional Human Review.

Human Review requires:

- eligible reviewer;
- defined scope;
- reviewed version;
- review decision;
- timestamp;
- trace.

For low-risk personalization, simple confirmation may be sufficient.

For professional conclusions or protected actions, the system must not label ordinary confirmation as review.

---

## 35. Human Review Must Be Linked to the Output Reviewed

An Agent output may change after review.

Material changes may include:

- source;
- recipient;
- fee;
- deadline;
- class;
- goods and services;
- legal position;
- provider;
- attachment;
- recommendation basis.

Review applies to a specific version and purpose.

A revised output may require new review.

The Assistant or Guide should display the review state without becoming its authority.

---

## 36. Assistant and Guide Must Expose Review Boundaries

The user should know:

- whether the current output is a draft;
- whether professional review is required;
- who may review;
- what version is being reviewed;
- what happens after approval;
- which service will act.

A polished interface must not conceal that review remains pending.

Examples:

```text
Prepared for internal review

Approved for client presentation

Ready for Communication Service send preparation

Not approved for filing
```

These labels should preserve exact meaning.

## 37. Agent Errors Must Fail Safely

Agent operations may fail because of:

- PermissionDenied;
- PolicyRestricted;
- AgentSuspended;
- AgentContractRequired;
- CapabilityNotAllowed;
- HumanReviewRequired;
- RestrictedDataRequested;
- KnowledgeAccessRestricted;
- invalid context;
- unavailable Skill;
- missing source.

Failure should not trigger an uncontrolled fallback.

The system may:

- narrow scope;
- request clarification;
- select another eligible Skill;
- return a partial result;
- require human handling;
- refuse safely.

Error messages must not expose restricted policy, hidden prompts, secrets, or private data.

- Permission;
- Policy;

## 40. Multi-Agent Operation Must Not Hide Responsibility

A request may involve multiple Agents.

For example:

```text
KnowledgeRetrievalAssistant
→ retrieves sources

ClassificationAssistant
→ prepares classification result

DraftingAssistant
→ prepares explanation

AuditSupportAssistant
→ prepares trace summary
```

The user may see one coherent output.

The system should preserve:

- participating Agents;
- operations;
- sources;
- handoffs;
- final output assembly;
- review requirement.

Multi-Agent orchestration must not create one untraceable collective actor.

No Agent should assume that another Agent completed required review.

## 43. Failure Modes This Chapter Prevents

### Persona authority

A friendly Assistant persona appears to be a lawyer, employee, provider, or final decision-maker.

### Super-agent architecture

One Agent receives every Capability, Skill, and data source.

### Hidden-agent execution

AI operations occur under the human user’s identity with no Agent trace.

### Conversation-as-permission

The user asking for something is treated as authorization to perform it.

### Memory accumulation

The Assistant permanently accumulates every client, Matter, and conversation.

### Cross-organization carryover

A personal AI clone carries one organization’s private context into another.

### Guide-workflow collapse

A guided screen directly controls formal lifecycle state.

### Questionnaire masquerading as AI

A long form is converted into chat without meaningful context reuse or intelligence.

### Recommendation-decision collapse

An AI recommendation becomes a filing instruction, provider appointment, or formal choice.

### Confirmation-review collapse

A button click is recorded as professional Human Review.

### Preparation-execution collapse

A draft or prepared package is represented as sent, filed, paid, or completed.

### Local-agent exemption

A local model is given unrestricted Local Vault access.

### Cloud-context flooding

A cloud Agent receives the entire Workplace context.

### Multi-agent opacity

Several Agents contribute to an output without trace or responsibility boundaries.

### Unsafe fallback

A blocked Agent operation silently routes to a broader Agent or data scope.

### Review drift

Reviewed output changes without new review.

These designs may feel seamless.

They do not conform to MarkOrbit architecture.

## 46. Chapter Conclusion

Assistant, Guide, and AI Agent solve different problems.

The Assistant provides continuity.

The Guide provides direction toward a defined goal.

The AI Agent provides a governed runtime identity for bounded AI-assisted operations.

The correct relationship is:

```text
Assistant
→ helps the user continuously

Guide
→ structures a specific journey

AI Agent
→ performs declared operations

Skill
→ implements the required Capability

Knowledge
→ grounds professional meaning

Human Review
→ protects accountable judgment

Execution
→ governs protected action

Owning Service
→ changes formal business facts
```

This separation allows MarkOrbit to present a coherent AI-assisted Workplace without creating an invisible autonomous actor.

The user may experience one helpful companion.

The architecture still knows:

- which Agent acted;
- which Skill was used;
- what data was accessed;
- what Knowledge grounded the result;
- what remained missing;
- what review was required;
- which service must act next.

The constitutional outcome is:

```text
Assistant may accompany.

Guide may direct.

AI Agent may assist.

Human must remain responsible.

Execution must govern action.
```

CH17 now addresses the output of observation, Knowledge, Intelligence, Capabilities, Skills, Assistants, Guides, and Agents:

> How do signals become Value Candidates, recommendations, and Next Best Actions without silently becoming formal Tasks, Opportunities, provider selections, client instructions, or approved decisions?
