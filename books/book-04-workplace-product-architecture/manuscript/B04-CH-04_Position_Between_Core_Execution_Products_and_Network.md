# B04-CH-04 — Position Between Core, Execution, Products and Network

**Status:** Draft 1  
**Chapter Map:** B04-TOC-V0.1  
**Writing Pack:** B04-PACK-01 — Front Matter and Part I

## Chapter Purpose

This chapter defines the precise position of **Workplace** between MarkOrbit Core, governed Execution, Products, and the Mark Global Service Network.

The main architectural problem is not whether these parts exist.

The problem is whether their responsibilities remain distinct.

When the boundaries are unclear, one layer begins to absorb another:

- Core becomes too procedural;
- Execution becomes a product shell;
- Workplace becomes a task engine;
- Products become owners of organizational truth;
- MGSN becomes a central marketplace;
- AI Agents become informal authorities;
- Owning Services lose control of formal business state.

Book 04 exists to prevent that collapse.

The constitutional spine is:

```text
Industry and Professional Philosophy
                ↓
MarkOrbit Operating System
                ↓
MarkOrbit Core
                ↓
MarkOrbit Execution System
                ↓
Organization Workplace / Independent Orbit
                ↓
Products and Operating Models
                ↓
Mark Global Service Network
                ↓
Orbital Ecosystem Learning and Evolution
```

This sequence does not mean that every layer is a software service or that every layer executes after the previous one in time.

It means that each layer answers a different architectural question.

```text
Core:
What does the shared professional model mean?

Execution:
How does governed work proceed?

Workplace:
Which organization is operating,
under which identity, authority, context, and boundaries?

Product:
How does a user experience and complete a domain journey?

MGSN:
How do independent Workplaces discover, select,
and collaborate with one another?

Owning Service:
Which authority may change formal business facts?

Human Professional:
Who remains accountable for professional judgment?

AI Agent:
Which governed runtime role may assist,
under which identity, permission, and context?
```

The central proposition of this chapter is:

```text
Core defines.
Execution coordinates.
Workplace contextualizes.
Products compose.
MGSN connects.
Owning Services formalize.
Humans remain accountable.
AI assists under governance.
```

## 1. Why Positioning Matters

Architecture often fails through responsibility drift.

A team begins with a practical need.

A Product needs a new field, so the Product defines a new object.

A Workplace needs a faster approval flow, so the Workplace creates its own task states.

A network needs provider status, so the network invents a parallel capability model.

An AI Agent needs to complete an action, so the Agent writes directly to formal records.

Each decision may appear efficient in isolation.

Together, they fragment the operating system.

MarkOrbit must therefore distinguish between:

```text
where meaning is defined;
where work is coordinated;
where organizational context is supplied;
where user experience is composed;
where organizations connect;
where formal state changes;
where professional responsibility remains.
```

Positioning is not an editorial exercise.

It determines:

- who owns truth;
- who may mutate state;
- where governance applies;
- where product variation is allowed;
- where organizational autonomy is preserved;
- where AI may assist;
- where Human Review is mandatory;
- how cross-product consistency is maintained;
- how network participation remains safe.

A clear architecture does not eliminate overlap in use.

Workplace may display Tasks.

Products may initiate Workflows.

MGSN may produce recommendations.

AI Agents may prepare Artifacts.

But use is not ownership.

Visibility is not authority.

Invocation is not definition.

The whole chapter depends on that distinction.

## 2. Core Defines Shared Professional Meaning

MarkOrbit Core is the shared semantic foundation.

It defines the professional concepts that must remain consistent across Products, Workplaces, Execution patterns, and network interactions.

Core may define:

- domains;
- Core Objects;
- Core Services;
- service ownership;
- statuses;
- transition matrices;
- Workflow Contract components;
- API Contracts;
- Status Contracts;
- Common Contracts;
- permission and policy boundaries;
- Human Review boundaries;
- Event and audit rules;
- AI governance boundaries;
- validation rules.

Core answers questions such as:

```text
What is a Matter?
What is a Task?
What is a Communication?
What is a Document?
What is a Review?
What is an Event?
Which Service owns this Object?
Which status values are valid?
Which transition is permitted?
Which actor may perform which mutation?
```

Core must remain stable enough to support multiple products and organizations.

That stability is why organization-specific behavior must not be pushed into Core casually.

An organization may require two review steps before sending a client communication.

Another may require one.

A third may require escalation for a specific client type.

Those are important operating differences.

They do not automatically justify new universal Core semantics.

The rule is:

```text
Core defines shared meaning.
Core does not contain every organization’s operating preference.
```

Workplace consumes Core.

Products consume Core.

Execution coordinates Core contracts.

MGSN references Core concepts when collaboration requires them.

None of those layers may silently redefine Core.

## 3. Execution Coordinates Governed Work

Execution exists because valid semantics do not automatically produce safe operation.

A Task may exist, but it may not be ready to complete.

A Communication may exist, but it may not be approved to send.

A Workflow may be available, but it may not be permitted to advance.

An AI draft may be useful, but it is not Human Review.

Execution coordinates the movement from valid structure to governed action.

It may govern:

- workflow progression;
- task lifecycle;
- review and approval gates;
- communication boundaries;
- permission and policy checks;
- protected-action controls;
- Human–AI handoff;
- safe failure;
- retry;
- idempotency;
- audit;
- trace;
- replay;
- escalation;
- execution completion.

Execution answers:

```text
When does work begin?
What must happen next?
What must pause?
Who must review?
Which condition blocks progress?
When may a protected action proceed?
How is failure recorded?
How is the result traced?
```

Execution does not own the organization.

It does not own:

- organizational identity;
- client relationships;
- private knowledge;
- pricing;
- commercial rules;
- product composition;
- partner networks;
- Workplace branding.

It uses organizational context supplied by Workplace.

The distinction is:

```text
Workplace supplies the authorized context.
Execution applies governed coordination.
```

If Execution absorbed Workplace, the execution layer would become an organization-management platform.

If Workplace absorbed Execution, each Workplace edition could invent different Workflow, Task, Review, and protected-action semantics.

Both outcomes would break the architecture.

## 4. Workplace Supplies Authorized Organizational Context

Workplace is the unified professional operating environment of an organization.

It is the primary organizational operating unit of the MarkOrbit ecosystem.

Workplace carries the context that shared semantics and governed execution require in order to become legitimate organizational work.

That context includes:

- organization identity;
- users;
- roles;
- permissions;
- accountability;
- clients;
- relationships;
- private knowledge;
- business rules;
- pricing;
- preferences;
- service catalog;
- partner relationships;
- AI context;
- product access;
- work visibility;
- review visibility;
- organizational memory.

Workplace answers:

```text
Which organization is operating?
Which user is acting?
Which role do they hold?
Which client context applies?
Which private knowledge may be used?
Which pricing and service rules apply?
Which product has access?
Which reviewer is authorized?
Which partner relationships are trusted?
Which records remain local or private?
Which organization remains accountable?
```

Workplace does not define shared semantics.

It does not define Workflow meaning.

It does not define Task lifecycle.

It does not own every Product feature.

It does not automatically own formal business state.

It supplies the organizational setting in which those things are consumed.

The most concise distinction is:

```text
Core defines the shared professional model.
Execution governs movement through work.
Workplace establishes whose organization is operating.
```


MGSN connects Workplaces.

## 7. Owning Services Change Formal Business Facts

Formal business facts require clear mutation authority.

That authority belongs to an Owning Service.

An Owning Service is responsible for changing formal state within its domain.

Examples may include:

- creating or updating a formal Matter;
- changing a Task status;
- recording an approved Communication send;
- recording payment state;
- confirming an official filing event;
- recording provider instruction;
- updating a formal Opportunity;
- recording an approved Document or Artifact outcome.

The key principle is:

```text
Only the proper Owning Service changes formal business facts.
```

This protects the system from informal mutation.

Workplace may collect context.

Product may collect user intent.

Execution may coordinate the process.

AI Agent may prepare a proposal.

MGSN may recommend a provider.

Human Professional may approve a decision.

But the formal mutation still belongs to the Owning Service.

This distinction is especially important in AI-assisted systems.

Without it, an AI Agent may appear to complete work simply because it generated a plausible output.

But generation is not mutation.

Recommendation is not recordal.

Approval is not execution.

The correct line is:

```text
AI prepares.
Human reviews.
Execution coordinates.
Owning Service records the formal fact.
```

| Concept | Primary responsibility | May consume or expose | Must not own |

| Execution | Governed coordination, workflow progression, Task lifecycle, review gates, protected-action controls, trace | Workplace context, Core contracts, Product entry points, Human Review | Organization identity, private knowledge, product UX, client ownership |

This matrix is not an implementation topology.

It is an authority map.

A future system may place several responsibilities in one codebase.

Another may separate them into multiple services.

The technical form may vary.

The responsibility boundaries must remain.

## 12. Visibility Does Not Equal Ownership

A recurring source of confusion is the assumption that the layer displaying something owns it.

That is false.

Workplace may display a Task.

Execution owns the Task lifecycle.

Product may display a provider recommendation.

MGSN produces the candidate and explanation.

Product may display a Matter.

The relevant Owning Service owns the formal Matter state.

Workplace may show a Communication draft.

Execution governs the review and send boundary.

A dashboard may display an Event.

The proper Owning Service emits or records the Event.

The rule is:

```text
Visible in a surface
≠
owned by the surface
```

This rule is essential for Product and Workplace architecture.

Without it, user-facing layers accumulate authority simply because they are where people interact.

The architecture must resist that drift.

## 15. Product Variation Must Not Create Semantic Drift

Products need freedom to innovate.

Lite may use a Today Feed.

MarkReg may use a lifecycle journey.

An enterprise Workplace may use portfolio views.

MGSN may use network-oriented interfaces.

That variation is desirable.

But Product variation must not create semantic variation in shared concepts.

A Task must not mean one thing in Lite and another in MarkReg.

Human Review must not disappear in one Product because the interface is simplified.

A Communication draft must not become equivalent to an approved send in one Product.

A provider recommendation must not become an appointment in one Product.

The rule is:

```text
Product experience may vary.
Shared semantics and authority boundaries may not drift.
```

Products compose.

They do not redefine.

## 18. The Cross-Layer Handoff Model

The architecture becomes clearer when viewed as a handoff model.

Consider a client need.

### Step 1 — Workplace supplies context

The organization, user, client, private knowledge, pricing rules, permissions, and partner history are known.

### Step 2 — Product composes the journey

The user sees a relevant flow, recommendation, or form.

### Step 3 — AI and capabilities prepare

The system may analyze, compare, draft, or recommend.

### Step 4 — Execution coordinates

Tasks, reviews, approvals, retries, and protected-action gates are governed.

### Step 5 — Human Professional decides

Professional judgment and approval remain accountable.

### Step 6 — Owning Service records

The formal business fact is changed by the proper authority.

### Step 7 — MGSN connects when external capability is needed

Candidates may be identified, explained, and selected.

### Step 8 — Outcome returns

The outcome becomes visible to Workplace, Product, audit, and governed learning.

The flow can be summarized as:

```text
Workplace context
→ Product journey
→ AI/capability preparation
→ Execution
→ Human Review
→ Owning Service
→ Outcome
→ Workplace and ecosystem feedback
```

This is not one linear implementation pipeline.

It is an authority-preserving interaction model.

## 23. Architectural Tests for New Features

A new feature should be tested against the responsibility model.

Ask:

### Meaning test

Is this feature defining a new shared professional concept?

If yes, it may belong in Core or require a formal change process.

### Coordination test

Is this feature governing how work progresses, pauses, reviews, or completes?

If yes, it may belong in Execution.

### Context test

Is this feature supplying organization identity, permissions, private knowledge, clients, rules, or preferences?

If yes, it may belong in Workplace.

### Journey test

Is this feature shaping a user-facing domain experience?

If yes, it may belong in a Product.

### Network test

Is this feature discovering or connecting external capabilities?

If yes, it may belong in MGSN.

### Mutation test

Is this feature changing formal business state?

If yes, the proper Owning Service must remain authoritative.

### Accountability test

Does this feature make or approve a professional judgment?

If yes, Human responsibility must remain explicit.

### AI test

Is an AI Agent involved?

If yes, identity, permission, context, limits, review, and audit must be defined.

These tests prevent implementation convenience from rewriting architecture.
