# B04-CH-17 — Value Candidates, Recommendations and Next Best Action

**Status:** Draft 1  
**Chapter Map:** B04-TOC-V0.1  
**Part:** Part III — Knowledge, Intelligence and Capability Consumption

## Chapter Purpose

Chapter 13 defined how Workplace assembles authorized professional context.

Chapter 14 defined how governed Knowledge is consumed through the logical Brain.

Chapter 15 defined how Capability is discovered and a concrete Skill is selected.

Chapter 16 defined Assistant, Guide, and AI Agent as distinct user-facing and runtime roles.

This chapter defines what those layers may produce before governed action begins:

- observations;
- signals;
- Value Candidates;
- recommendations;
- Next Best Action candidates;
- user selections;
- prepared-action intentions.

The central question is:

> How does Workplace identify possible value and recommend what may deserve attention next without silently creating formal Tasks, Opportunities, routing decisions, client instructions, professional conclusions, or approved actions?

Professional systems observe many things:

- an approaching renewal;
- a missing document;
- an unanswered client message;
- a new trademark filing need;
- a possible class expansion;
- a provider response delay;
- a changed official fee;
- a high-risk Evidence gap;
- a potential commercial opportunity;
- a stalled review;
- a source conflict;
- a draft ready for attention.

These observations may create value.

They may also be wrong, duplicated, stale, irrelevant, commercially inappropriate, or outside the user’s authority.

The system therefore requires a candidate-first value architecture.

The central chain is:

```text
Observation
→ Signal
→ Value Candidate
→ Recommendation
→ Next Best Action Candidate
→ Human Selection or Review
→ Prepared Action
→ Governed Execution
→ Formal Result
```

The constitutional distinctions are:

```text
Observation ≠ fact

Signal ≠ formal work

Value Candidate ≠ Opportunity

Recommendation ≠ decision

Next Best Action ≠ Task

Priority score ≠ formal Priority

Ranking ≠ authorization

User interest ≠ client instruction

User confirmation ≠ professional approval

Provider recommendation ≠ Routing decision

Commercial potential ≠ accepted engagement

Prepared Action ≠ execution

Expected value ≠ guaranteed outcome

Model confidence ≠ professional certainty

No recommendation ≠ no risk

Dismissal ≠ formal cancellation
```

Value Factory may produce derived-value candidates.

Mo Intelligence may score, match, rank, recommend, and explain.

Workplace provides organization, client, user, Product, and responsibility context.

Humans remain accountable for consequential choices.

Execution coordinates protected action.

Owning Services create and mutate formal business facts.

Book 02 remains authoritative for Opportunity, Task, Routing, Permission, Policy, Priority, and other Core concepts.

Book 03 remains authoritative for Workflow, Task lifecycle, Human Review, Communication, and protected-action execution.

This chapter defines the pre-formal value and attention layer.

It does not redefine Core or Execution.

---

## 1. Value Begins as a Possibility

A professional system should identify possible value before requiring a formal object.

Examples include:

- a renewal that may require client contact;
- an application status that may justify follow-up;
- an unused Capability that may improve delivery;
- a possible cross-border filing need;
- an Evidence gap that may weaken a case;
- an unanswered provider email;
- a recurring organization error that may justify a checklist;
- a client portfolio pattern that may justify review.

At the moment of detection, the system may not yet know:

- whether the observation is correct;
- whether the client wants action;
- whether a service need exists;
- whether a formal Task is required;
- whether the user has authority;
- whether the issue is already handled elsewhere;
- whether the expected benefit exceeds cost.

The system should therefore represent possibility honestly.

The first rule is:

```text
Possible value
must not be presented
as an already accepted obligation.
```

---

## 2. Observation Is the Initial Perception

Observation means:

> a recorded perception that something relevant may have occurred, changed, remained absent, or deserves examination.

An observation may come from:

- Event;
- source acquisition;
- official record;
- Communication;
- user input;
- Product state;
- local data;
- AI extraction;
- monitoring;
- historical comparison;
- human note.

Examples:

```text
No client reply recorded for fourteen days.

A renewal date appears within six months.

The selected goods may exceed the jurisdiction limit.

A provider quotation differs from the stored cost.

The filing package lacks a signed POA.
```

An observation should preserve:

- source;
- time;
- subject;
- observer;
- extraction or detection method;
- uncertainty;
- applicable scope.

Observation does not prove professional significance.

---

## 3. Signal Is an Interpreted Observation

Signal means:

> an observation interpreted as potentially relevant to a defined professional, operational, commercial, risk, or learning purpose.

A signal adds context such as:

- why the observation may matter;
- who may be affected;
- which Product or service is relevant;
- which time window applies;
- which risk or value dimension may exist.

Examples:

```text
Observation:
No reply for fourteen days.

Signal:
Client follow-up may be overdue under the organization’s relationship rule.
```

```text
Observation:
Renewal date is within six months.

Signal:
A maintenance service need may require qualification.
```

A signal remains pre-formal.

It may be:

- confirmed;
- weak;
- duplicated;
- suppressed;
- expired;
- unresolved.

---

## 4. Signal Is Not Event

An Event records something that occurred in the governed system or an accepted external observation.

A signal is an interpretation that the occurrence may deserve attention.

Example:

```text
Event:
Official status changed.

Signal:
The change may create a deadline or client Communication need.
```

One Event may create several signals.

Several Events may contribute to one signal.

A signal may also arise from absence or comparison rather than one Event.

The distinction is:

```text
Event preserves occurrence.

Signal proposes relevance.
```

Signal creation must not fabricate Events.

---

## 5. Signal Is Not Notification

A Notification communicates selected information to a recipient.

A signal is the underlying candidate relevance.

The system may decide:

- not to notify;
- to group several signals;
- to delay notification;
- to show the signal only in Today;
- to escalate it;
- to create a review candidate.

Notification policy should consider relevance, urgency, role, duplication, and interruption cost.

The existence of a signal does not require immediate interruption.

---

## 6. Value Candidate Represents a Possible Beneficial Response

Value Candidate means:

> a governed pre-formal proposition that a defined response, service, correction, decision, learning action, or attention shift may create professional, commercial, operational, risk-reduction, or organizational value.

A Value Candidate may concern:

- missed-work prevention;
- professional judgment;
- service conversion;
- delivery quality;
- organizational learning;
- risk reduction;
- client relationship;
- cost or time efficiency.

Examples:

```text
Contact the client about a possible renewal service.

Review the Evidence gap before the deadline.

Update the organization checklist because the rule changed.

Compare alternative providers for the current Matter.

Prepare a draft response to the unanswered Communication.

Launch a class-selection Guide.
```

The candidate identifies possible value.

It does not assert that the response is approved.

---

## 7. Value Candidate Is Not Formal Opportunity

A Value Candidate may suggest possible demand.

A formal Opportunity is a Core-recognized potential service need with its own identity, source, status, qualification, and conversion lifecycle.

The path may be:

```text
Signal
→ commercial Value Candidate
→ user or professional review
→ qualification request
→ Opportunity Service
→ formal Opportunity or rejection
```

The system must not create formal Opportunities silently from every:

- deadline;
- client click;
- Product interaction;
- AI suggestion;
- status change;
- marketing signal.

A Value Candidate may be dismissed without ever becoming an Opportunity.

A formal Opportunity may later exist independently of the original recommendation.

---

## 8. Value Candidate Is Not Task Suggestion

A Task Suggestion proposes that active work may be needed.

A broader Value Candidate may instead propose:

- learning;
- recommendation;
- client discussion;
- commercial qualification;
- Product handoff;
- risk review;
- information acquisition.

Where work is appropriate, the candidate may produce a Task plan or Task creation request.

It does not create an active Task.

The sequence is:

```text
Value Candidate
→ possible work response
→ Task Suggestion
→ validation and governance
→ Task Service request
→ active Task or safe rejection
```

---

## 9. Value Candidate Is Not Professional Conclusion

A Value Candidate may be generated from:

- rules;
- AI;
- scoring;
- patterns;
- Knowledge;
- historical outcomes.

It may state:

```text
Potentially high risk.

Possible class conflict.

Likely renewal opportunity.

Evidence may be insufficient.
```

These are not final professional conclusions.

The user should know:

- what was observed;
- why value may exist;
- what uncertainty remains;
- which source supports the candidate;
- which review is required.

---

## 10. Value Dimensions Must Be Explicit

A candidate should identify the type of value it may create.

Canonical value tests include:

```text
Reduce Missed Work

Improve Professional Judgment

Accelerate Service Conversion

Improve Delivery Quality

Accumulate Organizational Capability
```

Additional contextual dimensions may include:

- reduce legal risk;
- reduce operational cost;
- improve client trust;
- improve response speed;
- preserve confidentiality;
- reduce rework.

A candidate may support several dimensions.

The system should not reduce all value to revenue.

A high-revenue recommendation may be inappropriate if it harms client interest or professional responsibility.

---

## 11. Value Candidate Requires a Beneficiary

The candidate should identify who may benefit:

- client;
- organization;
- responsible professional;
- team;
- network participant;
- Product user;
- broader ecosystem.

A recommendation that benefits the platform but not the client or organization should not be disguised as professional value.

Potential conflicts should be visible.

Examples:

```text
Client benefit:
avoid renewal loss

Organization benefit:
service conversion

Provider benefit:
new instruction

Platform benefit:
network usage
```

The recommendation should preserve priority of professional and client obligations over hidden commercial optimization.

---

## 12. Value Candidate Requires a Subject

The candidate should be linked to a defined subject such as:

- client;
- trademark;
- Brand;
- Matter;
- Communication;
- Document;
- Evidence;
- provider relationship;
- organization process;
- Product journey;
- Knowledge asset.

A generic recommendation without subject can be difficult to validate or act upon.

Stable references help prevent:

- duplicate candidates;
- cross-client contamination;
- irrelevant ranking;
- ambiguous dismissal;
- false urgency.

---

## 13. Candidate Source and Provenance Must Remain Visible

A candidate may originate from:

- official-source change;
- Event;
- user observation;
- AI Agent;
- deterministic rule;
- Mo Intelligence;
- Value Factory;
- Product analytics;
- provider Communication;
- organization memory;
- network outcome.

The source should remain identifiable.

A user should be able to distinguish:

```text
Official-source-triggered candidate

Organization-rule candidate

AI-inferred candidate

Historical-pattern candidate

Commercial campaign candidate

Human-created candidate
```

The origin affects trust, review, and allowed use.

---

## 14. Value Factory Produces Derived Candidates

Value Factory may transform structured information into derived-value candidates.

It may identify:

- likely service need;
- missed-work risk;
- possible deadline;
- data inconsistency;
- portfolio pattern;
- provider performance concern;
- content opportunity;
- organizational learning opportunity.

Value Factory does not create formal business facts.

Its output should preserve:

- input sources;
- transformation method;
- assumptions;
- time window;
- confidence;
- omissions;
- privacy boundary;
- intended value dimension.

The normal route is:

```text
Structured Information
→ Value Factory
→ derived Value Candidate
→ contextual evaluation
→ recommendation or rejection
```

---

## 15. Mo Intelligence Scores, Matches, Ranks, Recommends, and Explains

Mo Intelligence is the logical intelligence responsibility that may:

- score;
- match;
- rank;
- recommend;
- explain.

These are different operations.

### Score

Assigns a contextual measure to a candidate or criterion.

### Match

Compares requirements and attributes.

### Rank

Orders eligible candidates for attention or choice.

### Recommend

Proposes a response or selection.

### Explain

Provides the basis, sources, limitations, and alternatives.

Mo Intelligence does not own:

- formal Priority;
- Opportunity lifecycle;
- Task lifecycle;
- Routing selection;
- professional approval;
- protected action.

---

## 16. Scoring Must Remain Multidimensional

One universal score is usually insufficient.

A candidate may have separate dimensions such as:

```text
urgency

professional risk

client impact

commercial potential

confidence

effort

reversibility

data quality

review burden

strategic fit
```

A high commercial score may coexist with low confidence.

A high urgency score may coexist with low organizational authority.

The system should avoid displaying one number that conceals important tradeoffs.

Where a composite score is used, its meaning and limitations should remain explainable.

---

## 17. Score Is Not Formal Priority

Formal Priority may belong to a Task, Matter, organization rule, or other governed context.

An intelligence score may influence attention.

It does not silently change the formal Priority field.

Example:

```text
Intelligence urgency score:
92

Task Priority:
Normal
```

The mismatch may justify review.

It does not authorize the recommendation engine to alter the Task.

The distinction is:

```text
Attention ranking
≠
formal business Priority
```

---

## 18. Ranking Determines Order of Attention, Not Authority

Ranking may determine what appears first in:

- Today Feed;
- recommendation panel;
- provider candidate list;
- review queue;
- opportunity candidates;
- risk summary.

Ranking does not determine:

- which action is legally required;
- which provider is appointed;
- which client instruction controls;
- which Task is formally highest priority;
- which recommendation is approved.

The user should be able to see important ranking factors without treating the order as mandatory.

---

## 19. Recommendation Proposes a Response

Recommendation means:

> an explainable, context-bound proposal that a user, reviewer, Product, or service should consider a defined response to one or more signals or Value Candidates.

A recommendation should identify:

- subject;
- proposed response;
- value dimension;
- supporting sources;
- assumptions;
- urgency;
- confidence;
- risk;
- alternatives;
- review requirement;
- validity period.

Examples:

```text
Review the renewal record before contacting the client.

Ask the client to confirm ownership before preparing filing.

Compare two eligible providers.

Launch the classification Guide.

Prepare a Communication draft for review.
```

A recommendation is more specific than a candidate.

It remains non-binding.

---

## 20. Recommendation Is Not Decision

A recommendation may be accepted, modified, deferred, rejected, or ignored.

A decision may require:

- user authority;
- professional judgment;
- client instruction;
- reviewer eligibility;
- commercial approval;
- Policy;
- Owning Service.

The system should preserve:

```text
recommended option

selected option

decision-maker

decision basis

formal result
```

These may differ.

Human override is not necessarily an error.

It may reflect information unavailable to the intelligence system.

---

## 21. Recommendation Must Include Alternatives Where Material

A single recommendation may imply false inevitability.

Where material, the system should present:

- recommended option;
- reasonable alternatives;
- do-nothing option;
- consequence of delay;
- uncertainty;
- tradeoffs.

Example:

```text
Option A:
file now with the current list

Option B:
obtain additional goods clarification

Option C:
defer pending brand strategy review
```

The user should know why one option ranks first.

Not every low-risk recommendation requires a full alternative matrix.

The depth should follow consequence.

---

## 22. Recommendation Must Be Time-Bound

A recommendation may become stale because:

- deadline changes;
- client responds;
- source changes;
- provider availability changes;
- price changes;
- Task is created;
- Matter is closed;
- Policy changes;
- another user acts.

A recommendation should have:

- generated time;
- source freshness;
- validity period;
- refresh condition;
- invalidation trigger.

The Today Feed must not continue presenting an obsolete recommendation as current.

---

## 23. Next Best Action Is an Attention Proposition

Next Best Action means:

> the highest-ranked currently eligible proposed response for a defined user, organization, subject, purpose, and time, presented for consideration rather than treated as completed work or binding instruction.

The phrase “best” is contextual.

It may mean best according to:

- risk;
- urgency;
- client value;
- organizational objective;
- professional responsibility;
- expected effort;
- current availability.

Next Best Action should not imply universal optimality.

A safer conceptual form is:

```text
Best currently known eligible action candidate
under the stated context and assumptions.
```

---

## 24. Next Best Action Is Not Active Task

Next Best Action may recommend:

```text
Review the official notice today.
```

An active Task exists only when Task Service creates it.

The user may choose to:

- open the source;
- dismiss the recommendation;
- create or request a Task;
- delegate;
- defer;
- launch a Guide;
- prepare an action.

The UI must not label Next Best Action as “assigned work” unless a formal Task exists.

---

## 25. Next Best Action Is Not Obligation

The recommendation engine may not know:

- contractual duties;
- legal privilege;
- hidden client instruction;
- off-system work;
- professional judgment;
- personal availability;
- emergency context.

A Next Best Action is not automatically:

- legal obligation;
- client mandate;
- manager instruction;
- professional duty;
- Task assignment.

Where a formal obligation exists, the system should link the recommendation to the authoritative source rather than letting ranking create the obligation.

---

## 26. Eligibility Must Precede Ranking

The system should not rank ineligible actions.

Eligibility may require:

- relevant subject;
- current organization;
- user role;
- Permission;
- Policy;
- data access;
- Product entitlement;
- Agent eligibility;
- current source;
- available Capability;
- reviewer availability;
- no unresolved conflict.

The chain is:

```text
Candidate generation
→ validation
→ eligibility filtering
→ ranking
→ recommendation
```

Ranking before eligibility can produce attractive but unusable actions.

---

## 27. Permission and Policy Must Constrain Recommendations

A recommendation may describe a protected operation.

The recommendation engine should know whether the user may:

- view details;
- prepare;
- request review;
- approve;
- execute.

If the user cannot execute, the recommended action may instead be:

```text
Request review from an eligible professional.
```

Permission denial should not hide the existence of a serious risk where safe visibility is required.

It may reduce detail and offer an authorized escalation route.

---

## 28. Professional Responsibility Must Influence Ranking

A recommendation system should not optimize only for:

- revenue;
- conversion;
- engagement;
- clicks;
- platform usage;
- speed.

Professional responsibility may require higher ranking for:

- critical deadlines;
- Evidence integrity;
- confidentiality;
- conflict issues;
- client instructions;
- legal review;
- correction of misleading information.

The architecture should allow professional risk to override commercial attractiveness.

---

## 29. Client Interest Must Not Be Replaced by Platform Interest

MarkOrbit may benefit when users:

- buy services;
- use Products;
- route work;
- generate content;
- engage more frequently.

Those benefits must not be hidden inside professional recommendations.

A commercial recommendation should disclose its nature.

The system must avoid:

- recommending unnecessary services;
- preferring providers because of undisclosed commission;
- suppressing low-revenue but client-beneficial options;
- creating false urgency.

Professional recommendation and commercial promotion must remain distinguishable.

---

## 30. User Preferences May Personalize Attention

Preferences may influence:

- preferred work hours;
- communication style;
- compact or detailed explanations;
- provider shortlist;
- risk tolerance within allowed bounds;
- local processing;
- notification timing;
- preferred Product entry.

Preferences may rank eligible candidates.

They cannot override:

- critical risk;
- Permission;
- Policy;
- client instruction;
- professional obligation;
- formal deadline.

Personalization improves relevance.

It must not hide mandatory or critical items.

---

## 31. Recommendation Diversity Prevents Tunnel Vision

A recommendation system may repeatedly favor one type of action.

Examples:

- always recommend client follow-up;
- always recommend new filing;
- always recommend the cheapest provider;
- always recommend AI drafting.

This can create tunnel vision.

A useful Today experience may balance:

- urgent risk;
- client relationship;
- professional quality;
- commercial value;
- organizational learning;
- pending review.

Diversity should not suppress critical items.

It should prevent one metric from monopolizing attention.

---

## 32. Duplicate and Competing Candidates Must Be Reconciled

Several sources may produce the same candidate.

Examples:

- official-status monitor;
- Task rule;
- AI Agent;
- human user;
- Product logic.

The system should identify:

- duplicate subject;
- duplicate proposed response;
- overlapping time window;
- existing formal Task;
- already completed action;
- conflicting recommendation.

It may:

- merge evidence;
- select one presentation;
- link related candidates;
- suppress duplicates;
- escalate conflict.

Duplicate suppression must preserve provenance.

---

## 33. Existing Formal Work Must Influence Recommendation

Before recommending a new action, Workplace should check whether:

- an active Task already exists;
- a Workflow is active;
- a Communication draft exists;
- a reviewer is assigned;
- an Opportunity is already formalized;
- Routing is already decided;
- the Matter has changed;
- the action was completed.

The recommendation should not create parallel shadow work.

A useful response may be:

```text
Continue the existing Task
```

rather than:

```text
Create another Task
```

---

## 34. Provider Recommendation Is Not Routing Decision

Mo Intelligence may compare providers and recommend candidates.

Formal Routing has its own:

- request;
- candidate set;
- evaluation;
- decision;
- selected target;
- review;
- override;
- audit.

The path is:

```text
Capability need
→ provider candidates
→ intelligence comparison
→ recommendation
→ Human Review or selection
→ Routing Service
→ Routing decision
```

A provider recommendation must not:

- appoint the provider;
- assign the Matter;
- approve fees;
- create representation authority;
- send instructions.

---

## 35. User Interest Is Not Client Instruction

A user may interact with a recommendation by:

- opening it;
- saving it;
- selecting an option;
- asking for a draft;
- marking interest.

These interactions may indicate user intent.

They do not automatically prove the client instructed the organization.

Client instruction may require:

- authorized client participant;
- confirmed Communication;
- engagement context;
- effective scope;
- accepted terms;
- formal record.

The system should not convert internal exploration into external commitment.

---

## 36. Recommendation Acceptance Must Be Typed

“Accept” can mean several things:

```text
show me more

use this preference

prepare a draft

create a Task request

submit for review

record a decision

execute an approved action
```

The system should use typed acceptance rather than one ambiguous button.

The resulting state should be explicit.

Examples:

```text
Recommendation acknowledged

Option selected

Draft preparation requested

Task creation requested

Submitted for professional review
```

None should be shown as execution unless the proper service confirms it.

---

## 37. Dismissal Must Also Be Typed

A user may dismiss a candidate because it is:

- irrelevant;
- duplicate;
- already handled;
- incorrect;
- unauthorized;
- deferred;
- not currently valuable;
- commercially inappropriate.

Dismissal should not automatically mean:

- risk resolved;
- formal Task cancelled;
- client declined;
- Opportunity rejected;
- underlying signal invalid.

A critical signal may require reason or escalation.

The system should distinguish:

```text
hide from personal view

snooze

reject candidate

mark duplicate

mark already handled

record formal rejection through owning service
```

---

## 38. Snooze Is Not Resolution

Snooze changes attention timing.

It does not resolve the underlying need.

A snoozed recommendation should preserve:

- reason;
- return time;
- source freshness;
- deadline risk;
- invalidation conditions.

The system should prevent snooze beyond a critical deadline where Policy forbids it.

---

## 39. Feedback Must Not Automatically Become Truth

Users may provide feedback:

- useful;
- not relevant;
- wrong;
- already handled;
- good recommendation;
- bad provider match.

This feedback can improve:

- ranking;
- candidate generation;
- Product design;
- Knowledge gaps;
- Skill evaluation.

It should not automatically:

- validate a professional rule;
- alter shared Knowledge;
- change provider trust globally;
- modify formal objects;
- create organization policy.

Feedback is evidence for learning.

It requires scope and governance.

---

## 40. Outcome Feedback Should Evaluate the Chain

When a recommendation leads to action, the system may learn from:

- user acceptance;
- review result;
- execution outcome;
- client response;
- service conversion;
- delivery quality;
- missed-work reduction;
- correction;
- later reversal.

The system should distinguish:

```text
recommendation accepted

action executed

action successful

client benefited

organization benefited

professional outcome validated
```

Acceptance alone is not success.

A rejected recommendation may still have been reasonable given the information available.

---

## 41. Recommendation Explainability Must Be Practical

The user should be able to understand:

- why the candidate appeared;
- which sources support it;
- what value may exist;
- why it ranks now;
- which assumptions apply;
- what alternatives exist;
- what is missing;
- what review is required;
- what will happen after selection.

A useful explanation may say:

```text
This appears first because:
- the official response deadline is in five days;
- no active response Task exists;
- the client instruction is confirmed;
- the Evidence package is incomplete;
- professional review is required before filing.
```

Explainability does not require revealing proprietary scoring formulas or private data.

## 43. Required Distinctions

```text
Observation ≠ fact
```

An observation may be incomplete or incorrect.

```text
Signal ≠ Event
```

Signal interprets possible relevance.

```text
Signal ≠ Notification
```

Not every signal should interrupt a user.

```text
Value Candidate ≠ Opportunity
```

Formal Opportunity requires its Owning Service and lifecycle.

```text
Value Candidate ≠ Task
```

Possible value does not create active work.

```text
Recommendation ≠ decision
```

Authorized human or service authority remains required.

```text
Next Best Action ≠ active Task
```

It is an attention proposition.

```text
Score ≠ formal Priority
```

Intelligence ranking does not mutate business state.

```text
Ranking ≠ authorization
```

Position in a list creates no authority.

```text
Provider recommendation ≠ Routing decision
```

Routing Service owns formal selection context.

```text
User interest ≠ client instruction
```

Internal interaction does not bind the client.

```text
User confirmation ≠ professional approval
```

Review authority remains distinct.

Recommendation acceptance is treated as proof of client or professional value.

```text
Observations, signals, candidates,

Scores remain multidimensional where consequence requires it.

Ranking determines attention, not authority.

Next Best Action remains a candidate,
```

## 47. Chapter Conclusion

A professional Workplace should not wait until work is already formalized before helping the user see value.

It should observe, interpret, compare, and recommend.

But intelligence becomes unsafe when possibility is presented as obligation or when ranking silently changes formal state.

The correct chain is:

```text
Observe
→ create Signal
→ identify possible value
→ form Value Candidate
→ validate eligibility
→ score and rank
→ recommend
→ explain
→ let the user or reviewer decide
→ prepare
→ hand off to governed Execution
```

The constitutional outcome is:

```text
Value Factory creates candidates.

Mo Intelligence evaluates and recommends.

Workplace contextualizes.

Assistant and Guide present.

AI Agent assists.

Human selects and reviews.

Execution governs action.

Owning Services formalize.
```

Next Best Action is therefore not the next action already taken.

It is the best currently known eligible action candidate presented for accountable consideration.

CH18 now addresses the next boundary:

> Once a recommendation is selected and its inputs are prepared, what must a Prepared Action contain before it can enter governed Execution without allowing Product, Assistant, Guide, or AI Agent to perform the protected action directly?
