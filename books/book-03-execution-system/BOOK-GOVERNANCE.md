# Book 03 Governance

## 1. Purpose

Book 03 defines how approved Book 02 Core contracts become governed operational progress.

It owns execution coordination, context assembly, Workflow behavior, Task progression, Human Review gates, Communication preparation, retry, safe failure, version control, agent assistance and audit.

It does not redefine Core, Workplace or Product responsibilities.

## 2. Source Precedence

```text
1. Frozen Book 02 Core Specification Baseline v0.1
2. Owner-accepted Book 03 canonical manuscript
3. Repository Architecture Canon and approved cross-book interpretation
4. Approved implementation contracts and fixtures
5. Tests expressing approved acceptance criteria
6. Implementation code
```

Implementation does not silently overrule publication authority.

## 3. Responsibility Lock

```text
Core defines shared meaning.
Execution governs coordinated work.
Workplace provides authorized organization context.
Products compose focused user journeys.
Integration connects approved external systems.
Owning Services change and record formal business facts.
Humans remain professionally accountable.
AI assists under explicit governance.
Events trace governed outcomes.
```

## 4. Execution Authority

Book 03 preserves:

- preparation is not approval;
- approval is not execution;
- approval is consequence-specific and version-bound;
- a Task plan is not an active Task;
- draft is not send;
- send is not delivery;
- recommendation is not decision;
- Capability is not authority;
- AI confidence is not Human Review;
- Workflow observation is not a Domain Event;
- Product presentation does not redefine execution truth.

## 5. Workplace and Product Boundary

Workplace is the organization-level operating environment of an independent Orbit.

It may expose Product and operating surfaces.

Those surfaces consume Execution outcomes but may not invent Workflow depth, Human Review validity, Event authority, mutation authority or protected-action readiness.

## 6. AI and Agent Boundary

AI assistants and agents may retrieve, extract, summarize, compare, draft, recommend, prepare and sequence explicitly permitted steps.

They may not independently approve, send, submit, pay, appoint, instruct, mutate protected state, bypass Permission or Policy, emit governed Events or define professional truth.

## 7. MVP Depth

- Depth A: three approved internal-apply workflows.
- Depth B: five preview-only workflows.
- Depth C: external protected actions deferred.

Product design, connector availability or model capability does not expand depth.

## 8. Change Control

Explicit review is required before any change that:

- adds or deepens a Workflow;
- adds an allowed mutation;
- enables external action;
- changes Human Review;
- changes Service or Event ownership;
- introduces a Core semantic;
- expands agent tools or authority;
- activates a production connector.

## 9. Historical Records

Planning and historical reviews are not rewritten merely because later publication titles changed.

The active manuscript and current portfolio baseline govern present interpretation.

## 10. Non-Authorization

This governance record does not authorize production, unrestricted implementation or external protected action.
