# B01-PUB-0001 — Editorial Style and Terminology

**Status:** Release Candidate 1

## Purpose

Book 01 is an architectural essay for professional readers. It should remain readable without reducing constitutional precision.

## Style Rules

- Use short paragraphs and clear transitions.
- Prefer professional-language explanations over software jargon.
- Preserve British English already used by the manuscript.
- Capitalize defined MarkOrbit concepts when they carry architectural meaning: Core, Execution, Workplace, Product, Capability, Skill, Knowledge, Review, Routing, Artifact, Owning Service and Orbit.
- Use lower case for generic uses.
- Avoid PRD, sprint, module, database-schema and implementation language.
- Explain a boundary before giving an example.
- Do not present a current technical implementation as permanent architecture.

## Canonical Responsibility Language

```text
Core defines shared meaning.
Execution governs coordinated work.
Workplace provides authorized organization context.
Products compose focused user journeys.
MGSN connects independent organizational Orbits.
Owning Services change and record formal business facts.
Humans remain professionally accountable.
AI assists under explicit governance.
```

## Required Distinctions

```text
Organization ≠ Workplace
Workplace ≠ Product
Product ≠ Operating System
Information ≠ Knowledge
Knowledge Candidate ≠ approved Knowledge
Recommendation ≠ Decision
Routing Recommendation ≠ Provider Appointment
AI Draft ≠ Approved Output
Prepared Action ≠ Execution
```

## Prohibited Drift

Book 01 must not describe MarkOrbit as:

- a centralized marketplace;
- one universal tenant database;
- one mega-Product;
- an AI-owned professional practice;
- a provider directory that automatically distributes work;
- a system that learns from all private activity by default.
