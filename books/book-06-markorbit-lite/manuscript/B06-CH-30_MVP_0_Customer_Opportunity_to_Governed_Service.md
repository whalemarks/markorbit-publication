# B06-CH-30 — MVP 0: Customer Opportunity to Governed Service

**Status:** Complete Draft 1  
**Chapter Map:** B06-TOC-V0.1 — Owner Accepted  
**Part:** Part VII — Conformance, Economics and Evolution  
**Primary controls:** ML-AC-01–ML-AC-12, ML-E01

## Chapter Purpose

This chapter defines the first bounded Product proof for MarkOrbit Lite.

The central proposition is:

> Lite has not proved its Product value merely because it can generate content, list prospects, display recommendations or create files. It must help a professional identify relevant client value, prepare trustworthy work, take a confirmed action, recognize a real service need and move that need into a governed service journey without losing context or authority boundaries.

The controlled name of this proof is:

```text
Customer Opportunity-to-Governed-Service Loop
```

It is the minimum complete expression of the Lite Product Charter.

## 1. Why MVP 0 must be a complete loop

A weak MVP often validates the easiest visible output:

- a generated article;
- a daily content calendar;
- a list of companies;
- a dashboard full of recommendations;
- a drafted email;
- a rendered PDF;
- a high number of clicks;
- a large number of created candidates.

These outputs may be useful, but none proves that Lite changes professional practice.

A complete Product proof must connect:

```text
authorized context
→ relevant professional value
→ trustworthy work product
→ Human judgment
→ confirmed client action
→ typed response
→ qualified service need
→ governed Handoff
→ destination result
→ continued Product value
```

The loop matters because every transition tests a different responsibility:

- whether Lite can use the right context;
- whether it can identify something genuinely relevant;
- whether it can explain rather than merely score;
- whether the professional work product is usable;
- whether the user retains judgment and control;
- whether the customer response is interpreted correctly;
- whether formal state remains with the proper destination;
- whether Return continuity preserves meaning;
- whether learning remains scoped and governed.

## 2. The bounded MVP population

MVP 0 should begin with a small, authorized population.

A suitable pilot may include:

- a limited number of independent trademark professionals;
- a small number of participating Organizations;
- a bounded set of active or recently inactive customers;
- a bounded set of trademark records;
- a narrow group of service types;
- a limited number of jurisdictions;
- explicit access and purpose rules;
- named reviewers and destination owners.

The purpose of a bounded population is not to make the test artificially easy.

It is to make the evidence interpretable.

If the pilot mixes hundreds of unrelated users, multiple service lines, uncertain customer ownership, uncontrolled public sources and undefined provider processes, failure cannot be diagnosed.

The Evaluation Run must therefore identify:

```text
who is participating
which Organization context applies
which customer/trademark set is authorized
which service-value patterns are in scope
which destinations are available
which time window is tested
which Product and source versions are used
```

This is `ML-E01 Evaluation Run` applied to MVP 0.

## 3. Acceptance criterion 1 — Authorized context

`ML-AC-01` requires an explicit Organization and access scope.

The pilot must demonstrate that Lite can distinguish:

- public information;
- Organization Internal information;
- Personal Private information;
- Client Restricted information;
- Matter Restricted information;
- Network Shared information;
- Local Only information.

A successful run does not begin with the instruction:

> Use everything available about this customer.

It begins with a purpose-bound context:

```text
For this user
within this Organization
for this customer and trademark set
for this defined service-development purpose
using these allowed sources
with these disclosure limits
```

If access is missing, the correct Product result is to block or minimize the operation.

A blocked unauthorized action is evidence of control, not Product failure.

## 4. Acceptance criterion 2 — Opportunity relevance

`ML-AC-02` requires at least one timely and explainable `ML-O03 Service-Value Candidate`.

The strongest initial source remains the existing customer portfolio.

Candidate examples may include:

- an upcoming renewal or maintenance need;
- a new market not covered by the existing portfolio;
- a product expansion not reflected in current classes or goods/services;
- inconsistent applicant or ownership information;
- an examination, opposition or portfolio result that creates a next professional step;
- an earlier deferred service that has become relevant because circumstances changed.

The candidate must explain:

- what was observed;
- which customer or trademark it relates to;
- why it may matter now;
- which source supports the observation;
- what remains uncertain;
- what client value may exist;
- what professional risk or limitation applies;
- which work product and next action may be useful.

The candidate must not be accepted merely because it has a high commercial value.

```text
revenue potential
≠ professional relevance
```

A low-value but necessary maintenance action may rank above a high-fee service that is not in the client's interest.

## 5. Acceptance criterion 3 — Source quality

`ML-AC-03` requires source, freshness, confidence and missing information to remain visible.

The MVP must test at least:

- a current authoritative source;
- a stale source;
- conflicting sources;
- a user-provided correction;
- a missing key fact;
- a source that is traceable but not sufficiently authoritative.

The Product should respond differently to each.

For example:

```text
current official status
→ may support a stronger Candidate

old internal spreadsheet
→ may support an observation but requires verification

conflicting owner names
→ must expose conflict and block confident action

AI-generated summary
→ may help organize material but is not source authority
```

Source visibility is part of the user value.

It enables the professional to judge whether the proposed action is trustworthy.

## 6. Acceptance criterion 4 — Work-product value

`ML-AC-04` requires a trustworthy work product that the user actually uses or meaningfully edits.

The MVP work product should be tied to the Candidate and customer action.

Examples include:

- a customer portfolio gap explanation;
- a maintenance reminder with supporting facts;
- a jurisdiction expansion proposal;
- a quotation explanation;
- a concise client recommendation card;
- a PDF analysis;
- a customer-development package;
- a MarkReg launch support package.

The test must preserve:

```text
Structured Content
→ Artifact Draft
→ immutable Artifact Version
→ Review where required
→ Render / Edit
→ purpose-specific Readiness
```

The following do not satisfy `ML-AC-04` by themselves:

- a topic;
- a headline;
- raw AI text;
- a generic template with no current facts;
- a script when the promised output is a video;
- a rendered file with incorrect deterministic fields;
- a package the user cannot use without starting over.

Meaningful edit is not a failure.

It may show that the Product supplied a useful starting point while the professional supplied judgment.

However, repeated complete rewrites may indicate poor relevance, source quality or fit.

## 7. Acceptance criterion 5 — Confirmed user action

`ML-AC-05` requires final user confirmation before consequential customer action.

The MVP must show the exact transition:

```text
Prepared Action
→ Review where required
→ final user confirmation
→ governed or manual Communication
```

The confirmation should identify:

- recipient;
- channel;
- selected Artifact Version;
- attachments;
- purpose;
- expected consequence;
- opt-out or suppression status;
- any professional Review result.

The Product must not count a prepared draft as a contact action.

```text
prepared
≠ sent
```

The pilot should also measure the percentage of accepted Candidates that users choose not to contact.

That result is informative.

It may reveal poor relevance, weak timing, missing authority, unsuitable communication or an appropriate professional decision not to approach the client.

## 8. Acceptance criterion 6 — Typed response

`ML-AC-06` requires distinct response states.

The Product must preserve at least:

- confirmed need;
- request for more information;
- request for quotation;
- request for meeting;
- factual correction;
- another adviser is responsible;
- rejection;
- opt-out;
- invalid contact;
- no response;
- delivery failure;
- unknown external outcome.

A response taxonomy prevents false conversion claims.

```text
message opened
≠ response

positive wording
≠ qualified need

no response
≠ rejection

unknown delivery state
≠ sent successfully
```

The Product should allow user correction when a response is misclassified.

The correction should become `ML-M02 Correction Record`, not a hidden model adjustment.

## 9. Acceptance criterion 7 — Qualification

`ML-AC-07` requires Lite to distinguish a real service need from weak activity.

A Qualification Result should identify:

- who expressed the need;
- whether the person can represent the customer;
- which service is needed;
- the jurisdiction and procedure;
- timing and urgency;
- expected scope;
- supporting facts;
- remaining unknowns;
- conflicts, rights or authority concerns;
- the appropriate formal destination.

The following remain insufficient:

- a click;
- an open;
- a generic expression of interest;
- a request to “send more information” with no identified service;
- a user assumption that the customer probably needs something;
- a public event with no customer response.

```text
Qualification Result
≠ formal Opportunity
```

The destination still decides whether formal state should be created.

## 10. Acceptance criterion 8 — Governed Handoff

`ML-AC-08` requires the qualified need to be accepted or explicitly rejected by MarkReg or another proper destination.

A conforming Handoff must include:

- origin and Organization;
- customer, contact, applicant and trademark references;
- service need;
- sources and selected Artifact Version;
- user decisions;
- authority and instruction context;
- missing information;
- required Review;
- intended consequence;
- Return address.

The destination must revalidate.

MVP 0 therefore passes this criterion when the destination returns one of the supported results:

```text
formal reference returned
more information required
rejected
duplicate
blocked
unsupported
failed
```

A rejection may still be a successful test of governance.

It demonstrates that Lite did not silently create formal truth.

## 11. Acceptance criterion 9 — Return continuity

`ML-AC-09` requires the returned result to appear in Lite without loss of origin or false state.

The user should be able to answer:

- which Handoff produced this result;
- which destination returned it;
- what formal reference exists;
- whether more information is required;
- what the user may do next;
- whether the result is current;
- whether the result remains unknown or incomplete.

The returned item may become a Today Attention Item or Continuation State.

It must not become a Lite-owned duplicate of the formal MarkReg, Opportunity, Task or Communication record.

## 12. Acceptance criterion 10 — Capability accumulation

`ML-AC-10` requires at least one scoped learning result.

Possible evidence includes:

- a source correction;
- a corrected customer preference;
- a reusable work-product component;
- a Case Lesson Candidate;
- a safer communication pattern;
- an improved Handoff package;
- a retired stale template;
- a Reuse Evidence record.

The MVP does not require automatic promotion to Organization Memory or canonical Knowledge.

It requires that useful experience can enter a governed candidate path.

```text
outcome
→ Feedback / Correction
→ scoped candidate
→ Review
→ accepted bounded capability or retirement
```

## 13. Acceptance criterion 11 — Safety and privacy

`ML-AC-11` requires zero blocking scenario violations.

This means the Product must not:

- expose Customer or Matter Restricted facts without authority;
- convert a Product-local Candidate into formal Opportunity state;
- prioritize revenue over client interest and professional risk;
- contact an opted-out customer;
- treat a found contact as permission for every channel;
- represent raw AI text as approved professional work;
- represent Render completion as Review approval;
- represent Publish Package readiness as publication;
- reuse restricted case facts;
- promote Organization Memory without Review;
- allow a destination to trust Lite without revalidation;
- upload Local Only data without synchronization authority.

A blocked unsafe action contributes positively to the pilot result.

The MVP should report:

- number of blocked actions;
- reason;
- affected journey;
- whether the block prevented a real defect;
- whether the user understood how to correct or proceed.

## 14. Acceptance criterion 12 — Product and commercial evidence

`ML-AC-12` collects recurring use, cost and plan evidence.

It does not impose one permanent commercial threshold.

The MVP should measure:

- active professionals by week;
- days with meaningful use;
- time to first useful result;
- repeated use across Today, Growth and Work Product surfaces;
- Candidate acceptance, rejection and suppression;
- customer contact and response;
- qualified need;
- governed Handoff;
- returned formal references;
- work-product use and edit rate;
- correction and reuse;
- safety/privacy findings;
- AI, Render, TTS, storage, support and Review cost;
- perceived value and willingness to renew.

Commercial evidence is interpreted in CH32.

## 15. Pass rule

The controlled pass rule is:

```text
ML-AC-01–ML-AC-11: mandatory
ML-AC-12: evidence must be collected
commercial threshold: experiment-specific
```

MVP 0 must not pass merely because it:

- creates thirty files;
- fills thirty calendar days;
- supplies many organization names;
- records high page views;
- produces many recommendations;
- creates unqualified Opportunity records;
- reports user-claimed publication as confirmed publication;
- generates revenue through unsafe or misleading action.

It must demonstrate an end-to-end professional value loop.

## 16. Interpreting partial results

A pilot may produce mixed evidence.

Examples:

### Strong work products, weak customer action

This may indicate:

- poor opportunity timing;
- weak recipient or channel selection;
- insufficient user trust;
- a product that helps production more than growth.

### High response, weak qualification

This may indicate:

- messages attract attention but do not reveal real need;
- the Candidate explanation is too broad;
- user incentives favor outreach volume.

### Strong Handoff, poor Return continuity

This may indicate:

- destination integration exists but Product continuity is broken;
- Lite risks becoming a one-way intake form.

### Strong recurring use, unsustainable cost

This may indicate:

- Product value exists;
- the entry-plan packaging, entitlement or operating model must change.

### Low activity, strong value in a few cases

This may indicate:

- the Product is valuable but not daily for the tested population;
- a weekly rhythm, portfolio event model or higher-value edition may fit better.

MVP interpretation should not force every result into the original commercial hypothesis.

## 17. Exit states after MVP 0

A completed Evaluation Run should end with one of the following decisions:

```text
PROCEED
→ Product loop is valuable and safe enough for a broader pilot

PROCEED WITH CONDITIONS
→ value is demonstrated but specific controls, economics or journeys require correction

REPEAT BOUNDED TEST
→ evidence is incomplete or confounded

REFRAME COMMERCIAL PLAN
→ Product value exists but price, entitlement or delivery model is unsuitable

REFRAME PRODUCT INCREMENT
→ one loop is valuable but the tested Product scope is too broad or too narrow

STOP
→ no sufficient user value, unacceptable risk or no credible path to sustainable operation
```

The decision and supporting evidence should remain separate from marketing claims.

## Chapter Conclusion

MVP 0 is not a feature demo.

It is a proof that Lite can transform authorized professional context into a relevant client action, a governed service journey and reusable capability while preserving Human responsibility and destination ownership.

The shortest complete proof remains:

```text
authorized customer and trademark context
→ explainable service-value candidate
→ trustworthy work product
→ confirmed customer action
→ typed response and qualification
→ governed Handoff
→ returned result
→ scoped learning
```

Only after this loop works should content frequency, prospect quantity, plan price and scale become decisive commercial questions.
