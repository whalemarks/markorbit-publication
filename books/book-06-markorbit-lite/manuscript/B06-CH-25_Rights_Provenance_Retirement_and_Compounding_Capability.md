# B06-CH-25 — Rights, Provenance, Retirement and Compounding Capability

## Chapter Role

This chapter defines the controls that keep accumulated lessons, memories and reusable Assets trustworthy over time.

The central proposition is:

> Capability compounds only when every reused element remains traceable, authorized, current, correctable and removable.

Without these controls, accumulation becomes technical and professional debt.

---

## 1. Compounding capability is the long-term value of Lite

A generic AI tool may answer a question well today but begin from nearly the same place tomorrow.

Lite should improve through governed continuity:

```text
real work
→ Outcome Observation
→ Feedback / Correction
→ Lesson / Memory / Asset candidate
→ Human Review
→ accepted bounded capability
→ safer and faster future work
→ new Outcome Observation
```

The value is not simply that the Product stores more information.

The value is that the Product becomes better able to:

- identify relevant customer value;
- prepare suitable professional work products;
- avoid known errors;
- preserve user and Organization preferences;
- select useful sources and Assets;
- prepare cleaner Handoffs;
- recognize when an old rule no longer applies.

## 2. More memory can make a Product worse

Uncontrolled accumulation creates:

- duplicate and contradictory templates;
- stale legal rules;
- expired prices;
- former-client information;
- hidden Matter restrictions;
- incorrect personal preferences;
- outdated provider assumptions;
- unsupported case conclusions;
- copied third-party material;
- unclear ownership;
- resources no one is responsible for retiring.

Therefore:

```text
more records
≠ more capability
```

Quality depends on lifecycle control.

## 3. Provenance is the foundation

Provenance answers:

- where this item came from;
- who created or proposed it;
- which sources support it;
- which case, Artifact or correction influenced it;
- what transformations occurred;
- who reviewed it;
- which version was accepted;
- where it has been used.

A useful provenance chain may be:

```text
Matter reference
→ authorized case projection
→ Case Lesson Candidate v1
→ reviewer correction
→ Case Lesson Candidate v2
→ Reusable Asset Candidate v1
→ approved Asset v1
→ Artifact Version A3
→ customer use
→ Correction Record
→ Asset v2 or retirement
```

If the chain cannot be reconstructed, broad reuse should be limited.

## 4. Provenance is not the same as source authority

A record may have clear provenance but weak authority.

Examples:

- an internal note;
- a user recollection;
- an AI-generated summary;
- an unofficial blog;
- a provider's informal email;
- an outdated government page.

Provenance explains origin.

Source evaluation explains reliability and appropriate use.

```text
traceable
≠ authoritative
```

Both are required.

## 5. Rights must travel with the Asset

Rights information should not remain in a separate forgotten file.

Every reusable Asset or publishable work-product source should preserve, where relevant:

- rights holder;
- creator;
- client ownership or consent;
- third-party license;
- permitted audience;
- permitted channels;
- permitted modification;
- attribution requirement;
- geographic scope;
- time limit;
- commercial-use permission;
- revocation conditions;
- prohibited uses.

A Render Result does not erase the rights of its inputs.

```text
new file generated
≠ new unrestricted rights created
```

## 6. AI generation does not solve rights questions

AI-assisted output may still be influenced by:

- client inputs;
- third-party images;
- licensed templates;
- provider material;
- protected marks;
- personal data;
- confidential examples;
- user-uploaded source content.

The Product should preserve the relevant source and usage context rather than treating “AI generated” as a complete rights category.

## 7. Audience and purpose are rights controls

An item may be permitted for:

- personal analysis;
- Organization internal training;
- a specific client;
- private communication;
- public education;
- commercial promotion.

These uses are materially different.

```text
permitted for internal training
≠ permitted for public marketing
```

A new audience or purpose should trigger a new rights and Review decision.

## 8. Versioning is required for capability records

The same version discipline used for Artifacts should apply to:

- accepted preferences;
- Personal Memory;
- Organization Memory;
- Case Lessons;
- Reusable Assets;
- Knowledge Contribution Candidates.

Substantive changes should create a new version.

Examples:

- jurisdiction scope changes;
- new source added;
- client restriction removed or added;
- legal rule updated;
- template wording changed;
- allowed audience broadened;
- required Review changed;
- rights expire.

Version history allows the Product to identify which output used which capability version.

## 9. Correction must be propagatable

A correction is useful only if affected uses can be found.

The Product should be able to trace:

```text
Correction Record
→ affected memory / lesson / Asset / Knowledge candidate
→ affected Structured Content
→ affected Artifact Versions
→ affected Render Results
→ affected Delivery / Publish Packages
→ known external uses
```

The appropriate response may be:

- no action required;
- add a warning;
- require Review before reuse;
- replace the Asset;
- notify an internal user;
- prepare a corrected client communication;
- withdraw a public item;
- retire the source capability.

## 10. Correction does not rewrite history

Historical versions should remain auditable where retention rules permit.

Unsafe approach:

```text
replace old content silently
→ all history appears to have always been correct
```

Controlled approach:

```text
old version remains recorded
+ correction linked
+ current version identified
+ future use blocked where required
```

This protects professional accountability and allows analysis of how the error occurred.

## 11. Expiry and review dates

Not all capability has the same useful life.

### Long-lived

- stable Product principles;
- approved writing tone;
- Organization review responsibility;
- general work-product structure.

### Medium-lived

- procedural guidance;
- provider capability information;
- client preferences;
- internal process checklists.

### Short-lived

- fees;
- official forms;
- deadlines;
- contact details;
- current office practice;
- campaign-specific content;
- temporary client instructions.

The item should carry an appropriate:

```text
review_due_at
expires_at
source_refresh_requirement
```

Expiry need not always delete the item. It may change the state to `stale` or `review_required`.

## 12. Retirement states

A capability item may become:

```text
active
limited
review_due
stale
superseded
revoked
blocked
retired
archived
```

These states should not be flattened into active/deleted.

Examples:

- `limited`: still usable for internal reference, not client-facing output;
- `superseded`: a newer version should be used;
- `revoked`: permission was withdrawn;
- `blocked`: legal, policy or confidentiality issue prevents use;
- `retired`: intentionally removed from future recommendation;
- `archived`: retained for audit or historical analysis only.

## 13. Retirement triggers

Retirement may be triggered by:

- rights expiry or revocation;
- client request;
- changed law or practice;
- factual correction;
- repeated poor outcomes;
- replacement by a better Asset;
- provider relationship ending;
- changed Organization policy;
- security or privacy finding;
- inability to verify source;
- no responsible owner;
- scope no longer relevant.

A retired item should stop appearing in normal recommendations.

## 14. Downstream effects of retirement

Retirement does not automatically invalidate every historical use.

The Product should distinguish:

- future use blocked;
- existing draft requires replacement;
- approved Artifact requires re-review;
- delivered work requires correction;
- public content requires withdrawal;
- historical record remains valid as evidence of past action.

This requires lineage, not blanket deletion.

## 15. Ownership and stewardship

Every accepted capability should have a steward responsible for:

- source refresh;
- rights;
- scope;
- review;
- correction;
- retirement;
- conflict resolution.

Possible stewards include:

- the individual user;
- an Organization administrator;
- a professional reviewer;
- a Product owner;
- Knowledge governance;
- another Owning Service.

An item with no steward should have a limited lifespan or remain a candidate.

## 16. Revocation must be practical

The right to revoke or narrow a memory or Asset is meaningless if the Product cannot stop future use.

A practical revocation should:

- update the item state;
- stop recommendation and automatic inclusion;
- identify active drafts using it;
- preserve audit history;
- prevent unauthorized synchronization;
- propagate to local or cached contexts where possible;
- show where external copies may remain outside Product control.

The Product must be honest about technical limits.

## 17. Local/private capability

A private local Asset may offer significant value without being uploaded.

Possible pattern:

```text
local Asset
→ local retrieval
→ minimum necessary derived context
→ Product-local Artifact preparation
→ no source upload without authorization
```

Its provenance should still indicate:

- local source;
- device or vault scope;
- allowed processing;
- synchronization state;
- revocation behavior;
- whether derived content may leave the local zone.

## 18. Contradiction management

Capability may contradict:

- another Asset;
- accepted Organization memory;
- current Knowledge;
- a formal source;
- a client instruction;
- a destination requirement.

The Product should expose contradiction rather than select silently.

A conflict record should identify:

- competing statements;
- scopes;
- source strength;
- effective dates;
- affected work;
- required reviewer;
- temporary safe behavior.

Possible safe behavior is to block consequential use until resolved.

## 19. Measuring compounding capability

The Product should not measure capability only by the number of saved items.

Better measures include:

- proportion with complete provenance;
- proportion with an owner and review date;
- reuse that led to a useful outcome;
- reuse that required correction;
- time saved without quality loss;
- errors prevented;
- stale items detected before use;
- corrections propagated successfully;
- Assets retired when no longer trustworthy;
- user or Organization confidence in recommendations.

```text
large library
≠ strong capability
```

## 20. Reuse feedback loop

A controlled loop is:

```text
accepted capability version
→ recommended for matching purpose
→ user inspects and applies
→ new Artifact Version
→ Review and external use
→ Outcome Observation
→ Reuse Evidence
→ confirm suitability / narrow / correct / retire
```

This makes the library improve through real outcomes rather than assumed usefulness.

## 21. Example: outdated fee explanation

An approved quotation Asset contains a fee example.

Later, the official fee changes.

Controlled response:

```text
source refresh detects change
→ Correction Record
→ Asset marked stale
→ active quotation drafts identified
→ deterministic fee values refreshed
→ affected Artifact Versions re-reviewed
→ old Asset superseded
```

Unsafe response:

```text
old Asset remains recommended because it was previously approved
```

## 22. Example: withdrawn client permission

A client previously allowed an anonymized case example for internal training and later withdraws permission.

The Product should:

- revoke the applicable Asset scope;
- stop future recommendation;
- identify internal training Artifacts using it;
- apply the client's instruction according to legal and contractual requirements;
- preserve minimal audit evidence where required;
- avoid claiming that public copies can always be deleted if none exist or control is limited.

## 23. Security and privacy findings

`ML-E04 Safety / Privacy Finding` may identify:

- restricted facts appearing in an Asset;
- local content synchronized without authority;
- rights metadata missing;
- retired content still recommended;
- one user's private preference exposed to the Organization;
- an Artifact using an expired source;
- correction propagation failure.

A blocking finding should count as Product evidence and may require disabling the affected capability.

## 24. Capability and commercial plans

A commercial plan may set limits for:

- number of stored Assets;
- local/private features;
- Organization memory;
- advanced Review;
- Knowledge contribution workflows;
- retention;
- source refresh frequency.

But commercial limits must not change semantic meaning.

```text
plan permits more Assets
≠ plan grants broader rights
≠ plan bypasses Review
```

## 25. The product moat

The durable value of Lite is not a generic model or a large template count.

It is the governed combination of:

```text
user and Organization context
+ authorized customer/trademark context
+ reviewed work products
+ real outcomes
+ corrections
+ scoped memory
+ reusable Assets
+ MarkOrbit service continuity
```

This capability becomes difficult to replace because it is specific, explainable and connected to real professional work.

Its value depends on trust.

## 26. Part V conclusion

Cases, memories and Assets create compounding value only through a controlled lifecycle:

```text
work and outcome
→ sourced lesson / preference / memory candidate
→ Human or Organization Review
→ accepted bounded capability
→ purpose-matched reuse
→ new work product and outcome
→ correction, scope change, expiry or retirement
```

The integrity locks are:

```text
experience ≠ universal rule
personal memory ≠ Organization memory
Reusable Asset ≠ canonical Knowledge
accepted once ≠ valid forever
technically available ≠ authorized to reuse
retired ≠ erased history
```

Part VI will connect this accumulated capability to MarkReg, MGSN, Review, Communication, Opportunity, Task and Execution through governed gateways and typed continuity.
