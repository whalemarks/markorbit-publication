The matrix collects recurring Book 06 distinctions in one place. It does not replace the chapters or Specifications.

## Matrix

| ID | Distinction | Operational rule | Primary reading |
| --- | --- | --- | --- |
| <a id="b06-distinction-d01"></a>D-01 | **Today item ≠ active Task** | Today may show attention and continuation; only the Task/Execution service creates formal Task state. | CH07, CH11, CH28; `ML-S02–S03`, `ML-H08` |
| <a id="b06-distinction-d02"></a>D-02 | **Candidate ≠ formal truth** | A Candidate remains Product-local until the relevant Owning Service accepts a formalization request and returns a formal reference. | CH09–CH10, CH16; `ML-O03–O08` |
| <a id="b06-distinction-d03"></a>D-03 | **Recommendation ≠ Decision** | Recommendation explains options and uncertainty; a Human or destination makes the consequential Decision. | CH10–CH11; `ML-O07`, `ML-S04` |
| <a id="b06-distinction-d04"></a>D-04 | **User confirmation ≠ Human Review** | Confirmation answers whether an exact action should occur; Review answers whether an exact version is professionally acceptable for a defined scope. | CH15, CH18, CH20, CH28 |
| <a id="b06-distinction-d05"></a>D-05 | **Content ≠ Artifact** | Content is structured material; an Artifact is a purpose-bound, audience-specific and versioned business result. | CH17–CH18; `ML-W01–W03` |
| <a id="b06-distinction-d06"></a>D-06 | **Artifact ≠ Document** | An Artifact is a Product result; a Document is a formal or controlled business record owned by the relevant service. | CH18, CH24–CH25 |
| <a id="b06-distinction-d07"></a>D-07 | **Artifact ≠ Evidence** | An Artifact may explain or reference Evidence, but does not inherit Evidence authenticity, purpose or chain-of-custody meaning. | CH18, CH22 |
| <a id="b06-distinction-d08"></a>D-08 | **Artifact ≠ file** | One Artifact Version may have multiple file representations; file existence alone does not establish purpose, version, Review or readiness. | CH18–CH20 |
| <a id="b06-distinction-d09"></a>D-09 | **Render complete ≠ approved** | Render creates a representation; professional Review remains separate and may still block use. | CH19; `ML-W07–W10`, `ML-SCN-14–15` |
| <a id="b06-distinction-d10"></a>D-10 | **Package ready ≠ externally completed** | A Communication, Delivery or Publish Package may be ready while send, delivery or publication remains unconfirmed. | CH20, CH28; `ML-W09`, `ML-HC-04`, `ML-HC-07` |
| <a id="b06-distinction-d11"></a>D-11 | **Prepared Action ≠ execution** | Lite prepares exact-purpose action context; the destination revalidates and owns execution result. | CH11, CH28; `ML-W05`, `ML-H08` |
| <a id="b06-distinction-d12"></a>D-12 | **Handoff ≠ destination acceptance** | A Handoff is a request. The destination may accept, request more information, reject, block, fail or return unknown. | CH11, CH26–CH29; `ML-H01`, `ML-H03–H08` |
| <a id="b06-distinction-d13"></a>D-13 | **Return ≠ Lite-owned formal truth** | Lite presents a sourced destination result and must not create an independently mutable shadow record. | CH16, CH26–CH29; `ML-H02` |
| <a id="b06-distinction-d14"></a>D-14 | **Capability Need ≠ provider appointment** | MGSN evaluates capability and Trust; candidate return, user selection and provider appointment remain separate states. | CH27; `ML-H04`, `ML-HC-02` |
| <a id="b06-distinction-d15"></a>D-15 | **Prospect Candidate ≠ Customer / Qualified Lead / Opportunity** | Public or supplied data may justify inspection, but formal identity and commercial state require separate evidence and Owning-Service acceptance. | CH14–CH16; `ML-O05`, `ML-J03` |
| <a id="b06-distinction-d16"></a>D-16 | **Personal Memory ≠ Organization Memory** | Personal continuity remains narrow; Organization Memory requires provenance, Organization Review, owner and retirement path. | CH23–CH25; `ML-M04–M05` |
| <a id="b06-distinction-d17"></a>D-17 | **Reusable Asset ≠ canonical Knowledge** | An Asset helps produce work within a scope; Knowledge governance accepts professional propositions for a defined broader scope. | CH24–CH25; `ML-M07–M08`, `ML-HC-08` |
| <a id="b06-distinction-d18"></a>D-18 | **Local readability ≠ synchronization / disclosure authority** | A locally readable source may remain local; only minimum approved information may cross the boundary. | CH08, CH23, CH25, CH29; `ML-SCN-24` |
| <a id="b06-distinction-d19"></a>D-19 | **Unknown outcome ≠ completed / failed / safe to retry** | Consequential action with uncertain destination result must be reconciled before retry or status change. | CH20, CH28–CH31; `ML-SCN-18` |
| <a id="b06-distinction-d20"></a>D-20 | **Product identity ≠ Product Edition / Commercial Plan** | Product identity defines enduring purpose and boundaries; editions and plans select capabilities, limits and price. | CH32–CH33 |
| <a id="b06-distinction-d21"></a>D-21 | **Payment or premium edition ≠ authority** | Commercial access cannot grant Customer/Matter permission, professional approval, provider appointment or protected-action authority. | CH06, CH32–CH33 |
| <a id="b06-distinction-d22"></a>D-22 | **Blocked unsafe action ≠ Product failure** | A correct block may be positive conformance evidence when it prevents unauthorized, misleading or duplicate action. | CH30–CH31; `ML-E04`, `ML-AC-11` |
| <a id="b06-distinction-d23"></a>D-23 | **Traceable source ≠ authoritative source** | Provenance explains origin; source evaluation determines reliability and appropriate use. | CH08–CH09, CH17, CH25 |
| <a id="b06-distinction-d24"></a>D-24 | **User-reported outcome ≠ platform-confirmed outcome** | Both may be useful, but evidence strength and source must remain visible. | CH15, CH20–CH21 |
| <a id="b06-distinction-d25"></a>D-25 | **Approved template ≠ every produced instance approved** | Current facts, variables, audience, destination and version still require validation and Review. | CH18, CH21, CH24 |
| <a id="b06-distinction-d26"></a>D-26 | **Existing relationship ≠ current permission for every channel** | Relationship history supports context, not perpetual contactability or consent. | CH13–CH15; `ML-SCN-07–10` |
| <a id="b06-distinction-d27"></a>D-27 | **Successful case ≠ universal rule** | Case outcome becomes a bounded lesson candidate with conditions, uncertainty and Review. | CH22–CH25; `ML-SCN-20` |
| <a id="b06-distinction-d28"></a>D-28 | **Anonymized ≠ authorized for reuse or publication** | Identifiability, client consent, Matter restrictions, rights and purpose still require evaluation. | CH20, CH22, CH24–CH25 |
| <a id="b06-distinction-d29"></a>D-29 | **Qualification Result ≠ formal Opportunity or MarkReg acceptance** | Lite may qualify a need; the destination owns formalization and lifecycle acceptance. | CH10, CH15–CH16, CH26; `ML-O08` |
| <a id="b06-distinction-d30"></a>D-30 | **Entitlement fulfilled ≠ generation job completed** | Fulfillment requires the promised usable unit, quality threshold, format and evidence—not merely a job or file. | CH19, CH32; `ML-E06` |

## How to use the matrix

Use the matrix when:

- reviewing a chapter or diagram;
- designing a Product Increment;
- evaluating UI language;
- checking a Handoff or Return state;
- reviewing a commercial plan;
- identifying constitutional drift;
- preparing conformance tests.

A conforming implementation must demonstrate the distinction through observable behavior. Merely displaying the correct phrase is insufficient.
