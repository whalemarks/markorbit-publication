## Adopted changes

### Canonical Capability definition

```text
Capability
= Stable Outcome Contract
+ Governed Implementation
+ Evidence Base
+ Version Lineage
+ Controlled Evolution
```

### Canonical hierarchy

```text
Capability Domain
→ Capability
→ Skill
→ Action / Invocation
```

### Capability asset classes

```text
Source Capability
→ Composite Capability
→ Experimental Capability
→ Verified MO Capability
```

### Separate lifecycles

```text
Production
Source → Distillation → Source Capability

Runtime
Request → Composition → Execution → Outcome

Evolution
Outcome → Reflection Candidate → Case / Principle Candidate
→ Change Proposal → Version → Release
```

### Controlled composition

```text
One Primary Capability
+ zero to three Supporting Capabilities
+ zero or one Critic Capability
+ Capability Budget
+ explicit Conflict handling
```

### Learning handoff

```text
Outcome
→ Session Receipt
→ Reflection Candidate
→ rights and privacy review
→ Case / Principle Candidate
→ Capability Change Proposal
```

## Rejected or narrowed ideas

- `Master Skill` is not adopted as a canonical term; use Capability Composition or Capability Orchestration.
- Founder is not adopted as a Capability Domain; it is a Persona or Capability Profile type.
- Founder OS is treated as a future Product direction rather than a replacement for Lite or the seven-book architecture.
- Reflection does not automatically become Knowledge, Principle or Capability canon.
- Capability Twin is not a personality replica, user score or Professional Qualification.
- XP, Skill Tree UI and Mentor language remain Product-local by default.
- Raw conversations are not automatically stored in Core through Session Receipt.
- One successful task does not automatically upgrade a Capability version.
