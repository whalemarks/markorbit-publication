# Book 02 v2
# MarkOrbit Core Specification — Shared Semantics for a Capability-Centered Platform

## Purpose

Core defines the stable language shared by Products, Workplaces, Execution, Data, Knowledge, Brain, contribution systems and managed networks.

Core is not the database, Product UI, workflow engine, model router, learning dashboard or commercial owner of every record. Its durability comes from defining the smallest set of interoperable semantics required across the platform.

## Part I — Constitutional semantics

### Chapter 01 — Core authority

Core owns controlled meaning, identifiers, lifecycle vocabulary, contracts, validation rules and compatibility expectations. Owning Services create and mutate formal business facts through governed Execution.

### Chapter 02 — Authority separation

Every record must distinguish:

- business sovereignty;
- semantic authority;
- formal-state authority;
- physical custody;
- legal, evidentiary and professional authority.

Capability semantics do not override these authorities.

### Chapter 03 — Candidate-before-Canonical

```text
Observation ≠ fact
Candidate ≠ approved record
AI output ≠ professional decision
Delivered Return ≠ accepted truth
Certification ≠ legal qualification
Reflection Candidate ≠ canonical Principle
Outcome ≠ automatic Capability upgrade
```

## Part II — Identity and business context

### Chapter 04 — Person, Organization and Workplace

Core defines identity references, Organization references, Workplace context and Membership. Product-local permissions may extend these semantics but may not erase the represented authority context.

### Chapter 05 — Relationship semantics

Core defines Customer, Contact, Relationship, Relationship Owner and authorized relationship projection. Customer data presence in a platform store does not transfer the customer relationship.

Capability Evidence derived from work also does not transfer ownership of the underlying customer, Matter or private Reflection.

### Chapter 06 — Product Installation

A Product Installation is a Workplace-scoped record of edition, entitlement, enabled Capabilities, configuration, permissions, data-access scope, projection scope and lifecycle status.

Product-local Capability Profile, Skill Tree, Mentor and growth experiences attach through the Product Installation or Workplace context rather than becoming universal Core judgments.

## Part III — Capability semantics

### Chapter 07 — Capability Definition

A Capability Definition represents a governed, versioned ability to produce a defined outcome.

It includes:

- Capability ID and version;
- domain and family;
- outcome contract;
- input and output contracts;
- implementation modes;
- required Data and Knowledge;
- authority prerequisites;
- risk class;
- review policy;
- qualification restrictions;
- Evidence requirements;
- status and compatibility;
- version lineage and change rationale.

```text
Capability
= Stable Outcome Contract
+ Governed Implementation
+ Evidence Base
+ Version Lineage
+ Controlled Evolution
```

### Chapter 08 — Capability hierarchy and domains

Core standardizes the relationship:

```text
Capability Domain
→ Capability
→ Skill
→ Action / Invocation
```

A Capability Domain groups related outcome classes. A Capability defines the stable governed outcome. A Skill defines a scenario-specific executable implementation. An Action or Invocation is one real attempt to use that Capability or Skill.

Personas and roles such as Founder, Attorney or Operator are not automatically Capability Domains. They may reference Capability Profiles spanning several Domains.

### Chapter 09 — Skill Definition

A Skill Definition may include:

- Skill ID and version;
- parent Capability reference;
- trigger and scenario;
- input form or prompt contract;
- workflow or tool bindings;
- model-routing constraints;
- review and authority policy;
- expected output and Return;
- privacy and provenance requirements;
- implementation status.

A Skill may implement only part of a Capability. A Prompt is one possible Skill component and is not a shared semantic unit by itself.

### Chapter 10 — Capability asset classes

Core distinguishes:

- `SourceCapability` — a provenance-bound reconstruction of one source method;
- `CompositeCapability` — a candidate synthesis across sources or practice;
- `VerifiedMOCapability` — a governed released platform Capability version;
- `ExperimentalCapability` — a non-canonical Capability allowed only under explicit conditions.

```text
Source Capability
≠ Verified MO Capability
```

These classes preserve provenance and validation state without claiming that every source method belongs in platform canon.

### Chapter 11 — Capability levels, modes and profiles

Core standardizes references for:

- L1–L5 proficiency or progression levels;
- M0–M5 human–AI collaboration modes;
- practiced, reflected, assessed and verified states;
- Capability Profile references.

Core does not own every training curriculum, XP rule, Skill Tree visualization or Mentor message.

### Chapter 12 — Certification, authorization and qualification

Core distinguishes:

- CapabilityCertification;
- ProductionAuthorization;
- ProfessionalQualification;
- AssessorAuthorization;
- CredentialStatus.

These records may reference different issuers, Evidence, jurisdiction and expiry rules.

```text
CapabilityCertification
≠ ProfessionalQualification
≠ Matter-specific Authority
```

## Part IV — Capability composition and evolution semantics

### Chapter 13 — Capability Composition

A Capability Composition records how several Capabilities are combined for one governed objective.

It may contain:

- exactly one Primary Capability;
- zero to three Supporting Capabilities by default;
- zero or one Critic Capability;
- Capability Budget;
- decision-node responsibilities;
- conflicts and assumptions;
- resolution rationale;
- final action-plan contract.

Core defines the interoperable contract. Execution evaluates and runs it. Products decide how it is presented.

### Chapter 14 — Capability Conflict

A Capability Conflict records materially different recommendations, source provenance, assumptions, context, trade-offs, selected resolution and unresolved uncertainty.

Conflict must not be silently converted into an averaged answer.

### Chapter 15 — Reflection Candidate

A Reflection Candidate is a structured proposal derived from one real use.

It may reference:

- problem type;
- Capability and Skill versions;
- user disposition and modifications;
- Outcome and Evidence;
- observed difficulty;
- possible reusable lesson;
- privacy, rights and de-identification state.

It is not the raw conversation and is not yet canonical Knowledge or Capability Evidence.

### Chapter 16 — Case and Principle Candidate

A Case records a governed practice example linking context, actions, Evidence, Outcome and boundaries. It may be Private, Team, Enterprise, Anonymous Public or Platform Verified.

A Principle Candidate proposes a reusable rule derived from one or more Reflections or Cases.

Recommended Principle states are:

```text
Candidate
→ Observed
→ Repeated
→ Verified
→ Established
```

### Chapter 17 — Capability Change Proposal and version lineage

A Capability Change Proposal binds:

- Capability and current version;
- proposed change;
- supporting Reflections, Cases, Principles and Evidence;
- compatibility and downstream impact;
- rights and privacy review;
- reviewer and decision;
- release or rejection reason.

No Outcome, Reflection or Case silently mutates the active Capability Definition.

### Chapter 18 — Capability Ledger

A Capability Ledger Entry records an attributable Capability-related event such as invocation, practice, Reflection, assessment, accepted Case, certification, authorization or version exposure.

Core may define the interoperable event contract. Product and Workplace systems decide which private events are visible, retained or displayed.

```text
Capability Ledger
≠ universal score of a Person
≠ professional qualification registry
```

## Part V — Demand, commerce and collaboration

### Chapter 19 — Need and Opportunity

Need expresses a required outcome. Opportunity expresses a potential route connecting a Need with Capability, asset, audience, capacity or Provider supply.

A Need may trigger a Capability Composition, but the composition does not itself create customer authority or Provider appointment.

### Chapter 20 — Engagement

Engagement records the agreed collaboration boundary: participants, roles, purpose, disclosure scope, commercial arrangement, responsibility, customer relationship and termination.

### Chapter 21 — Order and Work Package

Order represents a formal customer-facing commercial commitment. WorkPackage represents a bounded unit required to fulfill an Order or another authorized objective.

### Chapter 22 — Assignment and Contribution

Assignment authorizes a Person or Workplace to perform a WorkPackage. Contribution is the submitted work, opinion, verification, artifact or professional input.

Contribution is not an accepted Outcome until reviewed under the applicable policy. A Contribution may support a Case or Reflection Candidate but does not automatically prove a Capability.

### Chapter 23 — Review, Outcome and Evidence

Core defines review decisions, revision requests, accepted outcomes, rejected outcomes, Evidence lineage and confidence.

Evidence must preserve source, actor, time, method, authority, version and permitted use.

### Chapter 24 — Compensation and Settlement references

Core defines neutral references for task compensation, commission, revenue share, professional service fee and settlement status without prescribing all commercial policy.

## Part VI — Data and Knowledge contracts

### Chapter 25 — Source and provenance

All Data, Knowledge and Source Capability products preserve source identity, retrieval time, rights, hash, version, extraction path, transformation history and review state.

### Chapter 26 — Structured data products

Core defines canonical references for Trademark, Applicant, Owner, Agent, Attorney, GoodsServiceItem, Jurisdiction, StatusEvent, DeadlineCandidate and EntityResolutionCandidate.

### Chapter 27 — Knowledge objects

Core defines KnowledgeSource, KnowledgeDocument, KnowledgeClaim, KnowledgeVersion, Citation, ReviewState and KnowledgeLink.

Markdown is an approved representation, not the semantic authority itself. Knowledge Atoms remain provenance-bound and do not automatically become Capability Principles.

### Chapter 28 — Brain results

Brain outputs must be typed as retrieval, inference, Recommendation, candidate, conflict analysis or explanation.

Brain may cite formal facts and Source Capabilities but may not silently mutate formal state, resolve professional authority or release a new Capability version.

## Part VII — Handoff, Return and network semantics

### Chapter 29 — Projection

Projection exposes a purpose-limited view from an authorized source context. Projection never implies unrestricted reuse.

Capability Profile and Reflection projection require explicit scope and privacy controls.

### Chapter 30 — Handoff

Handoff carries typed context, purpose, permissions, minimum necessary information, version and acceptance state.

### Chapter 31 — Return and Session Receipt

Return carries Outcome, Evidence or status back to the origin. The destination validates Return before formal acceptance.

A Session Receipt is a typed Return for one Capability invocation. It may reference Capability and Skill versions, implementation binding, Outcome, user disposition, Evidence, privacy state and Reflection Candidate.

```text
Session Receipt
≠ raw conversation archive
≠ automatic learning approval
```

### Chapter 32 — Provider and contributor distinction

Core distinguishes a bounded Contributor from a Provider Organization bearing external professional responsibility. MGSN supply, MarkReg production and platform contribution may use overlapping people but different authority contracts.

Provider performance Evidence may support Capability evolution only through the governed candidate and Change Proposal path.

## Part VIII — Compatibility and governance

### Chapter 33 — Public, shared and Product-local records

Core defines public contracts and shared semantic records. Products may define local records where the concept does not require cross-Product interoperability.

Product-local records normally include detailed Skill Tree UI, XP, Mentor prompts, private journaling and Persona-specific Capability Twin presentation.

### Chapter 34 — Versioning

All public contracts use explicit versioning, compatibility policy, deprecation and migration Evidence.

Capability versions must preserve Evidence basis, change rationale and affected implementation bindings.

### Chapter 35 — Change governance

Changes to frozen Core semantics require impact analysis, Change Proposal, conformance updates and downstream migration planning. Product desire alone is not sufficient reason to alter Core.

Capability Change Proposals do not automatically become Core semantic changes. Most Capability evolution should occur inside the governed Capability registry without expanding Core vocabulary.

## Ten-year lock

Core must remain smaller than the total platform.

Its durability comes from semantic discipline, not from absorbing every new feature, database, workflow, business rule, Reflection, Case, user score or coaching experience.
