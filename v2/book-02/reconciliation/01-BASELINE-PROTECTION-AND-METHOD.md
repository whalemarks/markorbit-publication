# Baseline Protection and Reconciliation Method

## 1. Why Book 02 is different

Books 01, 03, 04, 05, 06 and 07 may expand strategic, product and operating narratives while preserving the active Canon.

Book 02 defines shared semantics. A casual wording change can alter identifiers, object boundaries, lifecycle expectations, validation rules or downstream implementation obligations.

Therefore Book 02 v2 must proceed by reconciliation rather than unrestricted expansion.

## 2. Authority order

For this work, the authority order is:

```text
accepted Core baseline
→ active Portfolio Canon
→ accepted governance decisions
→ v2 cross-book manuscripts
→ product proposals and implementation ideas
```

A later manuscript may reveal a real semantic gap, but it does not automatically overrule the frozen Core.

## 3. Classification test

Each v2 term is tested against five questions.

1. Does the frozen Core already define the same meaning?
2. Can the meaning be represented by composing existing objects, references, evidence and states?
3. Is the term only a profile, role, view or Product-local specialization?
4. Must multiple Products exchange this concept using one stable contract?
5. Would introducing it change existing meaning, lifecycle, ownership or compatibility?

The result is assigned one class.

### F0 — already covered

The concept is already present in the frozen semantic model, even if the v2 manuscript uses a more descriptive label.

### F1 — composition

No new atomic Core object is required. Existing objects and references can express the concept when combined under an explicit profile.

### F2 — explanatory profile

The concept deserves a documented shared profile or controlled specialization but should not become a new universal business object.

### F3 — candidate addition

The concept appears cross-product, durable and semantically distinct enough to justify a candidate shared contract. It still requires formal review.

### F4 — change proposal

The concept would modify an existing frozen meaning, introduce a new authoritative lifecycle or impose migration obligations. It requires a formal Core Change Proposal.

## 4. Core admission criteria

A concept should enter Core only when all of the following are substantially true:

- at least two independent Products or platform layers require it;
- the meaning must remain stable across those contexts;
- Product-local implementation would create semantic divergence;
- the object has a clear authority and lifecycle boundary;
- the object can be defined without importing Product-specific policy;
- compatibility, versioning and migration can be governed;
- the benefit exceeds the cost of expanding Core.

## 5. Core exclusion criteria

A concept should remain outside Core when it is mainly:

- a UI arrangement;
- a recommendation algorithm;
- a pricing policy;
- a course curriculum;
- a Product-specific workflow;
- a marketplace ranking rule;
- a provider procurement package;
- an internal operational checklist;
- a model prompt or tool binding;
- a temporary implementation convenience.

## 6. Authority protection rules

The reconciliation preserves these locks:

```text
Core defines meaning; it does not own every record.
Owning Services create and mutate formal business state.
Execution governs work; it does not become semantic or business owner.
Workplace preserves business sovereignty; it does not redefine Core meaning.
Brain produces typed intelligence; it does not silently create facts.
Projection does not transfer ownership.
Return does not become truth until validated.
Certification does not replace legal qualification.
Provider recommendation does not equal appointment.
```

## 7. Required output of this phase

This phase must produce:

- a term-by-term mapping;
- object admission decisions;
- candidate Change Proposals;
- a reconciled Book 02 v2 manuscript that clearly labels active, compositional and candidate semantics;
- a next-gate recommendation.

It must not produce silent semantic activation.
