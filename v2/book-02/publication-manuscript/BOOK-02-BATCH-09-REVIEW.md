## Main decisions

### 1. Versioning covers meaning, not only syntax

```text
Contract Version
≠ Record Version
≠ Source Version
≠ Product Version
≠ Implementation Version
```

Every public contract and shared record must remain interpretable against an explicit semantic version.

```text
Schema Compatible
≠ Semantically Compatible
```

Authority, lifecycle, Evidence and Unknown behavior are part of the contract.

### 2. Historical records remain bound to their original contract

```text
Latest Contract Published
≠ Historical Record Invalid
```

Prior records and Decisions must remain reconstructable under the contract, Evidence and authority active at the time.

### 3. Compatibility is multidimensional and directional

```text
Syntactic Compatibility
Semantic Compatibility
Lifecycle Compatibility
Authority Compatibility
Evidence Compatibility
Operational Compatibility
Historical Compatibility
```

A consumer that can parse a record may still be unable to interpret it safely.

### 4. Deprecation is not deletion

```text
Deprecated
≠ Superseded
≠ Withdrawn
≠ Invalidated
```

Each state requires its own consumer, creation, migration and historical-retention behavior.

### 5. Migration must preserve meaning and evidence

```text
Migration Job Completed
≠ Migration Semantically Accepted
```

Migration requires inventory, semantic mapping, conformance fixtures, unresolved-record handling, active-work policy, consumer acceptance, rollback and post-migration evidence.

```text
Template Updated
≠ Active Instance Updated
```

### 6. Core admission remains controlled by F0–F4

```text
F0 — Already Covered
F1 — Composition
F2 — Explanatory Profile
F3 — Candidate Shared Addition
F4 — Formal Core Change Proposal Required
```

The classification selects the smallest safe treatment. It is not a maturity ladder.

```text
F3 Candidate
≠ Active Core Object
```

### 7. Publication, prototypes and repeated local use do not activate Core

```text
Publication Prose
≠ Semantic Activation

Prototype Exists
≠ Core Object Active

Repeated Product Use
≠ Automatic Core Admission
```

Activation still requires approved contract, conformance, migration planning and Owner authority.

### 8. Contract approval and production migration remain separate

```text
Core Contract Approved
≠ Production Migration Authorized
```

Migration may require staged implementation, compatibility windows, supervised rollout and rollback evidence.

### 9. The ten-year lock keeps Core smaller than the platform

```text
Core must remain smaller than the total platform.
```

Core preserves shared meaning, authority, Candidate-before-Canonical, versioning and compatibility. It does not absorb every Product record, workflow, model, prompt, pricing rule or data store.

### 10. Canonical does not mean centralized

```text
Canonical Record
= accepted under declared authority

Canonical Record
≠ stored in Core by definition
```

Owning Services remain responsible for formal records while Core provides shared references and contracts.

## Cross-book consistency

### Book 03 — Execution

Book 03 continues to own runtime validation, Assignment, Review, Approval, Apply, Return and reconciliation.

```text
Core Contract Version
≠ Workflow Version
```

Active work cannot be silently reinterpreted under a new contract.

### Book 05 — MarkReg

MarkReg Product changes and domain-specific lifecycle updates remain Product or Owning Service concerns unless shared semantics genuinely require Core admission.

### Book 06 — Lite

Lite marketplace, content and Opportunity schema changes remain Product-local where cross-product stable meaning is not required.

### Data, Knowledge and Brain

Data Product, Knowledge and Brain versions evolve independently and reference Core contracts where necessary.

```text
One Layer Changed
≠ Every Layer Silently Upgraded
```
