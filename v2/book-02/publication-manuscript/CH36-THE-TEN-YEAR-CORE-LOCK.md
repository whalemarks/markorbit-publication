# CH36 — The Ten-year Core Lock

## Core must remain smaller than the platform

The final constitutional test for Book 02 is simple to state and difficult to maintain:

```text
Core must remain smaller than the total platform.
```

MarkOrbit will add new Products, Workplaces, data sources, Knowledge systems, AI models, provider networks, workflows, commercial routes and professional services.

If Core absorbs every new feature, object, status and business rule, it will stop being a stable semantic constitution and become a fragile mirror of current implementation.

The ten-year Core lock protects the opposite principle:

```text
Durability comes from semantic discipline,
not from modeling everything centrally.
```

## What Core must preserve over time

A durable Core should preserve the shared distinctions that independent systems cannot safely redefine for themselves.

These include:

- identity and reference continuity;
- Workplace business sovereignty;
- shared semantic authority;
- Owning Service formal-state authority;
- Candidate-before-Canonical progression;
- Evidence and provenance lineage;
- Product, Capability, Workflow, Tool and Implementation separation;
- Order, Matter, Work Package and Assignment separation;
- Contribution, Review, Outcome and formal-state separation;
- Projection, Handoff and Return boundaries;
- versioning, compatibility and change governance;
- professional and execution authority distinctions.

The exact technologies and Product experiences may change. These semantic boundaries should remain stable unless a formal constitutional change is justified.

## What Core should not attempt to freeze

Core should avoid freezing volatile details such as:

- user-interface layouts;
- model and prompt choices;
- retrieval architecture;
- provider ranking;
- pricing;
- marketing policy;
- course curricula;
- country-specific checklists;
- local workflow steps;
- temporary operational states;
- storage engines;
- Product-specific data models.

```text
Important to the Business
≠ Appropriate for Core
```

A concept can be strategically critical while remaining Product-owned, Workflow-owned, Knowledge-owned or Data-owned.

## Stable grammar, evolving vocabulary

The ten-year lock does not mean Core never changes.

It means change should occur at the smallest necessary level.

A stable grammar may support new profiles, controlled values and compositions without creating a new root object for every use case.

```text
New Use Case
→ Test Existing Grammar First
→ Compose or Profile Where Possible
→ Add Shared Contract Only When Necessary
```

This approach allows the platform to grow without forcing every Product to migrate each time one Product develops a new feature.

## Authority boundaries are more durable than workflows

Workflows will change as automation improves. AI may perform tasks that previously required manual work. Provider networks may expand. New interfaces may reduce the number of questions asked of a customer.

None of those changes should erase:

```text
Model Can
≠ Model May

Review
≠ Approval

Approval
≠ Apply

Return
≠ Official Truth

Execution
≠ Formal-state Ownership
```

Authority boundaries must survive changes in capability.

## AI does not invalidate the lock

More capable models may retrieve, reason, draft and execute with increasing autonomy.

The platform may move some work from M2 or M3 into governed M4 routes. It may replace human preparation with deterministic or AI-assisted implementation.

The Core meaning of authority, evidence, version, review and formal state should not change merely because the model is stronger.

```text
Model Improvement
≠ Constitutional Authority Expansion
```

AI implementation details belong in versioned Implementation Profiles, policies and execution contracts.

## Product success does not justify semantic capture

A successful Product may become the dominant entry point to the platform. Its internal terminology may become familiar to users and developers.

That does not give the Product authority to redefine platform-wide meaning.

```text
Most-used Product
≠ Semantic Authority
```

MarkReg should not redefine every Matter around trademark filing. Lite should not redefine every Opportunity around marketplace growth. Sites should not redefine shared records around public presentation.

Core must serve all legitimate consumers rather than mirror the largest current Product.

## One database is not the lock

A single physical database may be practical during one implementation phase. Multiple services may later separate.

The ten-year lock is not a deployment topology.

```text
One Database
≠ One Semantic Owner

Multiple Services
≠ Multiple Meanings
```

Even in one database, the platform should preserve Owning Service, authority and lifecycle boundaries. Even across many services, shared references and contracts should remain consistent.

## Canonical does not mean centralized

A record can be canonical within its Owning Service without being copied into a universal central table.

Core can define how the record is referenced, versioned and projected.

```text
Canonical Record
= accepted under declared authority

Canonical Record
≠ stored in Core by definition
```

This allows formal business records, official data, Knowledge Claims and Product-local records to remain with the accountable owner.

## Interoperability without ownership transfer

The platform should be able to exchange:

- identity references;
- Capability contracts;
- Evidence;
- Projections;
- Handoffs;
- Returns;
- Decisions and status references;
- migration evidence.

The exchange should not imply:

- customer relationship transfer;
- unrestricted data reuse;
- formal-state mutation rights;
- professional authority;
- shared commercial ownership.

```text
Interoperable
≠ Ownerless
```

## Unknown must remain representable

A durable semantic system must survive incomplete and conflicting information.

It must preserve:

```text
Unknown
Conflicted
Unverified
Candidate
Stale
Superseded
Withdrawn
```

A platform that requires one clean answer for every field will repeatedly convert uncertainty into false certainty.

```text
Clean Current Value
≠ Trustworthy State
```

Candidate-before-Canonical is therefore a ten-year rule, not merely an early-stage data practice.

## Historical truth must remain reconstructable

Over ten years, the platform will change rules, providers, models, source systems and product architecture.

It must still be possible to reconstruct:

- which contract version applied;
- which source and Evidence were available;
- which actor held authority;
- which model or implementation was used;
- what Decision was made;
- what external effect occurred;
- how formal state was established or corrected.

```text
Latest System State
≠ Complete Historical Explanation
```

Append-only history, supersession and correction are more durable than destructive overwrite.

## Core must be teachable

A semantic constitution is useful only if Product teams, developers, operators and professionals can understand it.

The Core should therefore favor:

- clear distinctions;
- limited root concepts;
- explicit authority;
- observable lifecycle;
- versioned examples;
- controlled profiles;
- consistent terminology.

A model that requires every user to understand hundreds of universal objects has failed the ten-year test.

```text
More Complete Model
≠ More Usable Constitution
```

## Core must be testable

Constitutional boundaries should be represented in conformance tests and invariants.

Examples include:

- Projection does not grant source mutation;
- Return does not directly establish formal state;
- Product entitlement does not create Assignment;
- Assignment does not create Professional Authority;
- AI inference does not become canonical without acceptance;
- payment orchestration does not imply custody;
- Product-local records do not become shared without admission;
- unsupported contract versions block or escalate rather than guess.

```text
Principle Written
≠ Principle Enforced
```

The ten-year lock should be visible in contracts, fixtures and release gates.

## Core must be replaceable only through evidence

There may come a time when the current Core baseline no longer supports the platform safely.

Replacement should require:

- exact inventory of the existing baseline;
- field and lifecycle mapping;
- cross-book impact analysis;
- alternative models;
- migration cost and risk;
- conformance strategy;
- Owner approval;
- reversible rollout where possible.

```text
New Manuscript Complete
≠ Frozen Core Replaced
```

Publication can prepare the argument and candidate architecture. It does not itself authorize replacement.

## The ten-year responsibility map

The long-term platform should continue to preserve:

```text
Core
= shared meaning, contracts and compatibility

Workplace
= business sovereignty

Product
= audience experience and commercial promise

Execution
= governed work and consequential transitions

Data
= sourced structured records and releases

Knowledge
= sourced Claims, Versions and Citations

Brain
= typed context-specific intelligence

Owning Service
= formal-state validation and mutation

Official or Professional Authority
= external reserved fact or action authority
```

No layer should silently absorb the others.

## Closing equation

Book 02 can be summarized as:

```text
Durable Core
= Shared Meaning
+ Explicit Authority
+ Candidate-before-Canonical
+ Versioned Contracts
+ Compatibility Governance
+ Evidence-backed Change
− Product-specific Volatility
− Centralized Ownership Assumptions
```

## Constitutional rule

```text
Core must remain small enough to stay stable,
precise enough to preserve interoperability,
and strict enough to prevent authority collapse.

It should define the language the platform shares
without becoming the platform itself.
```
