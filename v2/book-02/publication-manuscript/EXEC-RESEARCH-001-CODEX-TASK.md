```text
Production Authorization Permission Profile
Work Package Task / Workflow Profile
Bilateral Assignment Lifecycle Profile
Contribution Submission and Supersession Profile
Outcome Acceptance without Formalization Profile
```

## Minimum validation rules

### Production Authorization

Require:

- subject reference;
- action or Capability reference;
- bounded scope;
- Permission and Policy references;
- effective period or explicit no-expiry reason;
- implementation/model version for machine subjects;
- Human Review requirement where protected risk applies.

Reject:

```text
Production Authorization = Assignment
Production Authorization = Professional Authority
```

### Work Package

Require:

- package identifier;
- parent Matter, Order or declared standalone purpose;
- Workflow Contract reference;
- one or more Task references;
- package purpose and exclusions;
- acceptance criteria reference.

Reject:

```text
Every Task = Work Package
All Tasks Completed = Work Package Accepted
```

### Bilateral Assignment

Require:

- exact work and version;
- assigning and accepting parties;
- assignee identity;
- offer and acceptance evidence;
- Permission, Policy and Eligibility references;
- effective period;
- revocation behavior.

Reject:

```text
Eligibility = Assignment
Assignment = Professional Authority
Assignment = Customer Relationship Transfer
```

### Contribution

Require:

- contributor identity;
- represented organization where applicable;
- exact work basis;
- artifact or Evidence references;
- submission version;
- submission time;
- supersession reference where applicable.

Reject:

```text
Document exists = Contribution accepted
Contribution submitted = Task completed
```

### Outcome

Require:

- accepted input versions;
- declared acceptance purpose;
- Human Review reference;
- accepting actor or service;
- accepted scope and exclusions;
- expiry or recheck behavior where applicable.

Reject:

```text
Outcome accepted = Apply
Outcome accepted = external submission
Outcome accepted = formal-state mutation
```

## Acceptance criteria

- all research tests pass;
- frozen Core package dependency remains unchanged or is used read-only;
- no new public package export is introduced;
- all fixtures are visibly marked `noncanonical` and `research-only`;
- validation report lists passed and rejected cases;
- repository README or research README states that no semantic activation occurred;
- PR contains only fixture research and tests.

## Required final report

Codex must report:

- files added and changed;
- command used to validate;
- positive and negative fixture counts;
- exact boundary tests passed;
- whether any frozen contract proved insufficient;
- whether the result supports or rejects a later formal Change Proposal.
