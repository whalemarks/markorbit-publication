# Book 02 v2 — Publication Closing

## 1. Closing thesis

Book 02 defines the shared semantic constitution of MarkOrbit.

Its purpose is not to collect every useful noun, feature, workflow, database table or AI result into one central model. Its purpose is to preserve stable meaning, authority, lifecycle, evidence, versioning and compatibility across independent Products, Workplaces, Execution systems, Data services, Knowledge systems, Brain implementations and Owning Services.

The governing proposition is:

```text
Core defines shared meaning.
Core does not own every record.
```

The book therefore closes around five permanent separations:

```text
Semantic Authority
≠ Business Sovereignty
≠ Formal-state Authority
≠ Physical Custody
≠ Professional Authority
```

These distinctions allow MarkOrbit to become more integrated without becoming more centralized than necessary.

## 2. The constitutional architecture

The long-form manuscript now establishes the following responsibility map:

```text
Core
= shared semantics, identifiers, contracts and compatibility

Workplace
= business sovereignty and Product-operation boundary

Product
= audience experience, entitlement and commercial promise

Execution
= governed coordination of consequential work

Data
= sourced structured records, transformations and releases

Knowledge
= sourced Claims, Versions, Citations and review state

Brain
= typed retrieval, inference, recommendation, candidate and explanation

Owning Service
= formal-state validation and mutation

Official / Professional Authority
= reserved external fact or action authority
```

No layer acquires another layer's authority merely because it can read, display, process or coordinate the same record.

## 3. The permanent semantic locks

Book 02 preserves the following non-equivalences across the platform:

```text
Person ≠ Organization ≠ Workplace ≠ Membership
Customer ≠ Applicant ≠ Owner ≠ Payer ≠ Signatory
Product ≠ Capability ≠ Workflow ≠ Tool ≠ Implementation
Order ≠ Matter ≠ Work Package ≠ Assignment
Evidence ≠ Claim ≠ Decision ≠ Formal State
Need ≠ Opportunity ≠ Engagement
Contribution ≠ Review ≠ Outcome
Certification ≠ Qualification ≠ Production Authorization
Projection ≠ Handoff ≠ Return ≠ Transfer
Contributor ≠ Provider ≠ Professional Authority
Data Product ≠ Knowledge Claim ≠ Brain Result
Public ≠ Shared ≠ Product-local
Deprecated ≠ Superseded ≠ Withdrawn ≠ Invalidated
```

These locks are more important than any one object name. They prevent one Product, model or workflow from silently acquiring semantic, commercial or formal-state power.

## 4. Candidate-before-Canonical

The book treats uncertainty as a governed state rather than a defect to be hidden.

```text
Observation
→ Candidate
→ Review or Validation
→ Accepted Record or Outcome
→ Formal-state Mutation where authorized
```

Therefore:

```text
AI Output ≠ Professional Decision
Provider Return ≠ Accepted Truth
Contribution ≠ Outcome
Recommendation ≠ Customer Instruction
Candidate ≠ Canonical Record
```

This principle applies to identity resolution, deadlines, fees, provider practice, Knowledge Claims, Brain results, customer facts and formal-state transitions.

## 5. Core admission discipline

The F0–F4 gate remains the controlling method for every proposed semantic addition:

```text
F0 — Already Covered
F1 — Composition
F2 — Explanatory Profile
F3 — Candidate Shared Addition
F4 — Formal Core Change Proposal Required
```

Publication prose, repeated Product use, prototypes and implementation code do not activate Core semantics.

```text
Publication Prose
≠ Semantic Activation
```

A candidate becomes active only through an authorized contract, conformance, compatibility and migration process.

## 6. The ten-year lock

The manuscript closes with a deliberate asymmetry:

```text
Core must remain smaller than the total platform.
```

Core should remain stable while the following continue to change:

- user interfaces;
- Products and pricing;
- AI models and prompts;
- provider networks;
- Workflows and Tools;
- storage engines and indexes;
- jurisdiction-specific procedures;
- recommendation and ranking methods.

Core should absorb only the shared meaning whose divergence would create unacceptable interoperability, authority or lifecycle risk.

## 7. What this manuscript does not authorize

This publication reconstruction does not:

- modify Active Canon;
- replace the frozen Book 02 / Core baseline;
- activate any F3 or F4 candidate;
- authorize production implementation or migration;
- move Product-local policy into Core;
- make Core the universal database, Knowledge repository or Brain runtime;
- grant AI, Contributors or Providers professional authority;
- transfer customer relationships, payment custody or formal-state ownership.

## 9. Final statement

Book 02 is complete as a first-pass publication manuscript because it now explains the constitutional meaning of Core, the boundaries around authority and ownership, the shared semantic families, the admission gate and the long-term change discipline.

It is not complete as a replacement specification until every candidate is mapped against the frozen baseline, every material change is formally proposed, every compatibility obligation is tested and every migration decision is separately authorized.

The closing rule is therefore:

```text
Shared meaning should become stronger.
Shared ownership should not become broader by implication.
```
