## Main decisions

### 1. Data references do not transfer ownership or authority

```text
Source Record
≠ Structured Record
≠ Data Reference
≠ Data Product
≠ Projection
≠ Formal Business State
```

Data-owning services preserve source content, transformation, quality, rights and release responsibility. Core provides shared references and contracts.

```text
Data Reference available
≠ data ownership transferred
≠ source state editable
≠ unrestricted reuse permitted
```

Data Product remains an F2/F3 versioned product-reference concept. Canonical data content remains Data-owned.

### 2. Refinery output remains layered

```text
Raw Source Layer
→ Normalized Source Layer
→ Candidate Resolution Layer
→ Accepted Data Layer
→ Purpose-limited Projection Layer
```

Normalization does not establish verification. Candidate generation does not establish canonical identity, deadline or formal state.

```text
clean output
≠ truthful output by default
```

The Refinery owns transformation accountability, not the official facts represented or the customer relationship connected to them.

### 3. Knowledge objects preserve traceable meaning

```text
Knowledge Source
≠ Knowledge Document
≠ Knowledge Claim
≠ Knowledge Version
≠ Citation
≠ Knowledge Link
≠ Formal Business State
```

Source identifies origin. Document preserves representation. Claim states a proposition. Version preserves change. Citation connects use to support. Review establishes purpose-specific acceptance.

Markdown remains an approved representation rather than semantic or official authority.

```text
Markdown note
≠ official source
≠ accepted Claim
≠ formal business record
```

### 4. Knowledge promotion remains governed

```text
Source Acquired
→ Document Preserved
→ Claim Candidate Extracted
→ Source and Scope Validation
→ Review
→ Accepted Knowledge Version
```

AI extraction, storage or repetition does not create verified reusable Knowledge.

Matter Context, Workplace Knowledge, Shared Knowledge and Model-training Data remain separate permission domains.

### 5. Brain results remain typed

```text
Retrieval
Inference
Recommendation
Candidate
Explanation
```

Each type has a different purpose and authority boundary:

```text
Retrieved
≠ Verified

Inference
≠ Fact

Recommendation
≠ Customer Instruction

Candidate Generated
≠ Candidate Accepted

Explanation
≠ Authoritative Source
```

A single conversational interface may contain several result types, but the interface must not flatten their epistemic status.

### 6. Brain remains non-canonical

```text
Brain Result
≠ Canonical Fact
≠ Decision
≠ Approval
≠ Formal State
```

Brain may contribute to a governed route:

```text
Brain Result
→ Contribution or Candidate
→ typed Review
→ Decision or Approval
→ governed Apply
→ Evidence Return
→ Owning Service validation
```

Brain cannot self-promote a result, create professional authority, infer customer instruction or mutate formal state by confidence alone.

### 7. Core remains smaller than the intelligence stack

```text
Core
≠ Data Store
≠ Data Refinery
≠ Knowledge Repository
≠ Brain Runtime
≠ Formal-state Owning Service
```

Core defines shared grammar, identifiers, contracts, authority boundaries and compatibility. It does not own every record, source, Claim, model result, prompt, retrieval index or pipeline.

### 8. The cross-layer path remains reconstructable

```text
Source Record
→ Structured Data Record
→ Knowledge Claim Candidate
→ Reviewed Knowledge Version
→ Typed Brain Result
→ Contribution or Decision Candidate
→ Review and Approval
→ Governed Apply
→ Evidence Return
→ Owning Service Formal-state Validation
```

Each transition changes purpose or authority and must preserve versions, evidence and applicable acceptance rules.

## Cross-book consistency

### Book 03 — Execution

Book 03 continues to own:

- Capability Request validation;
- Work Package and Assignment runtime;
- Contribution and typed Review;
- Approval and Apply;
- Return, retry and reconciliation;
- AI provenance and memory governance.

```text
Core intelligence contract
≠ Execution runtime
```

Data or Knowledge availability does not establish execution readiness or formal truth.

### Book 05 — MarkReg

MarkReg consumes trademark, owner, deadline, fee, provider and jurisdictional Data and Knowledge. Filing packages, customer decisions, lifecycle workflows and Matter formal state remain Product or Owning Service records.

```text
MarkReg consumes shared references
≠ MarkReg owns every source or Data Product
```

### Book 06 — Lite

Lite may use Data, Knowledge and Brain for content, trademark supply, demand signals and Opportunity explanations. Public data does not create a Customer, marketing consent or relationship ownership.

### Knowledge and Brain

Knowledge preserves reusable sourced Claims. Brain produces context-specific typed results over Data, Knowledge and task context.

```text
Knowledge Claim
≠ Brain Inference

Brain Result accepted for one task
≠ reusable Shared Knowledge
```

### Workplace and Products

Workplace sovereignty, Product Installation boundaries, customer relationships and purpose-limited access remain preserved. Shared intelligence does not create cross-Workplace ownership or unrestricted reuse.

## Rights and training locks

```text
Operational Access
≠ Shared Knowledge Permission
≠ Publication Permission
≠ AI-training Permission
```

Customer and Matter data do not become reusable Knowledge or training data through de-identification or operational use alone.
