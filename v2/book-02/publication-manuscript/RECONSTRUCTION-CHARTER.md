# Book 02 v2 — Reconstruction Charter

## 1. Purpose

The purpose of this reconstruction is to turn the accepted and reconciled Core material into a readable long-form specification without changing the frozen semantic authority by implication.

Book 02 must explain why Core exists, what belongs in it, what remains outside it, how authority is separated, how shared contracts evolve and how downstream systems consume Core safely.

## 2. Source families

The reconstruction must reconcile four source families:

1. the accepted frozen Core baseline;
2. the active Portfolio Canon and accepted governance decisions;
3. `v2/book-02/BOOK-02-v2-MASTER.md` and the semantic-reconciliation package;
4. current Books 03, 05 and 06 publication reconstructions.

## 3. Special constraint

Book 02 is not an unrestricted strategic rewrite.

```text
Useful cross-product concept
≠ activated Core object
```

Every concept must be classified as one of:

```text
F0 — already covered
F1 — composition of existing semantics
F2 — explanatory profile or controlled specialization
F3 — candidate shared addition
F4 — formal Core Change Proposal required
```

## 4. Admission rule

A concept should enter Core only where stable interoperability requires one shared meaning across independent Products or platform layers and where product-local implementation would create unacceptable semantic divergence.

Core should reject concepts that are mainly:

- UI arrangements;
- recommendation algorithms;
- pricing policies;
- course curricula;
- product-specific workflows;
- provider procurement packages;
- model prompts or tool bindings;
- temporary implementation conveniences.

## 5. Cross-book boundary

Book 02 owns shared semantics and contract discipline.

Book 03 owns governed execution, including validation, assignment, review, approval, apply, return and reconciliation.

Book 05 owns MarkReg’s international trademark growth and lifecycle service model.

Book 06 owns Lite’s daily growth, content, trademark supply and demand, commercial opportunities and capability experience; Knowledge and Brain source-governance remain distinct from Core ownership.

Therefore:

```text
Core contract ≠ Execution runtime
Core object ≠ MarkReg product record
Core reference ≠ Knowledge store
Core semantics ≠ centralized business database
```

## 6. Writing standard

Each chapter should include:

- a sustained argument rather than a glossary dump;
- realistic platform examples;
- explicit authority and lifecycle boundaries;
- implementation consequences without prescribing one database or UI;
- compatibility and migration implications;
- candidate status where semantics are not activated.

## 7. Non-goals

This reconstruction does not:

- modify Active Canon;
- replace the frozen Core baseline;
- activate candidate additions;
- redefine Book 03 runtime behavior;
- move MarkReg, Lite or Knowledge product policy into Core;
- authorize production implementation or migration.
