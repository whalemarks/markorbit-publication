# CH22 — Assignment and Scoped Production Authority

## Assignment is where eligible capacity becomes bounded responsibility

A Work Package can define exactly what must be produced. A Capability record can describe what an actor or implementation may be able to do. Production Authorization can permit participation in a class of real work.

None of these determines who is responsible for the present package.

That transition requires Assignment.

```text
Capability
≠ Production Authorization
≠ Eligibility
≠ Assignment
```

Assignment is the candidate shared semantic that binds one exact Work Package version to one actor or implementation under explicit scope, access, authority, review, time and commercial conditions.

This chapter explains that proposed meaning. It does not activate Assignment in frozen Core.

## The central definition

Assignment answers:

> Who or what has accepted responsibility to perform this Work Package, for which represented context, using which access and authority, until when, and subject to what review, compensation, suspension and revocation rules?

A valid Assignment is narrower than a job title, Membership, provider relationship or platform role.

```text
Assignment
≠ employment
≠ Membership
≠ Provider appointment
≠ professional qualification
≠ permanent access grant
```

The same actor may hold many Assignments, and each may impose different boundaries.

## Preconditions

Before Assignment, the system should normally establish:

- a valid Work Package and version;
- performer or implementation identity;
- relevant Capability evidence;
- task-specific Eligibility;
- Production Authorization where real work is involved;
- professional Qualification where required;
- conflict and capacity checks;
- required reviewer availability;
- commercial or compensation terms;
- data-access feasibility;
- represented Workplace and role.

```text
candidate available
≠ candidate eligible
≠ candidate assigned
```

Book 03 owns the runtime validation of these conditions. Book 02 preserves the shared semantic distinctions.

## Minimum Assignment envelope

A proposed Assignment should preserve:

- stable identifier and version;
- Work Package identifier and exact version;
- assignee identity;
- assignee type: Person, Workplace, Provider route, AI implementation or deterministic service;
- represented Organization or Workplace;
- accepted role;
- Capability and implementation references;
- Eligibility decision and evidence;
- Production Authorization reference where applicable;
- professional Qualification and appointment references where applicable;
- allowed actions;
- prohibited actions;
- data scope and access operations;
- Tool and connector limits;
- customer-contact rule;
- review and supervision obligations;
- deadline and service level;
- compensation terms;
- acceptance time;
- effective period;
- suspension, expiry, revocation and reassignment rules;
- returned access and evidence obligations.

## Assignment offer and acceptance

Assignment should not exist solely because a dispatcher selected a name.

The lifecycle should distinguish:

```text
Candidate Identified
→ Assignment Offered
→ Assignment Accepted
→ Access Activated
→ Production Active
```

The proposed assignee may reject because of:

- conflict;
- insufficient time;
- missing data;
- inadequate compensation;
- scope ambiguity;
- absent professional authority;
- unavailable reviewer;
- security or jurisdiction restrictions.

```text
Assignment offered
≠ Assignment accepted
```

Acceptance should bind the assignee to the exact package version and conditions visible at acceptance time.

## Assignment and Product entitlement

A Product entitlement allows access to a Product capability or interface. It does not create responsibility for a specific customer outcome.

```text
Product entitled
≠ production authorized
≠ assigned
```

A person may be entitled to use MarkReg but not assigned to any filing package. An AI model may be available inside Lite but prohibited from processing a particular customer’s confidential documents.

## Assignment and Membership

Membership connects a Person to a Workplace. Assignment binds an actor to bounded work.

```text
Workplace administrator
≠ assignee for every Work Package
```

A Member may manage billing or configuration without having authority to view, prepare, review or approve professional work.

Conversely, an external contributor may hold a limited Assignment without becoming a general Workplace Member.

## Assignment and Provider appointment

An external provider relationship often requires a distinct lifecycle:

```text
Provider Candidate
→ Eligibility
→ Selection
→ Appointment
→ Provider Acceptance
→ Matter Instruction
```

A provider organization may then receive internal Work Package Assignments or return provider results under a Handoff.

```text
Assignment
≠ Provider Appointment
```

Provider Appointment concerns the external professional and commercial relationship. Assignment concerns bounded performance responsibility within a governed work structure.

## Scope of allowed actions

Assignment should express allowed actions explicitly.

Examples include:

- view specified source records;
- extract structured fields;
- draft a report;
- edit a document;
- contact a Provider through an approved channel;
- submit a Contribution;
- perform a declared Review type;
- use listed Tools under purpose limits.

Prohibited actions may include:

- contacting the customer;
- changing formal applicant data;
- appointing a new Provider;
- moving funds;
- filing with an office;
- publishing public content;
- reusing data for another Matter;
- training a model on the input.

```text
assigned to prepare
≠ assigned to approve
≠ assigned to apply
```

## Data access follows the Assignment

Access should be derived from the Work Package and Assignment, not from broad organizational convenience.

The Assignment should identify:

- visible fields and files;
- redactions;
- view, download, edit and disclosure rights;
- retention period;
- onward-sharing restriction;
- local-device rules;
- memory and training restrictions;
- post-completion deletion or archive duties.

```text
can view for delivery
≠ can reuse for marketing
≠ can retain indefinitely
```

When the Assignment ends, production access should normally expire even though historical evidence remains available to authorized auditors.

## Machine Assignment

AI and deterministic implementations can be assignees only through a bounded implementation reference.

A Machine Assignment should preserve:

- model, service, Skill and Tool versions;
- Capability and Work Package versions;
- M-mode;
- allowed and prohibited actions;
- data and memory scope;
- tool-call ceiling;
- cost or token ceiling;
- review requirement;
- stop and escalation conditions;
- side-effect and idempotency rules;
- evidence and audit requirements;
- expiry and reevaluation triggers.

```text
machine assigned
≠ autonomous general authority
```

An AI assignment to draft customer communication does not authorize sending. A deterministic connector assignment to validate fields does not authorize official submission.

## Human Assignment

A Human Assignment should not rely only on identity or job title.

It may require:

- verified Capability evidence;
- language and jurisdiction fit;
- professional Qualification;
- conflict clearance;
- current Production Authorization;
- secure environment;
- Tool access;
- reviewer relationship;
- contact permission.

```text
experienced professional
≠ eligible for every jurisdiction, role or deadline
```

## Hybrid responsibility

One Work Package may use several Contributions but should retain clear primary responsibility.

For example:

```text
AI Implementation
= extraction Contribution

Contributor
= draft preparation Contribution

Qualified Professional
= R2 review

Delivery Owner
= package completeness and handoff
```

Multiple Assignments may be used where the Work Package is deliberately decomposed or where different review roles are required.

```text
multiple actors involved
≠ shared undifferentiated responsibility
```

Each actor’s scope, authority and returned evidence must remain attributable.

## Review and supervision

Assignment may require supervision without transferring the supervisor’s authority.

```text
under supervision
≠ holds supervisor authority
```

The supervision contract should identify:

- supervisor;
- review frequency;
- mandatory checkpoints;
- approval types;
- escalation availability;
- maximum unsupervised action;
- evidence required before progression.

A named supervisor who is unavailable does not satisfy the control requirement.

## Customer contact

Customer contact must be explicit because it can affect relationship ownership, confidentiality and professional responsibility.

Possible settings include:

```text
No Customer Contact
Platform-mediated Contact
Supervised Direct Contact
Authorized Direct Professional Contact
Emergency Contact under Defined Rule
```

```text
access to customer data
≠ permission to contact customer
```

Direct contact also does not transfer the customer relationship.

## Compensation terms

Assignment may reference:

- fixed fee;
- hourly rate and cap;
- milestone;
- revision allowance;
- urgent-work adjustment;
- cancellation payment;
- expense approval;
- dispute route.

Compensation terms should be accepted with the Assignment rather than inferred after work begins.

```text
Assignment accepted
≠ every future scope change included
```

Material package amendments may require renewed commercial acceptance.

## Assignment lifecycle

A useful lifecycle may include:

```text
Draft
Offered
Accepted
Active
Paused
Suspended
Completed by Assignee
Closed after Acceptance
Expired
Revoked
Cancelled
Reassigned
```

`Completed by Assignee` should mean that the assignee claims the required Contribution has been submitted. It does not mean the Contribution was accepted or the Work Package was completed.

```text
assignee says complete
≠ Contribution accepted
≠ Work Package accepted
```

## Suspension

Suspension temporarily removes production permission while preserving history and potentially preserving the Assignment relationship.

Triggers may include:

- credential expiry;
- conflict discovered;
- security incident;
- customer instruction change;
- missing reviewer;
- quality threshold breach;
- unknown external effect;
- model or Tool change;
- payment or contract dispute.

Suspension should specify whether the actor may still:

- view prior work;
- submit pending evidence;
- respond to review questions;
- correct a security issue;
- communicate through the Delivery Owner.

## Revocation

Revocation ends future authority under the Assignment.

```text
Assignment revoked
≠ historical work erased
≠ external action reversed
```

The system must handle:

- access removal;
- active Tool sessions;
- pending drafts;
- already-sent communications;
- external actions;
- incomplete Contributions;
- compensation;
- reassignment and handover;
- customer and deadline impact.

## Reassignment

Reassignment is not a simple replacement of the assignee field.

A governed transition should preserve:

- original Assignment;
- reason for replacement;
- prior Contributions;
- access revocation;
- current package version;
- open Review findings;
- deadline impact;
- compensation status;
- new assignee acceptance.

```text
new assignee added
≠ old authority automatically ended
```

The old Assignment must be explicitly suspended, revoked, completed or closed.

## Completion and closure

Assignment closure should normally follow:

```text
Contribution submitted
→ Review or validation
→ disposition recorded
→ access returned or expired
→ compensation checkpoint
→ Assignment closed
```

An Assignment can close even when the Work Package is reassigned after rejection. Conversely, a Work Package can be accepted after several Assignments and Contributions.

## Evidence and observability

The Assignment record should make it possible to reconstruct:

- why the actor was selected;
- what they accepted;
- what data they could access;
- which Tools they could use;
- what they produced;
- which reviews applied;
- when authority changed;
- whether access was revoked;
- how compensation was resolved.

```text
actor appears in activity log
≠ Assignment reconstructable
```

## Candidate status and admission

Assignment is a strong F3 candidate shared semantic. It is used conceptually across Execution, Workplace, MarkReg, Lite and professional networks.

Before activation, the platform must still resolve overlap with:

- Membership;
- Workflow task assignment;
- Provider appointment;
- professional engagement;
- access-control grants;
- Production Authorization;
- employment and contractor records.

It also requires lifecycle fixtures, revocation tests and migration analysis.

## Constitutional rule

```text
Assignment converts task-specific eligibility
into bounded responsibility for one exact Work Package.

It grants only the declared production authority,
data access and actions.
It does not create professional qualification,
customer ownership, final approval or formal-state authority.
```
