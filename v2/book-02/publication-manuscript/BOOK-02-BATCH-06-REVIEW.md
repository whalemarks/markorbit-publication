## Main decisions

### 1. Work Package is a bounded production contract

```text
Work Package
≠ Matter
≠ Order
≠ Workflow
≠ Step
≠ Product Task
```

The proposed Work Package contract preserves:

- objective and purpose;
- exact input versions;
- required output;
- scope and exclusions;
- Facts, Candidates, assumptions and Unknowns;
- eligibility requirements;
- authority ceiling;
- data boundary;
- deadline and dependencies;
- review policy;
- acceptance criteria;
- Evidence Return contract;
- compensation reference;
- amendment and cancellation rules.

Work Package remains a candidate shared semantic and was not activated.

### 2. Assignment converts eligibility into bounded responsibility

```text
Capability
≠ Production Authorization
≠ Eligibility
≠ Assignment
```

Assignment binds one exact Work Package version to one actor or implementation with explicit:

- represented context;
- allowed and prohibited actions;
- data and Tool access;
- customer-contact rule;
- review and supervision;
- compensation;
- expiry, suspension and revocation.

```text
Assignment accepted
≠ work completed
≠ professional authority granted
≠ protected action approved
```

Assignment remains a candidate shared semantic and was not activated.

### 3. Contribution, Review and Outcome remain separate

```text
Contribution
≠ Review
≠ Outcome
```

Contribution preserves attributable production and provenance. Review preserves typed judgment over an exact object version. Outcome preserves an accepted result for a declared purpose.

```text
Contribution submitted
≠ Contribution accepted
≠ Work Package complete

reviewed
≠ approved
≠ applied

Outcome accepted
≠ customer approved
≠ external action performed
≠ formal state changed
```

Contribution and Outcome remain candidate shared semantics. Review remains the established governance mechanism.

### 4. Financial state cannot be reduced to `paid`

```text
Evidence
≠ Compensation
≠ Settlement
≠ Payment Custody
```

The batch distinguishes:

- compensation entitlement;
- payment request and approval;
- payment initiation and processing;
- funds debit and receipt;
- application of funds;
- refund and credit;
- commission and dispute;
- financial reconciliation and Settlement.

```text
Customer Paid
≠ Provider Paid
≠ Official Fee Paid
≠ Contributor Compensated
≠ Settlement Complete
```

### 5. Payment orchestration remains outside custody

```text
Payment Orchestration
≠ Payment Custody
```

The manuscript does not authorize escrow, pooled customer funds, money transmission, stored value or discretionary control of third-party funds.
