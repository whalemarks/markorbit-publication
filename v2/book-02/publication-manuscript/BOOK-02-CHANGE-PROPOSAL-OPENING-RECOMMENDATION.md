# Book 02 — Change Proposal Opening Recommendation

## 1. Decision purpose

Batch 12 was designed to determine whether the highest-risk v2 candidate semantics are mature enough to enter the frozen Book 02 Change Proposal process.

Reviewed candidate clusters:

```text
CP-A — Professional Authority and Production Control
CP-B — Governed Production Object Chain
```

This recommendation distinguishes:

- research complete enough to open a formal proposal;
- research that should remain profile or composition work;
- research that requires external legal, professional or consumer evidence;
- concepts that should remain outside Core.

## 2. Opening criteria

A formal Book 02 Change Proposal should open only when the candidate has:

1. a stable cross-Product problem;
2. insufficient F1 composition and F2 profile alternatives;
3. a clear identifier and purpose;
4. a distinct lifecycle;
5. an Owning Service;
6. mutation and authority rules;
7. compatibility and migration consequences;
8. positive and negative fixture requirements;
9. at least two independent consumers where applicable;
10. legal or professional review where external authority is represented.

A proposal should not open merely because a term is useful in publication prose.

## 3. CP-A decision

### 3.1 Capability Evidence

Decision:

```text
DO NOT OPEN A CHANGE PROPOSAL
```

Reason:

Frozen Evidence, Document, Human Review, Event, Versioning and Capability references can likely represent the required semantics through an Evidence profile.

Next action:

- define a noncanonical Capability Evidence profile fixture;
- test it in Capability assessment and AI/system evaluation use cases;
- keep all records purpose-limited.

### 3.2 Capability Certification

Decision:

```text
DO NOT OPEN A ROOT-OBJECT PROPOSAL YET
PREPARE AN F2/F3 PROFILE STUDY
```

Reason:

Certification requires scheme, assessor, evidence basis, issue/expiry and revocation information, but a credential profile over Evidence, Human Review, Identity, Versioning and Events may be sufficient.

Next action:

- create a profile research artifact;
- distinguish platform certification from external qualification;
- obtain at least two consumer use cases;
- define negative fixtures preventing legal-authority inference.

### 3.3 Professional Qualification

Decision:

```text
OPEN A NARROW PRE-PROPOSAL FOR AN EXTERNAL QUALIFICATION REFERENCE
NOT FOR PLATFORM-ISSUED QUALIFICATION
```

Reason:

The frozen baseline can identify actors and source records but does not define a complete, governed representation of externally issued professional credentials, their jurisdiction, status, restrictions and verification freshness.

The proposed semantic must remain reference-only:

```text
Core represents the qualification record.
Core does not issue, renew, suspend or revoke the external qualification.
```

Required before formal opening:

- legal and professional review;
- source-authority and verification model;
- privacy and disciplinary-data analysis;
- external-state versus platform-verification-state separation;
- jurisdiction-specific examples.

Recommendation status:

```text
PRE-PROPOSAL AUTHORIZED
FORMAL CHANGE PROPOSAL NOT YET AUTHORIZED
```

### 3.4 Production Authorization

Decision:

```text
DO NOT OPEN A NEW OBJECT PROPOSAL
TEST AS A SPECIALIZED PERMISSION PROFILE
```

Reason:

Frozen Permission already contains subject, action, resource, scope, effect, status, organization, Policy and Capability references. Frozen Policy can express review, approval, audit and contextual constraints.

A new Object becomes justified only if effective intervals, implementation-version binding, supervision, suspension consequences and active Assignment review cannot be safely represented through Permission and Policy.

Next action:

- define a noncanonical Permission profile;
- test human and AI/system subjects;
- include model or service version where applicable;
- prove revocation behavior against active Tasks and Assignments.

### 3.5 Eligibility

Decision:

```text
KEEP IN BOOK 03 EXECUTION
```

Reason:

Eligibility is a current, exact-work decision based on Permission, Policy, qualification, conflict, capacity, data and review context. It should be recalculated and time-limited rather than stored as a permanent Core actor status.

Core may supply references and decision envelopes; Book 03 owns runtime evaluation.

### 3.6 Professional Authority

Decision:

```text
PREPARE A NARROW F3/F4 PRE-PROPOSAL
```

Reason:

No single frozen contract represents the complete chain of externally grounded authority for a reserved action:

```text
current qualification
+ jurisdiction
+ conflict clearance
+ appointment
+ exact Matter scope
+ authority to advise / sign / submit / represent
```

The proposal must not create authority. It may only represent evidence and references to externally grounded authority.

Required before formal opening:

- professional and legal review;
- appointment and authority-source model;
- Matter and jurisdiction scope;
- verification and expiry rules;
- privileged/conflict-data boundary;
- relation to Permission, Policy, Human Review and Owning Service mutation.

Recommendation status:

```text
PRE-PROPOSAL AUTHORIZED
FORMAL CHANGE PROPOSAL NOT YET AUTHORIZED
```

## 4. CP-B decision

### 4.1 Work Package

Decision:

```text
DO NOT OPEN A ROOT-OBJECT PROPOSAL
TEST AS A TASK / WORKFLOW PROFILE
```

Reason:

Frozen Task already represents actionable work with assignee, responsibility, Workflow, Matter, dependencies, deadline and artifact references.

A distinct Work Package is justified only when one package:

- contains several Tasks;
- has package-level acceptance;
- survives Task replacement;
- supports several Contributors;
- crosses Workplace boundaries;
- owns package-level correction or commercial obligations.

Required proof:

```text
Book 03 Execution
+ one independent Product consumer
```

### 4.2 Assignment

Decision:

```text
SPLIT THE CONCEPT
```

Simple internal allocation:

```text
retain frozen Task assignee and responsibility fields
```

Accepted bilateral or cross-Workplace Assignment:

```text
prepare F3 pre-proposal research
```

Reason:

Task does not explicitly preserve offer, acceptance, decline, expiry, suspension, revocation or commercial-term acceptance.

Formal proposal opening remains blocked until consumer proof demonstrates that the lifecycle must survive independently of Task state.

### 4.3 Contribution

Decision:

```text
DO NOT OPEN A ROOT-OBJECT PROPOSAL
TEST AS A SUBMISSION PROFILE
```

Reason:

Document, Evidence, Event, Task and Human Review can represent most submission behavior. A stable Contribution identity is justified only when a submission groups several artifacts, supports correction/supersession and requires independent attribution and review continuity.

### 4.4 Review

Decision:

```text
NO CHANGE PROPOSAL
RETAIN FROZEN HUMAN REVIEW CONTRACT
```

Reason:

The frozen Human Review Contract already contains the canonical cross-domain review envelope and clearly separates review from downstream execution.

Domain-specific review must specialize or reference the frozen contract rather than replace it.

### 4.5 Outcome

Decision:

```text
TEST AS AN F2 ACCEPTANCE PROFILE
DO NOT OPEN A ROOT-OBJECT PROPOSAL YET
```

Reason:

Human Review decision, Evidence, Event and Owning Service results can compose an accepted result. A stable Outcome identity may become necessary for cross-Product continuity, correction and purpose-limited acceptance, but that need has not yet been demonstrated by independent consumers.

## 9. Formal proposal gate

No formal Change Proposal should be opened in Batch 12.

The batch has established that several concepts are important but has also shown that composition and profile alternatives remain viable.

Opening an omnibus proposal now would violate the frozen admission discipline because it would combine:

- external credentials;
- platform certification;
- Permission;
- Execution eligibility;
- Task assignment;
- professional authority;
- production artifacts and outcomes.

These concepts must remain separated.
