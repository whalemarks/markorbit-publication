# Book 02 — Chapter and Thesis Map

## Book thesis

Book 02 argues that MarkOrbit Core is a shared semantic constitution: it standardizes meaning, authority, lifecycle, evidence, versioning and compatibility without becoming the universal owner of every Product, record, workflow, source, Claim or formal state.

## Part I — Constitutional semantics

| Chapter | Main thesis |
|---|---|
| CH01 | Core is shared meaning, not centralized ownership. |
| CH02 | Business, semantic, formal-state, custody and professional authority must remain separated. |
| CH03 | Observation and Candidate must precede accepted or canonical state. |
| CH04 | A concept belongs in Core only where stable interoperability truly requires it. |

Constitutional lock:

```text
Core Meaning ≠ Record Ownership
Candidate ≠ Canonical
Important Concept ≠ Core Object
```

## Part II — Identity, relationships and references

| Chapter | Main thesis |
|---|---|
| CH05 | Person, Organization, Workplace and Membership represent different identity and authority relationships. |
| CH06 | Customer, Contact and Relationship semantics must preserve actual roles and relationship ownership. |
| CH07 | Product Installation and Entitlement remain Workplace-scoped and do not create business ownership. |
| CH08 | References preserve identity continuity while resolution remains Candidate-before-Canonical and reversible. |

Constitutional lock:

```text
Membership ≠ Assignment
Customer ≠ Applicant ≠ Payer
Reference ≠ Mutation Authority
Probable Match ≠ Verified Identity
```

## Part III — Product, work and formal-state semantics

| Chapter | Main thesis |
|---|---|
| CH09 | Product, Capability, Workflow, Tool, Service and Implementation must remain distinct. |
| CH10 | Order, Matter, Work Package and Assignment represent different commercial and production concerns. |
| CH11 | Evidence supports Claims; authorized actors make Decisions; Owning Services establish Formal State. |
| CH12 | Projection, Handoff, Return and Correction govern cross-context movement without implied transfer. |

Constitutional lock:

```text
Capability ≠ Implementation
Order ≠ Matter
Evidence ≠ Decision ≠ Formal State
Return ≠ Accepted Truth
```

| Chapter | Main thesis |
|---|---|

| CH18 | Opportunity is a potential value route, not a customer instruction or Order. |

```text
Need ≠ Opportunity
Opportunity ≠ Customer Instruction

```

| Chapter | Main thesis |
|---|---|

| CH23 | Contribution, Review and Outcome must remain separately attributable and version-bound. |
| CH24 | Compensation, Payment events and Settlement require distinct evidence and do not imply custody. |

Constitutional lock:

```text
Work Package ≠ Assignment

```

| Chapter | Main thesis |

| CH31 | Contributor, Provider Organization and responsible professional carry different responsibility. |

Constitutional lock:

```text
Projection ≠ Ownership Transfer
Handoff ≠ Assignment
Contributor ≠ Provider ≠ Professional Authority

```

## Part IX — Versioning, compatibility and constitutional durability

| Chapter | Main thesis |
|---|---|
| CH33 | Every public contract and shared record must expose the versions necessary to reconstruct meaning. |
| CH34 | Compatibility, deprecation and migration require semantic and evidentiary proof, not only successful parsing. |
| CH35 | F0–F4 governs admission of proposed semantics into Core. |
| CH36 | Core must remain smaller than the platform over a ten-year horizon. |

Constitutional lock:

```text
Schema Compatible ≠ Semantically Compatible
Migration Completed ≠ Migration Accepted
Publication Prose ≠ Semantic Activation
Most-used Product ≠ Semantic Authority
```

## Complete semantic path

```text
Source or Observation
→ Structured Record or Claim Candidate
→ Review / Validation
→ Accepted Knowledge, Data or Business Record
→ Need and Opportunity
→ Engagement or Order
→ Capability Request
→ Work Package
→ Assignment
→ Contribution
→ Review and Outcome
→ Approval and Apply
→ Handoff / Return
→ Owning Service Validation
→ Formal State
→ Versioned Correction and Learning
```

The path is explanatory, not a single universal Workflow. Each transition retains its own authority, evidence and ownership.
