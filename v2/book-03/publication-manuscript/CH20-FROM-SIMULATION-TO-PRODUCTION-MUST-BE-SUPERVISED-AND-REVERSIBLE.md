# CH20 — From Simulation to Production Must Be Supervised and Reversible

## 1. Passing simulation does not grant production authority

Simulation can produce valuable Capability Evidence, but real production adds new conditions:

- real customers;
- confidential data;
- binding deadlines;
- professional obligations;
- actual fees;
- external providers;
- official systems;
- relationship consequences.

The governing progression is:

```text
Simulation
→ Shadow Work
→ Supervised Production
→ Bounded Independent Production
→ Complex Production
→ Review Authority
```

Every arrow is a separate governance decision.

```text
Assessment Passed
≠ Production Authorization
≠ Task-specific Eligibility
≠ Professional Authority
```

## 2. Capability Evidence, Certification and Authorization remain distinct

The system should preserve:

```text
Capability Evidence
→ Certification Decision
→ Production Authorization
→ Task-specific Eligibility
→ Assignment
```

Capability Evidence records observed performance. Certification confirms that defined criteria were met. Production Authorization allows a bounded category of real work. Eligibility determines whether the person or system may perform the current task now.

None of these creates jurisdiction-specific legal qualification.

## 3. Production onboarding adds non-assessment gates

Before real work, the organization may require:

- identity verification;
- contractual relationship;
- confidentiality terms;
- security training;
- conflict process;
- data-protection commitments;
- payment and tax setup;
- tool access;
- professional-qualification evidence;
- represented-role clarity;
- customer-contact rules.

A candidate can be technically capable and still not be ready for production onboarding.

## 4. Shadow Work preserves observation without outcome control

In Shadow Work, the participant may:

- inspect an authorized production case;
- prepare a parallel analysis;
- compare with the live performer;
- draft a non-operative Contribution;
- receive review feedback.

But the shadow result does not control the customer Outcome.

```text
Shadow Contribution
≠ Live Contribution by Default
```

If a shadow artifact is useful, it must enter production through a separate review and acceptance path.

## 5. Shadow access must still be minimal

Shadow status does not justify unrestricted production visibility.

Access should be limited by:

- purpose;
- customer authorization;
- data minimization;
- redaction;
- Matter scope;
- time;
- download restrictions;
- confidentiality.

Synthetic or de-identified cases should be preferred when real data is unnecessary.

## 6. Supervised Production requires actual supervision

A named supervisor is not enough.

Supervised Production should define:

- Supervisor;
- Delivery Owner;
- Reviewer;
- Work Package types;
- allowed actions;
- prohibited actions;
- review depth;
- customer-contact mode;
- data scope;
- deadline buffer;
- compensation;
- escalation route;
- stop conditions.

```text
Supervisor Assigned
≠ Supervision Available
```

The supervisor must have capacity to intervene before consequential action.

## 7. Live work should start with bounded, reversible outcomes

Early production assignments should favor work that is:

- clearly scoped;
- well supported by templates;
- reviewable;
- reversible;
- low in external authority;
- rich in feedback.

Examples include:

- source verification;
- structured data preparation;
- document completeness checks;
- draft communication;
- non-final package preparation;
- evidence organization.

Early assignments should generally exclude unsupervised official filing, binding legal communication, payment movement and final professional approval.

## 8. Production Authorization should be typed

Possible authorization states include:

```text
Training Only
Shadow Work Permitted
Supervised Production
Independent Standard Production
Advanced Production
Review Authority
Suspended
Expired
Revoked
```

Authorization should specify:

- Capability;
- jurisdiction;
- Work Package type;
- M-mode;
- R-tier limit;
- data class;
- customer-contact rule;
- required reviewer;
- effective period;
- restrictions.

## 9. Independent production remains bounded

```text
Independent Production
≠ Independent Professional Approval
```

A performer may independently prepare a standardized package while final Review, Approval or External Protected Action remains with another authority.

The system should avoid the misleading label `fully authorized` without scope.

## 10. Progression depends on evidence, not volume alone

Promotion should consider:

- recent quality;
- defect severity;
- correction quality;
- escalation judgment;
- deadline reliability;
- source use;
- customer communication;
- AI governance;
- reviewer confidence;
- diversity of cases;
- independence level.

Completing many easy tasks does not prove readiness for complex work.

## 11. Review burden is part of readiness

A performer whose work is usually correct but requires extensive reviewer reconstruction may not yet be ready for independent production.

Useful measures include:

- review time;
- number of material changes;
- repeated defect types;
- missed Unknowns;
- reviewer intervention before Apply;
- downstream correction rate.

These measures require context and should not become simplistic productivity rankings.

## 12. Progression must be reversible

Authorization may be reduced when:

- serious quality failure occurs;
- errors repeat;
- the person has been inactive;
- rules change;
- a security incident occurs;
- scope is exceeded;
- escalation is avoided;
- professional qualification expires;
- Reviewer confidence falls.

Possible actions include:

```text
Additional Review
Scope Restriction
Refresher Learning
Reassessment
Return to Supervised Production
Temporary Suspension
Revocation
```

```text
Previously Authorized
≠ Permanently Authorized
```

## 13. Reversal should preserve fairness and evidence

A downgrade should identify:

- triggering evidence;
- affected scope;
- effective time;
- immediate access changes;
- active assignments;
- appeal or review route;
- remediation plan;
- reinstatement criteria.

One disputed result should not automatically destroy all Capability standing.

## 14. Active work requires a transition plan

If authorization is suspended while work is active, Execution must:

- stop prohibited future actions;
- identify active Assignments;
- preserve Contributions;
- revoke access;
- reassign Work Packages;
- review work already produced;
- assess customer and deadline risk;
- prevent duplicate execution.

```text
Authorization Suspended
≠ Existing External Effects Reversed
```

## 15. Compensation begins when real production value begins

```text
Learning Value
≠ Substitute for Compensation
```

Simulation and assessment may be unpaid where they are genuinely evaluative and do not produce customer value. Shadow or supervised work that contributes to real delivery should have clear compensation terms.

The platform should not create a permanent unpaid trainee layer.

## 16. Customer and relationship boundaries continue to apply

Production progression does not grant:

- customer ownership;
- direct marketing rights;
- authority to bypass the Relationship Owner;
- permission to retain contact data;
- right to sell additional services.

```text
Production Participation
≠ Customer Relationship
```

## 17. AI systems also progress from evaluation to production

An AI model or agent should move through a comparable path:

```text
Offline Evaluation
→ Simulation
→ Shadow Recommendation
→ Supervised Production
→ Bounded M4 Execution
```

Each stage should specify:

- model version;
- Capability;
- source configuration;
- tool set;
- data scope;
- Review tier;
- stop conditions;
- drift monitoring;
- rollback.

```text
Model Benchmark Passed
≠ Production Authorization
```

## 18. Model or policy change can reset progression

A material change to:

- model version;
- prompt policy;
- retrieval source;
- tool permissions;
- autonomy;
- memory;
- target jurisdiction;

may require new simulation and supervised evidence.

## 19. Production evidence should update capability carefully

Real outcomes can strengthen or narrow Capability Evidence, but formal-state success alone is not sufficient.

```text
Favorable Official Outcome
≠ Perfect Execution
```

The system should examine process quality, evidence, review burden, authority compliance and recovery.

## 20. The governing principle

The path into production should be earned, bounded and reversible.

```text
Observed Capability
+ Current Authorization
+ Task-specific Eligibility
+ Available Supervision
= Safe Production Participation
```

MarkOrbit should make it possible for people and AI systems to grow into greater responsibility without treating one assessment, one certificate or one successful task as permanent authority.
