# Book 03 v2 Publication Reconstruction Charter

## 1. Mandate

Reconstruct Book 03 into a coherent publication manuscript that explains how MarkOrbit turns capability into governed human–AI work.

The manuscript must remain useful to product managers, system architects, professional-service operators, reviewers, contributors and investors without pretending to be implementation code or legal advice.

## 2. Source order

The reconstruction uses the following source hierarchy:

1. active Canon and accepted cross-book semantic boundaries;
2. frozen `B03-RC1-FROZEN-01` as a controlled publication baseline;
3. `BOOK-03-v2-MASTER.md` as the current strategic architecture;
4. the seven-part, 120-short-chapter v2 working manuscript;
5. accepted Book 01, 04, 05, 06 and 07 v2 boundaries where they clarify Workplace, Product, MarkReg, Lite, Capability and MGSN;
6. implementation repositories only as evidence of current engineering state, never as silent publication authority.

When sources conflict, the conflict is recorded rather than averaged away.

## 3. Central thesis

```text
Capability becomes trustworthy work only when
context, authority, inputs, implementation binding,
review, approval, failure, evidence and recovery
remain explicit throughout execution.
```

Execution is the platform trust engine. It coordinates work but does not absorb business sovereignty or formal-state authority.

The manuscript must preserve at least these distinctions:

```text
Need ≠ Capability Request

```

## 5. Human–AI authority rule

AI may retrieve, explain, compare, draft, simulate, recommend and—in explicitly governed modes—execute bounded actions. AI does not acquire authority merely because it is technically capable.

Every consequential AI-assisted path must answer:

- what model or implementation acted;
- which sources and versions were used;
- what uncertainty remained;
- which human or policy selected the collaboration mode;
- what review was required;
- who approved the exact version;
- which action boundary was crossed;
- what evidence and recovery path remain.

## 6. Protected-action rule

An External Protected Action is not simply a high-risk button. It is an action whose consequences extend outside the controlled preparation environment, including official filing, binding external communication, provider appointment, movement or custody of funds, disclosure of protected information or other formal-state mutation.

Protected actions require specific authority, exact-version approval, idempotent execution and evidence-backed reconciliation.

## 7. Publication limits

This reconstruction does not:

- replace the frozen Book 03 baseline;
- change active Canon;
- authorize production deployment;
- authorize autonomous AI professional practice;
- authorize payment custody;
- declare every proposed object a Core object;
- convert examples into universal jurisdictional rules.

## 8. Editorial method

Short working chapters are consolidated into substantial chapters with:

- a clear thesis;
- a complete operating chain;
- explicit semantic boundaries;
- failure and recovery treatment;
- human–AI implications;
- product and implementation consequences;
- practical examples without fabricated legal facts.

Repetition is retained only where it protects a different boundary. Final editing will later reduce formulaic restatement.
