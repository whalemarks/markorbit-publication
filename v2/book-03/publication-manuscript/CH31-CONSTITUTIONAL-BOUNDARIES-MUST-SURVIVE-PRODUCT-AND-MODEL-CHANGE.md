# CH31 — Constitutional Boundaries Must Survive Product and Model Change

## 1. Execution architecture must outlive the current interface and model

Products, models, connectors and operating structures will change. The constitutional boundaries of governed execution must remain intelligible across those changes.

```text
Product Version Changes
Model Version Changes
Workflow Version Changes
Provider Changes
≠ Authority Boundaries Disappear
```

The Book 03 constitution exists to ensure that a faster interface or more capable model cannot silently collapse distinctions that protect customers, professionals, Workplaces and formal truth.

## 2. Constitutional boundaries are semantic, not cosmetic

The following distinctions must survive implementation changes:

```text
Guide ≠ Workflow
Capability Request ≠ Assignment
Capability ≠ Professional Authority
Contribution ≠ Outcome
Review ≠ Approval
Approval ≠ Apply
Apply ≠ External Effect
Provider Return ≠ Official Truth
Execution State ≠ Formal State
Payment Orchestration ≠ Payment Custody
```

A new Product may rename screens and controls, but it must not erase the underlying transitions.

## 3. Product experience cannot create authority

A Product may simplify:

- service discovery;
- data collection;
- option comparison;
- package preparation;
- Review routing;
- Preview;
- customer communication.

It cannot create:

- customer instruction;
- professional qualification;
- Provider appointment;
- payment authority;
- formal-state ownership;
- external-action authority.

```text
Better UX
≠ Broader Authority
```

## 4. Product Installations preserve Workplace sovereignty

The same Product can be installed in multiple Workplaces with different:

- customer relationships;
- contracts;
- personnel;
- Provider networks;
- professional authority;
- pricing;
- data policy;
- review policy;
- operating model.

Execution must preserve the represented Workplace in every consequential event.

```text
Shared Product
≠ Shared Customer Ownership
≠ Shared Data Authority
```

## 5. White-label and co-delivery must not erase responsibility

A unified customer experience may involve:

- Relationship Owner;
- Contracting Party;
- Delivery Owner;
- Contributor;
- Reviewer;
- Professional Authority;
- Provider Organization;
- Owning Service.

These roles may be distributed across organizations.

```text
One Brand Experience
≠ One Hidden Actor
≠ One Universal Authority
```

Required professional, contractual or privacy disclosures must remain visible where applicable.

## 6. Execution coordinates but does not own formal truth

Execution may coordinate:

- requests;
- Work Packages;
- Assignments;
- Contributions;
- Review;
- Approval;
- Apply attempts;
- Returns;
- Reconciliation.

Owning Services retain authority over formal objects such as:

- Customer;
- Order;
- Matter;
- Payment;
- trademark right;
- official status;
- credential;
- Knowledge Object.

```text
Execution Evidence
→ Owning Service Validation
→ Formal-state Mutation
```

## 7. Core semantics cannot be activated by manuscript convenience

The publication manuscript may propose useful objects and relationships. It does not automatically make them Active Canon or implementation requirements.

```text
Publication Concept
≠ Activated Core Object
```

Promotion requires explicit cross-book and Core governance.

## 8. Model capability cannot expand invocation authority

A more capable model may:

- understand more context;
- produce better drafts;
- use more tools;
- manage longer workflows;
- detect more conflicts.

It still cannot exceed:

- Capability scope;
- M-mode;
- R-tier;
- data scope;
- tool policy;
- Approval;
- Professional Authority;
- Apply boundary.

```text
Model Can
≠ Model May
```

## 9. General agents remain bounded implementations

An agent capable of planning multiple steps is still an Implementation bound to a governed request.

It must not invent:

- new commercial scope;
- new customer instruction;
- new Provider appointment;
- new data purpose;
- new payment authority;
- new Professional Authority.

```text
Agent Plans Steps
≠ Agent Owns the Goal
```

## 10. Tool access must remain least-privileged

A model or service account should receive only the tools and permissions required for the current invocation.

```text
Can Read
≠ Can Edit
≠ Can Send
≠ Can Approve
≠ Can Apply
```

Long-lived universal credentials undermine the Assignment and Apply boundaries established in this book.

## 11. Model and policy updates require controlled migration

When models, prompts or tools change, the system should determine whether:

- earlier evaluation remains applicable;
- Simulation is required;
- shadow production is required;
- R-tier should increase temporarily;
- M4 authorization should be suspended;
- active workflows need revalidation.

```text
Operational Upgrade
≠ Governance Carryover Automatically Valid
```

## 12. Workflow templates are versioned policy artifacts

A template change may alter:

- required inputs;
- Work Packages;
- Review depth;
- Approval sequence;
- Provider route;
- Evidence Contract;
- recovery path.

Active instances must retain their original version or undergo explicit migration.

```text
Template v2 Released
≠ Active Instance Silently Rewritten
```

## 13. Source updates cannot silently alter completed decisions

A current official fee or rule may change after an earlier decision.

The system should distinguish:

```text
Correct Under Source v1 at Time T1
Current Rule Under Source v2 at Time T2
```

Historical decisions should remain understandable under their original context, while active and future work receives revalidation as required.

## 14. Professional Authority remains externally grounded

No internal change can create a jurisdictional professional qualification.

```text
Platform Role
Certification
Production Authorization
High Quality Score
AI Assistance
≠ Professional Qualification
```

Qualification, appointment, conflict and current status must remain externally or professionally grounded.

## 15. Commercial success cannot weaken governance

High conversion, large volume or strong margins must not justify:

- lower Review without evidence;
- hidden Provider relationships;
- unsupported claims;
- customer-data reuse;
- automatic payment;
- unsafe retry;
- reduced professional disclosure.

```text
Commercial Success
≠ Constitutional Exception
```

## 16. Urgency cannot become permanent bypass

Emergency procedures may permit bounded exceptions, but they require:

- defined trigger;
- authority;
- scope;
- expiry;
- evidence;
- post-action Review;
- customer notification where appropriate.

```text
Emergency Path
≠ New Default Path
```

## 17. Simulation remains isolated as systems evolve

New connectors and agents must be tested against the Simulation constitution:

- no live official submission;
- no real payment;
- no real customer communication;
- no Production formal-state mutation;
- no production credentials by default.

```text
New Sandbox Feature
≠ Production Isolation Proven
```

## 18. Learning cannot self-amend the constitution

Execution learning may recommend changes, but it cannot automatically remove core boundaries because they create friction.

```text
Frequent Approval
≠ Approval Is Unnecessary

Frequent Reconciliation
≠ Unknown Should Be Hidden
```

Constitutional changes require explicit governance and cross-book consistency review.

## 19. Data rights survive organizational and product change

A merger, Provider replacement, Product migration or model update does not automatically expand data rights.

The system must preserve:

- purpose limitation;
- customer consent;
- confidentiality;
- retention;
- deletion;
- training exclusion;
- relationship ownership.

```text
Data Migrated
≠ Purpose Expanded
```

## 20. Financial boundaries survive payment integration

A Product may integrate invoicing, payment links and reconciliation. That does not automatically authorize:

- custody;
- pooled funds;
- escrow;
- money transmission;
- unilateral refunds;
- netting across Workplaces.

```text
Integrated Payment Experience
≠ Payment Custody
```

## 21. Observability must survive component replacement

Replacing an AI provider, Connector or Workflow engine must not break the audit chain.

Stable semantic identifiers should preserve:

- request;
- package;
- Review;
- Approval;
- Apply;
- external reference;
- formal-state validation.

The infrastructure may change while the execution history remains reconstructable.

## 22. Constitutional tests should be executable

The system should test invariants such as:

- no Apply without current Approval where required;
- no Production action from Simulation;
- no R4 action without Professional Authority;
- no retry while external state is Unknown unless specifically approved;
- no formal-state mutation from unvalidated Provider Return;
- no training use from operational access alone;
- no payment movement from recommendation alone.

```text
Constitution Written
≠ Constitution Enforced
```

## 23. Exceptions require records

Where an invariant permits exception, the record should identify:

- exception type;
- authority;
- justification;
- affected scope;
- start and expiry;
- residual risk;
- post-action obligation.

Undocumented bypass is not an exception. It is a governance failure.

## 24. The constitutional promise

MarkOrbit may become more automated, more distributed and more intelligent. Its execution constitution should preserve one promise:

```text
No consequential transition becomes legitimate merely because it became technically easy.
```

Authority, evidence, version, responsibility and formal truth must survive every Product, model and organizational change.
