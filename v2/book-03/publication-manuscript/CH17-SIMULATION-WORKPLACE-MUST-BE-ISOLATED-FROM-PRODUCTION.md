# CH17 — Simulation Workplace Must Be Isolated from Production

## 1. Learning must not accidentally become real-world execution

A professional platform cannot train people, test AI or assess workflows safely if simulation and production share the same customers, credentials, providers, payment routes and official connectors.

The core rule is:

```text
Simulation
≠ Production
```

A Simulation Workplace is not merely a production account with a banner saying “training.” It is an isolated execution environment in which the system can reproduce professional complexity without creating real legal, commercial, financial or privacy consequences.

The minimum boundary is:

```text
Synthetic or Controlled Case
→ Simulation-only Objects
→ Simulation-only Workflow
→ Simulation-only Review and Approval
→ Simulated Outcome
→ Assessment Evidence

No Live Customer
No Live Provider Instruction
No Official Submission
No Real Payment
No Production Mutation
```

## 2. Simulation is a governed Workplace mode

Simulation should preserve the architecture of a real Workplace while replacing or isolating consequential resources.

It may contain:

- synthetic Customers;
- synthetic Applicants and Owners;
- fictional Trademarks;
- controlled Documents;
- staged Communications;
- simulated Deadlines;
- mock Provider Returns;
- controlled official-status snapshots;
- simulated fees and payments;
- assessment-specific roles;
- simulation-only Capability Requests, Assignments, Contributions, Reviews and Outcomes.

The point is not to simplify the environment until it becomes a quiz. The point is to preserve realistic decision structure while removing real-world consequence.

## 3. Production objects must not be silently copied into simulation

Real cases may sometimes inform training, but operational records do not automatically become simulation content.

```text
Production Matter
≠ Simulation Fixture
≠ Training Permission
```

Before a real case can become a simulation fixture, the system must address:

- customer and contractual rights;
- confidentiality;
- personal data;
- professional privilege where applicable;
- provider communications;
- document ownership;
- re-identification risk;
- whether synthetic reconstruction is safer;
- whether AI-training use is separately permitted.

Where risk remains, the correct route is a synthetic or heavily transformed scenario.

## 4. Simulation identities and credentials must be separate

Production credentials must never be available by default in a Simulation Workplace.

The environment should separate:

- user sessions;
- service accounts;
- API keys;
- provider credentials;
- official filing credentials;
- payment credentials;
- email identities;
- customer-contact identities;
- storage locations;
- event streams.

```text
Simulation Access
≠ Production Access
```

A person who is allowed to practise filing in simulation does not thereby receive permission to file a real application.

## 5. External connectors should be blocked, stubbed or redirected

Simulation may need to imitate an external system. It should do so through:

- deterministic stubs;
- sandbox APIs;
- replayed responses;
- synthetic provider actors;
- examiner-role simulation;
- explicit mock outputs.

It should not call a production connector unless the connector has a proven no-side-effect simulation mode.

```text
Simulation Connector Success
≠ Official Submission
```

If an external system has no safe sandbox, the simulation should stop before the external action and generate a simulated Return.

## 6. Communication must not reach real recipients

Simulated customer and provider communications should be delivered only to controlled addresses, in-platform role actors or sandbox mailboxes.

The system should block:

- real customer email addresses;
- real provider recipients;
- production messaging channels;
- real SMS or telephone notifications;
- automatic marketing routes;
- production webhook targets.

```text
Simulation Message Sent
≠ Real Message Sent
```

A visible simulation label is useful, but technical recipient controls are essential.

## 7. Simulated funds must never be real funds

Simulation can model:

- official fees;
- provider prepayments;
- customer payments;
- exchange-rate changes;
- refunds;
- insufficient funds;
- reconciliation failures.

But these must use simulation balances and synthetic transaction references.

```text
Simulation Payment Approved
≠ Funds Moved
```

No training exercise should create a real invoice, charge a payment instrument or instruct a provider to spend money.

## 8. Time may be controlled, but deadlines remain explicit

Simulation often needs accelerated time. A months-long procedure may be compressed into hours.

The system should distinguish:

```text
Real Clock
Simulation Clock
Scenario Event Time
Assessment Time Limit
```

A simulated deadline should remain traceable to the scenario rules. Time compression must not contaminate production deadline logic.

## 9. Scenario facts can be staged

A realistic simulation should not reveal all facts at the beginning.

It may introduce:

- a missing original POA;
- an applicant-name inconsistency;
- a fee increase;
- a provider delay;
- a customer objection;
- a conflicting source;
- a late document;
- an ambiguous external response;
- a potential duplicate submission;
- a hidden professional-authority requirement.

These events test whether the participant can revise the plan, preserve Unknown and escalate appropriately.

## 10. Hidden facts must not become arbitrary traps

A simulation should be difficult because professional work is difficult, not because the examiner invents unknowable rules after the fact.

Every hidden fact should have:

- a scenario purpose;
- a release condition;
- an evidence source;
- an expected effect on decisions;
- an assessment mapping;
- a fairness review.

```text
Surprise
≠ Arbitrary Scoring
```

## 11. Simulation modes should be explicit

Useful modes include:

```text
Practice Mode
Guided Practice Mode
Shadow Mode
Formal Assessment Mode
Calibration Mode
Incident Drill Mode
Tool Evaluation Mode
AI Evaluation Mode
```

These modes differ in:

- whether guidance is visible;
- whether answers can be retried;
- whether hidden facts are revealed;
- whether scoring applies;
- whether the case contributes Capability Evidence;
- whether human or AI assistance is allowed.

## 12. M0–M5 rules still apply in simulation

A simulation should specify the permitted Human–AI collaboration mode.

Examples:

- M0 assessment of unaided professional judgment;
- M2 practice with AI extraction assistance;
- M3 assessment of AI-draft review quality;
- M4 evaluation of bounded agent execution;
- M5 testing of hybrid professional coordination.

```text
Simulation
≠ Governance-free AI Playground
```

The participant should be assessed partly on whether AI was used within the allowed mode.

## 13. Simulation Approval is not production Approval

Approval records created in simulation must be tagged and isolated.

```text
Simulation Approval
≠ Production Approval
```

The same is true for:

- Certification Candidate;
- Provider Appointment;
- funds approval;
- customer instruction;
- Professional Authority;
- Apply result.

Nothing in the simulation should be reusable as a live authorization token.

## 14. Simulation outcomes should preserve full execution evidence

A simulation should produce the same structural evidence expected in production:

- Capability Request;
- validation result;
- Eligibility decision;
- Implementation Binding;
- Assignment;
- Contribution;
- Review;
- Approval;
- Preview;
- simulated Apply;
- Return;
- Outcome;
- incident and recovery events.

This allows the assessment to observe the whole path rather than only the final answer.

## 15. Assessment data is not automatically production data

Simulation performance may generate Capability Evidence, but it does not directly create Production Authorization.

```text
Assessment Passed
≠ Production Access Granted
```

The organization may still require:

- identity verification;
- contractual onboarding;
- confidentiality agreement;
- security review;
- conflict process;
- supervised production;
- current professional qualification;
- task-specific Eligibility.

## 16. Simulation fixtures need version control

A Case Fixture should identify:

- fixture version;
- scenario objectives;
- source basis;
- synthetic transformations;
- roles;
- hidden facts;
- release rules;
- expected decision points;
- scoring model;
- accepted alternative routes;
- known ambiguities;
- retirement date.

```text
Fixture v1 Result
≠ Fixture v2 Result without Recalibration
```

Changes to facts, rules or scoring may invalidate historical comparisons.

## 17. Simulation must support safe failure

Participants should be able to make mistakes without affecting real customers, but the system should still model consequences.

A simulated duplicate filing may trigger:

- incident record;
- reconciliation requirement;
- customer-notification draft;
- recovery plan;
- quality deduction.

The absence of real harm does not mean the simulation should ignore professional consequence.

## 18. Production imports require an explicit gate

A useful artifact produced in simulation should not be copied into a live Matter by default.

```text
Simulation Artifact
→ Review for Production Relevance
→ Source and Fact Revalidation
→ Production Request
→ New Review and Approval
```

Simulation data may be outdated, synthetic or intentionally misleading.

## 19. Observability must prove isolation

The platform should be able to demonstrate:

- no production credentials were used;
- no real recipient was contacted;
- no real funds moved;
- no official connector ran;
- no production formal object changed;
- all outputs remained in simulation storage;
- all events were tagged with Simulation Workplace identity.

Isolation should be verifiable, not assumed.

## 20. The governing principle

Simulation is valuable only when it is realistic enough to test judgment and isolated enough to make failure safe.

```text
Realistic Professional Complexity
+ Technical and Authority Isolation
= Trustworthy Simulation
```

A Simulation Workplace allows MarkOrbit to train people, evaluate AI, rehearse incidents and improve workflows without experimenting on real customers.
