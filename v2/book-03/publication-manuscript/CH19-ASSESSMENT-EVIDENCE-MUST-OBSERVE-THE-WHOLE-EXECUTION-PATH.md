# CH19 — Assessment Evidence Must Observe the Whole Execution Path

## 1. The final answer is only one piece of professional evidence

A correct final answer can result from guessing, unauthorized help or a flawed process that happened to end well. A wrong final answer can arise even when the participant handled facts, authority and escalation responsibly but encountered missing evidence or an intentionally adverse scenario.

Assessment must therefore observe the complete execution path.

```text
Assessment Evidence
≠ Final Answer Alone
```

The relevant chain is:

```text
Request Understanding
→ Fact Classification
→ Source Use
→ Risk Recognition
→ Route Selection
→ Work Production
→ Review and Communication
→ Approval and Apply Boundaries
→ Recovery
→ Outcome
```

## 2. Assessment Evidence is a governed evidence set

The record may include:

- Capability Requests created;
- questions asked;
- facts confirmed;
- Unknowns preserved;
- sources consulted;
- tool and AI use;
- Work Packages defined;
- Assignments and Contributions;
- Review requests;
- changes after findings;
- customer or provider communications;
- approvals requested;
- Apply attempts;
- deadline handling;
- incident and recovery actions;
- final Outcome;
- reflective explanation.

It should preserve versions and timestamps so later reviewers can reconstruct what the participant knew at each decision point.

## 3. Evidence dimensions should remain separate

A useful assessment may evaluate:

```text
Facts
Rules and Sources
Risk
Communication
Service Design
MarkOrbit Operation
Human–AI Judgment
Delivery Quality
Recovery
```

These dimensions should not be hidden behind one total score.

A participant may be strong in professional analysis but weak in customer communication. Another may operate the platform correctly but fail to identify a jurisdiction-specific authority issue.

## 4. Fact handling should be assessed explicitly

The evidence should show whether the participant:

- distinguished official facts from customer claims;
- labeled AI inference;
- identified conflicting records;
- requested missing evidence;
- avoided inventing certainty;
- updated conclusions when facts changed.

```text
Plausible Fact
≠ Verified Fact
```

## 5. Source use should be assessed for authority and freshness

Assessment should consider:

- source type;
- relevance;
- jurisdiction;
- date;
- conflict handling;
- citation quality;
- whether provider practice was mistaken for official rule;
- whether outdated sources were used after a newer source appeared.

A large number of citations does not compensate for weak source authority.

## 6. Risk evidence should capture what the participant noticed and did

Risk assessment includes:

- consequence severity;
- reversibility;
- deadline exposure;
- data sensitivity;
- financial impact;
- professional qualification;
- external uncertainty;
- duplicate-action risk;
- relationship risk.

The participant should not be rewarded merely for listing risks. The evidence should show whether risk changed the route, Review tier, Approval need or Apply decision.

## 7. Communication evidence must evaluate purpose and effect

Assessment should review whether communications:

- represented the correct Workplace and role;
- separated facts, assumptions and recommendations;
- explained options and exclusions;
- asked for a specific decision;
- stated deadlines accurately;
- avoided false urgency;
- preserved customer relationship boundaries;
- recorded the response correctly.

```text
Message Sent
≠ Communication Successful
```

## 8. Platform-operation evidence matters

Professional competence in MarkOrbit includes using the architecture correctly.

Assessment may observe whether the participant kept separate:

- Guide and Workflow;
- Capability Request and Assignment;
- Contribution and Outcome;
- Review and Approval;
- Preview and Apply;
- Provider Return and official truth;
- Execution state and formal state.

Incorrect object use can create systemic risk even when the written analysis is strong.

## 9. Human–AI evidence should evaluate judgment, not AI avoidance

The assessment should not assume that less AI use is always better.

It should evaluate:

- whether the selected M-mode was allowed;
- whether AI was useful for the task;
- whether sources and provenance were preserved;
- whether confidential data was controlled;
- whether human review was meaningful;
- whether the participant recognized uncertainty;
- whether AI autonomy was reduced when conditions changed.

```text
Used AI
≠ Weak Performance

Accepted AI Output Uncritically
= Governance Failure
```

## 10. Tool evidence must distinguish system failure from user failure

When a connector, model or script fails, the assessment record should identify:

- tool behavior;
- participant expectation;
- error returned;
- retry decision;
- reconciliation attempt;
- fallback route.

A participant should not be penalized for a controlled tool failure if they respond correctly. Conversely, a lucky successful result should not hide unsafe repeated attempts.

## 11. Review behavior should be observed

Assessment should examine whether the participant:

- requested the right Review type;
- selected an eligible reviewer;
- provided the correct version;
- responded to findings;
- distinguished Review from Approval;
- escalated disagreement;
- avoided ceremonial review.

## 12. Stopping and escalation are positive evidence

The participant may demonstrate competence by refusing to proceed when:

- applicant identity is unresolved;
- required professional authority is unavailable;
- source conflict affects a deadline;
- data disclosure is unauthorized;
- Apply state is Unknown;
- customer instruction is ambiguous.

```text
Blocked Unsafe Action
= Positive Capability Evidence
```

## 13. Recovery should be scored independently

A participant can make an initial error and still demonstrate valuable recovery competence.

Recovery evidence may include:

- detection speed;
- containment;
- preservation of evidence;
- customer communication;
- escalation;
- correction route;
- duplicate prevention;
- learning record.

The initial error remains visible, but recovery should not be ignored.

## 14. Outcome quality should include limitations

A high-quality Outcome may be:

- approved package;
- decision-ready options;
- verified status;
- recovery plan;
- no-safe-route conclusion.

The assessment should ask whether the Outcome accurately states:

- what is known;
- what remains Unknown;
- what was reviewed;
- what was approved;
- what action occurred;
- what next step is required.

## 15. Evidence must be attributable

Assessment events should identify whether they were produced by:

- candidate;
- AI;
- simulated customer;
- assessor;
- deterministic system;
- provider actor;
- reviewer actor.

```text
Shared Workspace Activity
≠ Candidate Activity by Default
```

## 16. Assessor intervention must remain visible

An assessor may clarify instructions, release facts or correct a platform defect. Such intervention must be recorded so the final evaluation does not attribute the result entirely to the candidate.

## 17. Scoring should combine gates and profiles

Some failures should be blocking gates, such as:

- unauthorized real-world action;
- deliberate data misuse;
- fabricated evidence;
- bypass of non-delegable Professional Authority;
- unsafe duplicate execution after Unknown.

Other dimensions can be shown as a profile:

```text
Fact Handling: Strong
Source Judgment: Moderate
Communication: Strong
Human–AI Governance: Strong
Recovery: Limited Evidence
```

A total score may support administration, but it should not erase the profile.

## 18. Evidence confidence should be recorded

Not every assessment signal has equal strength.

Possible confidence labels include:

```text
Directly Observed
System-recorded
Assessor-inferred
Self-reported
Outcome-inferred
Insufficient Evidence
```

## 19. Assessment Evidence does not automatically become permanent capability truth

```text
Assessment Result
≠ Universal Capability
```

The evidence remains scoped to:

- scenario;
- fixture version;
- jurisdiction;
- Capability;
- M-mode;
- tool set;
- difficulty;
- date.

Later production evidence may strengthen, narrow or contradict the assessment.

## 20. Rights and retention apply to assessment records

Assessment may contain sensitive personal-performance data. Governance should define:

- who can view it;
- retention period;
- appeal rights;
- correction process;
- use in routing;
- use in certification;
- whether it may be used for research or model training.

```text
Assessment Participation
≠ Unlimited Analytics or Training Consent
```

## 21. The governing principle

Assessment should explain not only whether the participant arrived at an answer, but whether they executed responsibly.

```text
Professional Capability Evidence
= Decisions + Authority + Process + Outcome + Recovery
```

This evidence gives MarkOrbit a defensible basis for certification and progression without confusing one exam result with permanent professional authority.
