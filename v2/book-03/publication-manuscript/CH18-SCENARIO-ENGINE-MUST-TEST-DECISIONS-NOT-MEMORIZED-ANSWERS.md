# CH18 — Scenario Engine Must Test Decisions, Not Memorized Answers

## 1. Professional competence is revealed through decisions under changing conditions

A multiple-choice test can confirm whether a person remembers a rule. It cannot reliably show whether that person can operate when facts are incomplete, deadlines move, evidence conflicts and several legitimate routes remain open.

The Scenario Engine therefore tests an execution path:

```text
Observe
→ Interpret
→ Clarify
→ Choose Route
→ Produce
→ Review
→ Communicate
→ Escalate
→ Recover
```

The objective is not to reproduce one hidden model answer. It is to determine whether the participant reaches a defensible Outcome through governed work.

## 2. A scenario is an executable case model

A scenario should contain:

- starting context;
- Workplace and Product context;
- participant role;
- customer objective;
- jurisdiction;
- case objects;
- documents and sources;
- deadlines;
- hidden or staged facts;
- simulated external actors;
- allowed Capabilities and tools;
- M0–M5 mode;
- expected decision points;
- possible branches;
- incident triggers;
- assessment dimensions;
- accepted alternative Outcomes.

```text
Scenario
≠ Static Question Set
```

## 3. The engine should test fact handling before conclusions

Many professional errors arise because the participant acts on an unverified fact.

Scenarios should reveal whether the participant distinguishes:

```text
Customer Statement
Official Record
Provider-reported Practice
Professional Interpretation
AI Inference
Unknown
```

For example, a customer may say that the owner has changed, while the official record still lists the old entity. The correct response is not simply to choose one value. The participant should identify the conflict, determine its effect and propose a governed route.

## 4. Branches should follow decisions and evidence

A scenario branch can be triggered by:

- a clarification request;
- selection of a provider;
- failure to identify a missing document;
- approval of a package;
- an unsafe retry;
- an escalation;
- customer silence;
- a deadline threshold;
- a fee decision;
- a data disclosure.

The engine should not force every participant through the same sequence when different defensible routes exist.

```text
Different Route
≠ Incorrect Route by Default
```

## 5. Alternative correct routes must be modeled

Professional problems often have several acceptable solutions.

A scenario may accept:

- minimum protective filing;
- broader filing after review;
- delay pending evidence;
- provider confirmation before action;
- escalation to qualified authority;
- no-safe-route conclusion.

Assessment should compare each route against:

- facts available at the time;
- authority;
- deadline;
- cost;
- customer objective;
- disclosure;
- residual risk.

## 6. Hidden facts should be discoverable through proper work

The strongest scenarios reward professional inquiry.

A hidden issue may be revealed when the participant:

- compares the applicant name with the registration record;
- opens the complete attachment;
- requests the original POA;
- checks the official fee source;
- asks who owns the customer relationship;
- verifies whether the provider accepted appointment;
- reconciles an Unknown submission.

```text
Hidden Fact
≠ Unobtainable Fact
```

## 7. Simulated actors should have governed roles

The Scenario Engine may simulate:

- customer;
- supervisor;
- colleague;
- contributor;
- reviewer;
- provider;
- official office;
- finance team;
- relationship owner;
- data-protection reviewer.

Each actor should have:

- knowledge boundary;
- communication style;
- response policy;
- authority;
- delay behavior;
- possible error modes.

An examiner actor should not provide information that the role would not reasonably possess.

## 8. Communication quality is part of the scenario

A participant may reach the correct internal conclusion and still fail professionally by communicating poorly.

Scenarios should test whether communication:

- identifies the issue;
- separates facts from assumptions;
- explains options;
- states costs and deadlines;
- requests a specific decision;
- discloses uncertainty;
- preserves represented role;
- avoids misleading certainty;
- records the response correctly.

```text
Correct Analysis
≠ Complete Delivery
```

## 9. Deadline pressure should test prioritization, not panic

A scenario can compress time and introduce urgent events, but urgency should not erase governance.

The participant should decide:

- what must be verified immediately;
- what can proceed in parallel;
- what requires Approval;
- whether emergency authority exists;
- whether a protective route is available;
- when to notify the customer;
- when no safe action remains.

```text
Urgency
≠ Permission to Guess
```

## 10. Unknown outcomes should create reconciliation branches

A simulated connector timeout or provider silence should not be scored as a simple technical failure.

The scenario should observe whether the participant:

- preserves Unknown;
- avoids unsafe retry;
- checks prior attempts;
- searches for an external reference;
- contacts the provider through the correct channel;
- assesses deadline risk;
- escalates;
- documents the recovery decision.

## 11. AI-use scenarios should test governance quality

Where AI is permitted, the engine should test more than output quality.

It should observe:

- whether the participant selected the correct M-mode;
- whether confidential data was disclosed appropriately;
- whether sources were checked;
- whether AI inference was labeled;
- whether hallucinations were detected;
- whether the human Review was meaningful;
- whether the participant escalated when confidence fell;
- whether memory and training rights were respected.

```text
Good AI Draft
≠ Good Human–AI Execution
```

## 12. Tool-use should create observable evidence

The Scenario Engine should record:

- tool chosen;
- purpose;
- input;
- output;
- side effect;
- error;
- retry;
- participant interpretation.

This makes it possible to distinguish:

- tool failure;
- participant misuse;
- incorrect source configuration;
- poor recovery.

## 13. Scenarios should include misleading but plausible signals

Real work includes:

- an outdated provider email;
- a document with a similar file name;
- a fee copied from another jurisdiction;
- a customer using “owner” loosely;
- an AI answer with confident but unsupported detail;
- a portal status that conflicts with an official receipt.

The purpose is to test verification, not to reward suspicion of everything.

## 14. Scenario difficulty should be multidimensional

Difficulty may increase through:

- more objects;
- more jurisdictions;
- lower-quality inputs;
- tighter deadlines;
- conflicting authority;
- multiple contributors;
- external silence;
- higher data sensitivity;
- more consequential Apply actions;
- greater commercial complexity.

```text
More Text
≠ More Difficulty
```

## 15. Scenario fairness requires calibration

Before formal assessment, scenarios should be tested by qualified reviewers.

Calibration should examine:

- whether required facts are available;
- whether instructions are clear;
- whether accepted routes are complete;
- whether scoring weights match risk;
- whether language creates unintended bias;
- whether tool behavior is stable;
- whether time limits are realistic;
- whether hidden facts release correctly.

## 16. Scenario versions must remain stable during assessment

A formal assessment should bind:

- fixture version;
- rule version;
- source snapshot;
- model/tool versions;
- scoring rubric;
- time policy.

```text
Candidate A on Fixture v1
≠ Directly Comparable to Candidate B on Materially Changed Fixture v2
```

## 17. Scenario output should include a decision trace

The final record should reconstruct:

```text
Facts Observed
→ Questions Asked
→ Sources Used
→ Risks Identified
→ Options Considered
→ Decisions Made
→ Contributions Produced
→ Reviews Requested
→ Approvals Obtained
→ Actions Attempted
→ Recovery Performed
→ Outcome Delivered
```

This trace is more valuable than a final score alone.

## 18. The engine should test stopping behavior

Professional competence includes knowing when not to act.

A high-quality response may be:

- request additional evidence;
- require qualified review;
- block disclosure;
- reject an unsafe provider route;
- preserve Unknown;
- state that no safe route exists.

```text
No Action
≠ No Competence
```

## 19. Scenarios can improve Products and Execution

Repeated participant difficulties may reveal defects in:

- Intake;
- Guide explanations;
- Workflow design;
- Work Package scope;
- Review rubric;
- source availability;
- tool design;
- escalation routes.

The system should not automatically blame participants when the scenario exposes a platform defect.

## 20. The governing principle

A Scenario Engine should reproduce the structure of professional judgment:

```text
Incomplete Facts
+ Multiple Authorities
+ Changing Conditions
+ Consequential Choices
→ Observable Governed Decisions
```

Its purpose is not to test whether someone can recite MarkOrbit terminology. Its purpose is to test whether they can use the system to reach an accountable, evidence-backed and safe Outcome.
