# MarkOrbit Capability Canon Decision

## Status

```text

Scope: Book 01–07 v2
Source basis: new Capability OS, Capability Engine, dual-track distillation, Business Capability Map and Founder OS strategy papers
Status: adopted for controlled manuscript revision
Implementation authorization: not created
```

## 1. Why this decision is necessary

The existing seven-book series correctly treats Capability as MarkOrbit’s durable operating unit: a governed competence with a stable outcome contract, explicit inputs and outputs, evidence, risk, review and implementation boundaries.

The new strategy papers add a missing dimension. They explain how Capability is discovered from sources, composed for real problems, tested through practice, reflected upon, verified and released as a new version.

The two views are complementary:

```text
Existing canon
Capability as a stable, governable and invocable contract

New strategy
Capability as an evidence-backed and versioned evolving asset
```

The revised canon combines both without weakening authority, ownership or execution boundaries.

## 2. Canonical Capability definition

> A Capability is a governed, versioned ability to produce a defined outcome under explicit input, authority, evidence, risk and review conditions, supported by one or more implementations and improved only through controlled evidence-backed evolution.

Compact form:

```text
Capability
= Stable Outcome Contract
+ Governed Implementation
+ Evidence Base
+ Version Lineage
+ Controlled Evolution
```

A Capability is not a Prompt, model, tool, workflow, person, licence, course or one successful answer. Any of those may implement, teach, evidence or constrain a Capability.

## 3. Canonical hierarchy

```text
Capability Domain
→ Capability
→ Skill
→ Action / Invocation
```

### Capability Domain

A broad field of related outcomes, such as Trademark Strategy, Brand, Product, Market, Finance or Operations.

A role such as Founder is not itself a Capability Domain. It is a Persona or Capability Profile that draws from multiple Domains.

### Capability

A stable governed outcome contract, such as Brand Positioning, US Trademark Filing or Cashflow Planning.

### Skill

A scenario-specific executable implementation of a Capability. A Skill may contain prompts, workflows, tools, forms, checklists, model routing, review rules and Return requirements.

### Action / Invocation

One real attempt to use a Capability or Skill in a defined context.

## 4. Capability asset classes

### Source Capability

A provenance-bound reconstruction of one author, book, course, practitioner or theory.

It preserves the source’s problem definition, argument, method, assumptions, cases and boundaries. It may be valuable even when it conflicts with another Source Capability.

### Composite Capability

A candidate capability synthesized from multiple Source Capabilities, Knowledge, Data and practice evidence around one real problem.

It is not automatically platform canon.

### Verified MO Capability

A platform Capability version supported by sufficient practice, Reflection, Case, Principle and review evidence under explicit governance.

```text
Source Capability
→ Composite Capability
→ Experimental Capability
→ Verified MO Capability
```

No transition is automatic.

## 5. Three separate lifecycles

### 5.1 Capability production lifecycle

```text
Source
→ Source-preserving Distillation
→ Source Model
→ Source Skill
→ Knowledge Atoms
→ Source Capability Package
```

### 5.2 Capability runtime lifecycle

```text
Capability Request
→ Eligibility
→ Capability Composition
→ Context Compilation
→ Implementation Binding
→ Execution
→ Review
→ Outcome
→ Return / Session Receipt
```

### 5.3 Capability evolution lifecycle

```text
Outcome
→ Reflection Candidate
→ Case Candidate
→ Principle Candidate
→ Evaluation
→ Verification
→ Capability Change Proposal
→ Version
→ Release
```

Runtime success does not directly mutate Capability canon. Reflection, Case and Principle remain candidates until reviewed.

## 6. Capability composition

A complex task should not load every related Skill.

The canonical composition model is:

```text
Capability Composition
├── Primary Capability: exactly one
├── Supporting Capabilities: zero to three by default
├── Critic Capability: zero or one
├── Capability Budget
├── Conflict Set
├── Resolution Rationale
└── Unified Action Plan
```

The Primary Capability owns the final decision structure. Supporting Capabilities answer bounded decision nodes. The Critic Capability checks a defined risk class.

“Master Skill” is not a canonical architecture term. The preferred terms are Capability Composition and Capability Orchestration.

## 7. Capability conflict

Source Capabilities may disagree. The system must not silently average them.

A governed conflict record identifies:

- the issue;
- competing positions;
- source provenance;
- assumptions;
- current context;
- material trade-offs;
- resolution rationale;
- unresolved uncertainty.

Conflict resolution produces one action plan unless the user explicitly chooses a comparison or Council experience.

## 8. Reflection, Case and Principle

### Reflection Candidate

A structured, privacy-controlled account of what happened in one real use: what was attempted, changed, adopted, rejected and learned.

It is not the raw conversation and does not automatically become platform knowledge.

### Case

A governed practice record linking context, Capability versions, actions, Evidence, outcome and boundaries. Cases may be Private, Team, Enterprise, Anonymous Public or Platform Verified.

### Principle Candidate

A reusable rule proposed from one or more Reflections or Cases.

Recommended states:

```text
Candidate
→ Observed
→ Repeated
→ Verified
→ Established
```

A Principle does not become canonical because an AI generated it or because one user succeeded once.

## 9. Capability Ledger and Capability Profile

### Capability Ledger

A governed record of Capability-related events and evidence, such as invocation, practice, Reflection, assessment, accepted Case, certification and version exposure.

### Capability Profile / Capability Twin

A Workplace- or Product-held projection of the Capabilities a Person or Organization has practiced, evidenced or is currently developing.

```text
Capability Twin
≠ personality replica
≠ universal judgment of the user
≠ automatic professional qualification
```

Core may define interoperable references. Products and Workplaces own presentation, private Reflection, growth indicators and coaching experiences.

XP, levels, mentor messages and Skill Tree visualizations are Product-local unless a shared contract is genuinely required.

## 10. Session Receipt

A Session Receipt is a typed execution Return for one Capability invocation.

It may include:

- session and invocation references;
- Capability and Skill versions;
- Product and Workplace context;
- input-scope reference;
- implementation bindings;
- outcome summary;
- user disposition and modifications;
- Evidence references;
- Reflection Candidate reference;
- privacy and rights state.

It must not default to storing the complete raw conversation in Core.

## 11. Context Compiler

The Context Compiler creates a purpose-limited execution context from authorized Data, Knowledge, Memory, Cases, Principles, preferences and Matter facts.

It performs:

- relevance selection;
- version selection;
- permission checks;
- privacy filtering;
- token and Capability budgeting;
- provenance preservation;
- conflict surfacing;
- omission of irrelevant material.

```text
Context Compilation
≠ unrestricted retrieval
≠ silent authority expansion
```

## 12. Product modes

The following are Product experience modes, not new Core authority classes.

### Lens Mode

One Source Capability or Source Skill provides a distinctive perspective. The interface must disclose provenance, method scope and that the output is not the source person’s actual advice.

### Council Mode

Several Source Capabilities present materially different positions and assumptions. It is primarily exploratory and educational.

### MO Mode

A governed Capability Composition produces one unified action plan, explicit conflicts, risks and validation needs.

## 13. Founder OS decision

Founder OS is adopted as a future Product direction, not as a replacement for the current seven-book architecture.

```text
Founder
= Persona / Capability Profile Type
≠ Capability Domain
```

A Founder Capability Profile may draw from Vision, Brand, Product, Market, Sales, Operations, Finance, People and Expansion.

Mo Lite remains the daily Capability Workplace for trademark professionals and small teams. A future Founder-facing Product may reuse Core and Execution Capability semantics without redefining Lite.

## 14. Ownership and authority locks

```text
Platform Capability
≠ User Capability Profile

Reflection Candidate
≠ Canonical Principle

Source Skill
≠ Verified MO Capability

Capability Certification
≠ Professional Qualification

Capability Composition
≠ Protected-action Authority

Outcome
≠ Automatic Capability Upgrade

Capability Twin
≠ Platform ownership of the user
```

Professional Qualification, formal-state authority, customer relationships, funds authority and protected actions remain governed by the existing seven-book authority model.

## 16. Revision rule

This decision authorizes controlled manuscript revision only.

It does not by itself:

- add production schemas;
- activate a Capability Engine;
- authorize model training on private Reflections;
- create a Founder OS Product;
- change Professional Qualification;
- grant automatic action authority;
- replace existing frozen editions.
