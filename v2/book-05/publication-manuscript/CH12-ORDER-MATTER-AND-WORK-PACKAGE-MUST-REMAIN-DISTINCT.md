# Chapter 12 — Order, Matter and Work Package Must Remain Distinct

A customer may buy “a US trademark filing.”

That commercial phrase appears singular. The work may contain:

- business-objective clarification;
- search;
- applicant verification;
- goods review;
- document collection;
- filing preparation;
- professional review;
- local Provider instruction;
- official-fee payment;
- receipt validation;
- examination monitoring;
- customer reporting.

The Product fails when it forces every layer into one record called an order, case, project or task.

MarkReg should preserve three different units:

```text
Order
→ commercial commitment

Matter
→ governed professional context

Work Package
→ bounded unit of production or review
```

They connect. They are not interchangeable.

## 1. Why the distinction matters

The distinction answers different questions.

### Order

- What did the customer purchase?
- At what price and under which terms?
- Which party is the Contracting Party?
- What is included and excluded?
- What payment conditions apply?

### Matter

- Which legal or professional context exists?
- Which customer, applicant, right, jurisdiction and procedure are involved?
- Who has confidentiality and professional responsibility?
- Which deadlines and official states matter?

### Work Package

- What exact bounded outcome must be produced?
- Who may perform it?
- What sources and data are needed?
- Which review and acceptance criteria apply?
- What evidence proves completion?

```text
Order
≠ Matter
≠ Work Package
≠ Task
≠ External Protected Action
```

## 2. One Order may support many Matters

A customer may accept one commercial proposal covering:

- US word mark;
- US device mark;
- EU word mark;
- UK word mark;
- China device mark;
- search and classification;
- future examination reporting.

This may become:

```text
One Order
→ US Word Matter
→ US Device Matter
→ EU Word Matter
→ UK Word Matter
→ China Device Matter
```

Each Matter may have different:

- applicant;
- mark representation;
- jurisdiction;
- goods scope;
- Provider;
- official identifier;
- deadline;
- examination result;
- status;
- later commercial scope.

One jurisdiction’s failure should not corrupt the formal state of the others.

## 3. One Matter may require many Work Packages

A US filing Matter may contain:

```text
Applicant Verification Work Package
Goods Normalization Work Package
Translation Review Work Package
Search Interpretation Work Package
Filing Package Preparation Work Package
Professional Review Work Package
Provider Instruction Work Package
Submission Evidence Check Work Package
Customer Delivery Work Package
```

A later OA may add:

```text
Official Notice Validation Work Package
Issue Decomposition Work Package
Customer Fact Collection Work Package
Evidence Review Work Package
Response Drafting Work Package
Professional Response Review Work Package
Response Submission Work Package
Acknowledgement Validation Work Package
```

The Matter provides context and responsibility continuity. Work Packages make production assignable and reviewable.

## 4. A Work Package is defined by Outcome

A useful Work Package should state:

- source Matter and Order references;
- expected Outcome;
- scope;
- exclusions;
- source artifacts;
- exact versions;
- jurisdiction and language;
- risk classification;
- capability and qualification requirements;
- assigned actor or organization;
- permitted tools and AI use;
- data-access scope;
- deadline;
- review criteria;
- acceptance evidence;
- revision rules;
- compensation where applicable;
- failure and escalation route.

Examples of bounded Outcomes:

- verified applicant identity record;
- reviewed goods list;
- translated and checked office notice;
- accepted Filing Package Candidate;
- validated Provider Return;
- customer-ready status report;
- evidence gap analysis.

“Handle the application” is not a sufficiently bounded Work Package.

## 5. Product Session and formal objects are different

A user may explore options, compare plans and prepare information in a Product Session before any Order or Matter exists.

```text
Product Session
→ Recommendation
→ Customer Decision
→ Commercial Instruction
→ Formalization Handoff
→ Order and Matter references
```

The Product Session may contain:

- candidate jurisdictions;
- draft marks;
- inferred goods;
- price scenarios;
- unresolved assumptions;
- saved options.

Those are not formal Order or Matter facts until the applicable Owning Services accept and create them.

```text
Selected Product Option
≠ Order

Intake Progress
≠ Matter
```

## 6. Formalization requires a governed Handoff

The Handoff should reference:

- accepted Quote;
- Commercial Instruction;
- customer and applicant context;
- service scope;
- jurisdictions and filing units;
- Formal Intake;
- Requirement Set;
- Readiness Assessment;
- payment conditions;
- required formal records;
- intended responsibility assignments;
- deadlines;
- exact artifact versions;
- idempotency key;
- expected Return Evidence.

```text
Handoff Sent
≠ Order Created
≠ Matter Created
≠ Work Assigned
≠ Filing Submitted
```

The receiving Owning Service evaluates and returns typed results.

## 7. The applicable Owning Service creates formal facts

MarkReg may prepare and request formalization. It should not silently create competing formal objects.

| Formal object | Controlled meaning | MarkReg role |
|---|---|---|
| Customer | recognized commercial or service relationship | reference authorized context |
| Order | accepted commercial scope and terms | prepare Handoff and project returned state |
| Matter | governed professional work context | reference and operate the Product journey |
| Financial Record | invoice, payment, credit, refund and reconciliation | display permitted projection |
| Document | controlled document identity and version | prepare, reference and use for purpose |
| Assignment | accepted responsibility and access | request and observe |
| Execution Context | coordinated work and protected-action entry | prepare request and receive result |

```text
Displayed in MarkReg
≠ Owned by MarkReg
```

## 8. Order scope must remain commercial

An Order should record:

- customer;
- Contracting Party;
- Payment Receiver;
- accepted services;
- included and excluded jurisdictions;
- pricing and currency;
- official and third-party cost treatment;
- payment milestones;
- validity and expiry;
- amendments;
- cancellation and refund rules;
- related Matter references.

The Order should not become the sole source for professional or official state.

```text
Order Paid
≠ Matter Professionally Ready
≠ Filing Approved
≠ Filing Completed
```

## 9. Matter scope must preserve professional context

A Matter may record:

- customer and represented parties;
- applicant or owner;
- jurisdiction;
- procedure;
- mark or right;
- confidentiality;
- Professional Owner;
- Delivery Owner;
- deadlines;
- official identifiers;
- source events;
- accepted instructions;
- active packages;
- related Orders;
- related Work Packages;
- current formal and official state references.

A Matter may continue after the original Order has been commercially completed because later OA, renewal, dispute or recordal work requires a new commercial event.

## 10. A new commercial event does not always require a new Matter

Suppose an application receives an OA not included in the original filing price.

Possible structure:

```text
Existing Application Matter
+ New OA Response Order or Order Amendment
+ New Response Work Packages
```

The professional context remains the same Matter, while the commercial scope changes.

In another case, a cancellation proceeding may justify a separate Matter because confidentiality, professional responsibility, participants and lifecycle are substantially different.

The decision should follow governance rather than one universal rule.

## 11. Work Packages can cross internal and external production

A Work Package may be fulfilled by:

- internal staff;
- AI under governance;
- deterministic system;
- certified Contributor;
- expert Reviewer;
- another Workplace;
- MGSN Provider;
- direct external Provider;
- Connector;
- mixed Human–AI route.

The Delivery Owner remains explicit.

```text
Production Source
≠ Delivery Ownership
≠ Professional Authority
```

## 12. Assignment is distinct from eligibility

The production chain is:

```text
Work Package Defined
→ Capability Need
→ Eligible Candidate
→ Recommendation
→ Human or Policy Selection
→ Assignment Offered
→ Assignment Accepted
→ Access Granted
→ Contribution or Delivery
→ Review
→ Acceptance or Revision
```

```text
Eligible
≠ Assigned

Assigned
≠ Authorized Beyond Scope

Task Access
≠ Customer Relationship
```

## 13. Responsibility needs more than one owner field

A Matter or Work Package may require:

- Relationship Owner;
- Contracting Party;
- Payment Receiver;
- Professional Owner;
- Delivery Owner;
- Work Package Assignee;
- Reviewer;
- Approver;
- Deadline Owner;
- Document Owner;
- Evidence Owner;
- Provider Manager;
- Customer Decision Owner;
- Finance Owner;
- recovery owner.

One generic `owner` field hides authority and creates gaps.

## 14. The Delivery Owner does not need to perform every task

The Delivery Owner is accountable for the promised Outcome and coordination. They may rely on:

- Contributors;
- reviewers;
- local Providers;
- systems;
- customer-supplied information;
- official sources.

They remain responsible for ensuring that:

- Work Packages are sufficient;
- dependencies are visible;
- review occurs;
- accepted results are integrated;
- exceptions are escalated;
- customer communication is accurate;
- final evidence is returned.

```text
Delegated Work
≠ Delegated Accountability Automatically
```

## 15. Professional authority attaches to action, scope and jurisdiction

A person may prepare data without being authorized to provide legal advice. A reviewer may assess a Package without having payment authority. A local Provider may file after appointment without owning the customer relationship.

Each consequential action should record:

- actor;
- acting organization;
- role;
- authority basis;
- jurisdiction;
- subject and version;
- purpose;
- consequence;
- validity period.

```text
Professional Access
≠ Filing Authority
≠ Commercial Approval
≠ Payment Authority
```

## 16. Payment records remain separate from work state

The Product should distinguish:

```text
Quote Amount
Invoice Amount
Amount Due
Amount Received
Amount Reconciled
Amount Remitted to Provider
Official Fee Paid
Refund Due
Refund Paid
```

Payer, Customer and Applicant may be different.

A payment may satisfy one commercial gate without proving:

- identity readiness;
- document readiness;
- Professional Review;
- Filing Approval;
- Provider Acceptance;
- official submission.

```text
Payment Received
≠ Work Complete
```

## 17. Amendments must propagate to affected objects

A material change may affect:

- Order scope;
- price;
- Matter strategy;
- applicant;
- mark version;
- goods;
- documents;
- Provider route;
- deadline;
- Work Packages;
- review;
- approval;
- customer communication.

The system should create an Amendment or superseding version rather than silently rewriting accepted history.

### Example

The customer adds a second class after the original filing Package was approved.

Required impact may include:

```text
Order Amendment
→ additional price and fee
→ updated Matter scope
→ revised goods Work Package
→ revised Filing Package
→ new Professional Review
→ new Filing Approval
→ updated Provider instruction
```

## 18. Work Package revision must be controlled

A Work Package may result in:

```text
Accepted
Accepted with non-material correction
Revision Required
Partially Accepted
Rejected for Specified Reason
Cancelled Because Dependency Failed
Superseded by New Scope
```

The system should distinguish:

- correction of assignee error;
- clarification within scope;
- new customer information;
- changed source;
- expanded scope;
- reviewer preference not in the criteria;
- urgent replacement.

Unlimited informal revision creates unfair labor and weak evidence.

## 19. Completion belongs to a defined Outcome

Different levels may be complete at different times:

```text
Work Package accepted
Matter stage complete
Order deliverable complete
Provider work complete
Official result verified
Customer delivery complete
Financial reconciliation complete
```

A complete Work Package does not make the Matter complete. A commercially completed Order does not make the official procedure complete.

The user surface should state which Outcome is complete.

## 20. Partial failure must remain visible

Formalization or production may result in:

```text
Order created; Matter creation pending
Matter created; payment pending
Payment received; reconciliation pending
Three Work Packages accepted; one blocked
Provider accepted one jurisdiction; declined another
Execution Context created; Filing Approval pending
Official receipt received for four of five rights
Result unknown; reconciliation required
```

The Product should not show the initiative as fully complete.

## 21. Idempotency prevents duplicate formal objects and actions

Every boundary-crossing request should carry an idempotency identity.

This matters for:

- Order creation;
- Matter creation;
- invoice generation;
- payment reconciliation;
- Assignment creation;
- Provider instruction;
- submission;
- Return processing;
- customer notification.

When the same request is processed twice, the receiving service should return the existing reference or reconcile the current state rather than create a duplicate.

```text
Retry
≠ Repeat Effect
```

## 22. Work Package access must be minimum necessary

An assignee should receive only the data required for the Outcome.

Examples:

- a translator receives the relevant notice and terminology context, not the full financial record;
- a visual reviewer receives the mark and comparison criteria, not unrelated customer correspondence;
- a local Provider receives the Filing Package and authority documents, not the customer’s entire portfolio;
- a data verifier receives source identifiers, not confidential strategy notes.

Access expires when the Assignment ends, subject to necessary evidence and retention controls.

## 23. Customer communication should follow relationship rules

A Contributor or Provider who prepares work does not automatically communicate with the customer.

The communication plan should define:

- sender;
- acting Workplace;
- Relationship Owner;
- visible brand;
- professional identity disclosure;
- purpose;
- approved content;
- permitted direct contact;
- response route;
- delivery evidence;
- correction and supersession.

```text
Can Perform Work
≠ Can Contact Customer Directly
```

## 24. AI participates as a bounded production actor

AI may:

- decompose service scope;
- suggest Work Packages;
- extract fields;
- draft artifacts;
- compare versions;
- identify dependencies;
- summarize Returns;
- flag duplicate or missing work;
- prepare progress explanations.

AI may not:

- create an Order acceptance;
- create a formal Matter unilaterally;
- assign final accountability;
- grant data access;
- approve a Package;
- appoint a Provider;
- authorize payment;
- submit externally;
- mark official state complete;
- infer customer instruction from inactivity.

## 25. Example: one filing service

A customer accepts one Order covering a US word mark and device mark.

The Product creates two Matters because the filing units and official procedures remain independent.

### Shared Work Packages

- customer identity verification;
- applicant authority review;
- business-objective confirmation.

### Word Matter Work Packages

- word-mark search review;
- word-mark goods preparation;
- word Filing Package;
- word submission validation.

### Device Matter Work Packages

- device-image validation;
- mark-description and color review;
- device search review;
- device Filing Package;
- device submission validation.

One Order and shared customer context reduce repetition. Independent Matters preserve official truth.

## 26. Example: a maintenance portfolio

A customer accepts an Order to review six registrations.

The professional structure may include:

- six right-specific Matters or governed right contexts;
- one portfolio-coordination context;
- six status-verification Work Packages;
- four renewal-preparation Work Packages;
- one ownership-chain Work Package;
- one cancellation-defense escalation;
- two use-evidence reviews;
- one customer decision package.

The portfolio service remains commercially coherent without pretending the six rights share one legal status.

## 27. Failure modes

The architecture fails when:

- Product Session becomes Order automatically;
- one Order is forced into one Matter;
- Matter and Work Package are used as synonyms;
- payment becomes professional readiness;
- customer becomes applicant automatically;
- Task access becomes authority;
- Contributor access becomes customer relationship;
- Provider acceptance becomes Matter creation;
- Work Package completion becomes official completion;
- commercial amendment silently changes approved Packages;
- retry creates duplicate Orders, Matters or submissions;
- partial success appears complete;
- one generic owner field hides responsibility;
- AI becomes accountable actor;
- Matter creation triggers external filing.

## 28. Product and technical implications

MarkReg requires:

- Product Sessions and pre-formal state;
- governed formalization Handoffs;
- stable Order, Matter and Work Package references;
- one-to-many and many-to-many relationships;
- role and responsibility records;
- typed Assignments;
- minimum-necessary access;
- versioned scope and amendments;
- outcome-specific completion;
- idempotency and reconciliation;
- partial-failure states;
- Return Evidence;
- customer and Provider communication contracts;
- links to Execution and Owning Services.

## 29. Commercial implications

The distinction supports clearer economics:

- Order pricing can reflect customer value and commercial scope;
- Matter accounting can show jurisdiction and procedure cost;
- Work Package compensation can reflect production value and complexity;
- Provider costs remain distinct;
- professional review can be priced explicitly;
- scope changes can be amended fairly;
- margin can be analyzed without hiding responsibility.

It also prevents the lowest-cost task performer from being confused with the Contracting Party or Delivery Owner.

## 30. The operating principle

```text
Sell through Orders
Govern professional context through Matters
Produce through bounded Work Packages
Assign explicitly
Review exact Outcomes
Preserve formal authority
Reconcile partial results
Never let one object impersonate another
```

> Order, Matter and Work Package are not database preferences. They are the structural separation between what the customer bought, what professional responsibility exists and what exact work must be performed and accepted.
