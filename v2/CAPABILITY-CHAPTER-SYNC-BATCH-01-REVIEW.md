## Main integrations

The revised chapters now carry the following publication-level canon:

```text
Capability Domain
→ Capability
→ Skill
→ Action / Invocation
```

They also distinguish:

```text
Source Capability
≠ Composite Capability
≠ Experimental Capability
≠ Verified MO Capability
```

The three Capability lifecycles are separated:

```text
Production:
Source → Distillation → Source Capability

Runtime:
Request → Eligibility → Composition → Execution → Outcome

Evolution:
Outcome and Evidence
→ Reflection Candidate
→ Case Candidate
→ Principle Candidate
→ Capability Change Proposal
→ Version Review
→ Release
```

Runtime composition is now bounded through:

```text
one Primary Capability
+ zero to three Supporting Capabilities
+ zero or one Critic Capability
+ Capability Budget
+ explicit Conflict handling
```

Book 03 now introduces the Context Compiler and Session Receipt as bounded runtime mechanisms without turning either into formal state, raw permanent memory or automatic learning permission.

## Important corrections

1. Capability is now presented as both a stable governed outcome contract and a versioned evolving asset.
2. The Professional Capability Map and Business Capability Map are separated.
3. Founder is treated as a Persona and Capability Profile pattern, not a Capability Domain.
4. Capability Profile, Capability Twin and Provider Supply Capability remain separate evidence contexts.
5. Source Skills and Source Capabilities preserve provenance rather than silently becoming platform canon.
6. Runtime Outcomes may propose learning but cannot mutate Capability definitions.
7. Capability composition cannot create Professional Authority or protected-action approval.

## Locks preserved

```text
Outcome ≠ automatic Capability upgrade
Source Skill ≠ Verified MO Capability
Reflection Candidate ≠ raw conversation archive
Reflection Candidate ≠ shared Knowledge
Reflection Candidate ≠ canonical Principle
Capability Composition ≠ Professional Authority
Capability Certification ≠ Professional Qualification
Session Receipt ≠ formal state
Session Receipt ≠ shared learning permission
Product wording ≠ Capability authority expansion
```

This batch does not:

- modify Active Canon;
- modify frozen Book baselines;
- activate new Core objects or public exports;
- authorize production Capability Engine implementation;
- create model-training permission;
- create Professional Qualification or Matter authority;
- authorize external protected actions or funds custody.

## Editorial result

The seven revised chapters no longer depend on the earlier static-contract-only explanation of Capability. They now form a continuous publication argument:

```text
Capability as governed outcome
→ Capability maps and Profiles
→ source-preserving production
→ shared semantic contracts
→ runtime request and composition
→ Session Receipt and controlled learning
→ governed version evolution
```
